なぜこの記録なのか

正直に書くと、この内容を書く動機は「自分が 2 ヶ月前に欲しかったもの」が外には売ってなかったから。

クラウドワークスの攻略本はある。Claude Code の使い方を解説する記事もある。でも「文系・非エンジニア・プログラミング未経験の人が、Claude Code で実際にクラウドワークス案件を取って納品するまで何をやればいいか」を、毎日のレベルで時系列に書いてるものは見つからなかった。

2026 年 1 月、Claude Code を触り始めた頃のわたしは、Web で検索しても出てくるのはエンジニア向けの解説か、AI 副業を煽る抽象記事ばかり。「文系大学生がクラウドワークスで何を作って納品するのか」の具体例が、本当にどこにもなかった。

「やってる人の内側」を、誰かに見せてもらえなかった。

このパッケージは、その「誰かの内側」になりたくて書いた。実際に 2 ヶ月でわたしが何をやって、どこで詰まって、どう抜けたかを、追体験できる順番で並べてある。

note 版（¥2,000）にも書いてるけど、note 版はあくまで「2 ヶ月の振り返り記事」。Brain 版は note 版に対してこういう違いがある。

・ note 版は時系列ストーリー中心、Brain 版は 毎日の作業手順を 14 日カリキュラム化

・ note 版はプロンプト 5 本抜粋、Brain 版は プロンプト 20 本全文（コピペ可）

一言で言うと「note 版は読み物、Brain 版は作業マニュアル」。読んだだけじゃなくて、開きながら手を動かして案件を取りに行く前提で組んでる。

note 版を読んでくれた人にも、Brain 版を買う 3 つの理由がある。

1 つ目。Day 別の作業手順が決まってる。 note 版は「2 ヶ月の流れ」を時系列で書いてあるけど、読み終わったあとに「で、明日何をすればいい？」が残る。Brain 版は Day1 から Day14 まで、毎日 30 分から 2 時間でやることが具体的に書いてある。

2 つ目。プロンプトとコードが全部コピペできる状態で並んでる。 note 版だと文章の中に部分的に貼られてるプロンプトを、自分で抜き出して整える必要がある。Brain 版は <pre><code> の形で全部独立して貼ってあるから、Brain の画面からそのままコピーできる。

3 つ目。失敗記録が 7 件に増えてる。 note 版の誤送信事件・テスト不足 v6 の 2 件に加えて、単価交渉ミス・継続案件のスコープ膨張・クライアント感情の読み違え・複数案件の優先度判断ミス・納期遅延の伝え方失敗、の 5 件を Brain 版で追加してる。

クラウドワークスには Day2 で登録するから、まだサイトを開かなくていい。今日やるのは 3 つだけ。

① Claude Code をインストール

公式サイト（claude.com/code）からダウンロード。手順は公式に書いてあるとおりに、画面の指示に従ってクリックするだけ。所要時間 15 分。

つまずきポイントを 1 つだけ書く。インストール時に「ターミナル」を開かされる場面がある。Mac は標準で入ってるアプリ、Windows は「PowerShell」か「Windows Terminal」で代用 OK。怖く見えるけど、コピペするだけだから安心して。

② 起動コマンドを覚える

ターミナルで claude と入力して Enter。これだけで起動する。終了は Ctrl + C（Mac は command + C）。

起動できると、画面に「>」みたいな入力受付マークが出る。ここに日本語で指示を書く。

③ 最初の依頼を投げる

最初の指示はこれをコピペして送ってみてください。

はじめまして。わたしは文系大学生で、プログラミング経験はありません。
クラウドワークスで業務自動化スクリプトの案件を受注したいと考えています。
Claude Code がどんなことをわたしの代わりにやってくれるのか、
3 つの具体例で教えてください。なるべく短い言葉で、専門用語を避けて。

Claude Code が 3 つの具体例を返してくれる。たとえば「CSV ファイルを別のフォーマットに変換するスクリプト」「メールの本文を自動生成するツール」「Web ページから情報を取得して表にまとめるスクリプト」みたいな感じ。

この瞬間に「あ、これ案件文に書いてあるやつだ」って気づく。クラウドワークスで募集されてる業務自動化案件って、このレベルのものが多い。

Day1 で詰まる人へ — わたしが本当に詰まったポイント 3 つ

「インストールするだけ」って書いたけど、わたしも Day1 はスムーズじゃなかった。詰まったポイントを 3 つ書く。

① ターミナルに何を打てばいいか分からなくて固まる。

ターミナルを開いた瞬間「>」だけが点滅してて、何を打てばいいか分からない。プログラミングの YouTube 動画みたいに、慣れた人がカタカタ打ってる絵が頭に浮かんで「これ自分には早すぎたかも」ってなった。

解決: 公式インストール手順のページを開いた状態で、書いてある通りにコピペするだけ。考えない。判断しない。コピペする。これに気づくのに 30 分かかった。

② インストール後、起動しても「何を頼めばいいか分からない」で止まる。

「>」が出ても、こっちが何を頼めばいいか思いつかない。「Hello World」って書いてみたけど、Claude が「Hello！何かお手伝いしましょうか？」って返ってきて、そこからまた固まる。

解決: Day1 の最初の依頼プロンプト（上で貼ったやつ）をそのままコピペ。自分で考えない。「文系大学生で副業案件を受注したい、何ができるか具体例 3 つ」を聞くと、Claude が「あ、なるほど」って前提を理解してくれる。

③ 出てきた回答を読んでも、結局自分にできるイメージが湧かない。

「CSV ファイルを別のフォーマットに変換するスクリプト」って言われて、「えっと…CSV って何？スクリプトって何？」で思考停止する。

解決: 出てきた答えを、もう一度 Claude Code に投げる。「上の説明に出てきた『CSV』『スクリプト』『API』を、それぞれプログラミング知らない人向けに 1 行ずつ説明してください」って。これで全部の用語が分解される。

Day3 初応募で詰まる全パターンと回避策

既読すらつかない原因 6 パターン

パターン 1: プロフィールが評価ゼロ・実績ゼロのまま

クラウドワークスは、応募する前にクライアントが「応募者プロフィール」をクリックして見る仕組み。プロフィールが何も書かれてないと、応募文を読まずに弾かれる。

回避策: プロフィールに最低限「できること 3 つ・使う道具・連絡可能時間・納期目安」を書く。Day2 の仮プロフィールは Day3 で Claude Code に整えてもらう。

パターン 2: 応募タイミングが遅すぎる

クライアントは案件公開後、最初の 24 時間で来た応募を中心に見てる。3 日経ってから応募すると、すでに候補が絞られてる可能性が高い。

回避策: 案件公開から 24 時間以内に応募する。CW の通知設定で「新着案件メール」を ON にする。スマホ通知も入れておく。

パターン 3: 単価が異常に低い案件に応募してる

「初心者歓迎」「副業 OK」みたいなタグがついてる激安案件は、競合が 30 人以上来る。新規参入者が勝ち抜くのは現実的じゃない。

回避策: ¥10,000 〜 ¥30,000 のレンジに絞る。¥3,000 以下の案件は最初は避ける。

パターン 4: 応募文がテンプレ感丸出し

「ぜひ私にお任せください！」「全力で対応します！」みたいなフレーズだけだと、クライアントは無視する。

回避策: 応募文テンプレ A の「案件への気づき」を必ず 1 つ入れる。これだけで返信率が体感 5 倍くらい変わる。

パターン 5: クライアントの公開実績がゼロ（怪しい案件）

そもそも返信する気がないクライアント、もしくは案件詐欺の可能性がある。プロフィールに過去の発注実績がないクライアントは避ける。

回避策: クライアントのプロフィールページで「過去の発注数」「平均評価」を確認してから応募する。発注数 5 件以上・評価 4.5 以上を最低ラインにする。

パターン 6: 応募文の量が極端

2 行で終わる応募文も、5,000 字超の長文も、両方とも読まれない。

回避策: 応募文は 400 〜 800 字に収める。案件への気づき・実装プラン・納期と概算費用・締め、の 4 ブロック構成。

3 月後半、Chrome 拡張機能の開発案件で、テスト範囲が甘くてクライアントから 6 回バグ報告を受けた。¥35,000 の案件で、修正対応に追加で 12 時間かかった。実質時給 ¥1,500 まで落ちた。

何が起きたか

案件は Amazon の商品ページを開いたら自動で商品説明を整形して、別の EC サイトに登録する Chrome 拡張機能。

・ v1 納品 OpenAI API のクレジット不足で、クライアントの環境で動かなかった

・ v2 納品 EC サイト認証後に Amazon ページで動かないバグ

・ v3 納品 書籍ページで商品説明が取れない

・ v4 納品 カテゴリが正しく設定されない

・ v5 納品 特殊記号（エンダッシュという長いハイフン）でエラー

・ v6 納品 ようやく安定

クライアントから「こちらが提示した商品だけでテストされてます？」って直接指摘された。完全にその通りで、サンプル 5 件だけでしかテストしてなかった。Amazon のページ構造が商品カテゴリごとに全然違うことを想定できてなかった。

受け入れテスト観点生成プロンプト

この失敗以降、納品前に必ず投げるプロンプトができた。

この機能の受け入れテスト観点を、以下の切り口で網羅的に出してください。

1. 正常系（意図通りに動くケース）
2. 異常系（エラー・例外ケース）
3. 境界値（数値・文字数の上限下限）
4. 環境依存（OS・ブラウザ・ネット状況）
5. データのバリエーション（フォーマット差・特殊文字・空データ）
6. ユーザー操作の順序違い

各切り口で 5 〜 10 項目ずつ、合計 50 〜 100 項目を出してください。

【機能の仕様】
{仕様を貼る}

【利用環境】
{利用環境を貼る}

このプロンプトを実装完了時に投げると、テスト観点が 50 〜 100 項目出てくる。全部やるのは無理だけど、見た瞬間に「あ、この観点抜けてた」って気づける。

これ以降、同じ系統のバグ報告で追加修正を食らうことはなくなった。納品前に 30 分かけてこのチェックを通すだけで、納品後の追加対応が 5 分の 1 くらいに減った。

納品前の必須チェックリスト 7 項目

受け入れテスト観点プロンプトの出力をベースに、わたしが毎回手動でチェックする 7 項目。

・ 正常系で 3 パターン以上のサンプルで動作確認（クライアント提示分 + 自分で用意した 2 件以上）

・ 異常系で「想定されるエラー」を 5 個試す（空データ・型不一致・大量データ・ネット切断・権限不足）

・ 環境依存を 1 件は試す（自分と違う OS / ブラウザ / Python バージョンで 1 度動かす）

・ 特殊文字を 3 種類試す（エンダッシュ・スマートクオート・絵文字）

・ データのフォーマット差を 2 件試す（クライアント提示と微妙に違う形）

・ 処理が中断された場合の挙動を 1 件試す（途中で止めて再開できるか）

・ README に「動かない時のチェック手順」を 5 行以上書く（クライアント側で 5 分以内に切り分けできる手順）

これを納品前に通しておくだけで、追加修正の発生率が体感 5 分の 1 になる。

Auto Mode とは

Auto Mode は Claude Code に作業を任せたときに、一つ一つの操作で「これをやっていいですか？」って確認を取らずに連続で進めてくれる機能。

Auto Mode を使う前のわたしは、毎回確認ダイアログが出るたびに「はい」「はい」って押してた。1 案件で 50 〜 100 回くらい確認が出ると、それだけで 30 分〜 1 時間かかってた。

Auto Mode 導入前 vs 導入後

・ 導入前 要件確認 → 指示 → 確認ダイアログ連打 → 手修正 → 再テスト ／ 1 案件 平均 8 〜 10 時間

・ 導入後 要件確認 → 指示 → 完成物チェック → 納品 ／ 1 案件 平均 2.5 〜 3 時間

同じ時間で受けられる案件数が、月 2 〜 3 件から 4 〜 5 件に増えた。これだけで売上が 1.5 倍以上になる計算。

Auto Mode 有効化の手順（実作業 約 3 分）

Auto Mode の有効化は本当に簡単。Claude Code を起動した状態で、キーボードの Shift を押しながら Tab キーを押すだけ。画面に「Auto Mode: ON」って出る。もう一度 Shift + Tab で戻せる。

最初「え、これだけ？」って思った。これだけです。

Auto Mode を切るタイミング 3 つ

Auto Mode は便利だけど、すべての場面で ON にするとリスクがある。わたしが切るタイミングが 3 つある。

・ 本番環境のデータに触る作業のとき（クライアントの実 DB を触る・ファイルを直接書き換える等）

・ クライアントから渡された未検証のコードを実行するとき（マルウェア混入・破壊的操作のリスクを Claude が判断できない）

・ 大きな金額を動かす処理を含むとき（決済 API 呼び出し・大量メール送信等）

それ以外はずっと Auto Mode で回してる。

deny rules で「絶対やらせない操作」を固定する

Auto Mode を ON にしてても、特定の操作だけは確認させたい。これは ~/.claude/settings.json に書く。

{
  "permissions": {
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force*)",
      "Bash(curl * | bash)",
      "Bash(*sudo*)",
      "Edit(/etc/**)",
      "Edit(.env)"
    ]
  }
}

これを設定ファイルに入れておくと、Auto Mode で動いてても破壊的操作だけは止まる。同梱の「CLAUDE.md テンプレ」セクションに、案件タイプ別の deny rules テンプレも 5 種類用意してある。

Day8 の到達目標

・ Auto Mode が有効化できている

・ deny rules が設定ファイルに書けている

・ 1 案件で「確認ボタン押す回数」が 80% 以上削減できている実感がある

トラブルシュート集 12 件

Claude Code 運用で実際に遭遇したトラブルと、それぞれの解決手順。

① Claude Code が起動しない

原因: 認証トークン期限切れ・ネットワーク不通・PC スリープ復帰直後の同期遅延。

解決: claude logout → claude login で再認証。それでも駄目ならネット接続を確認。

② Auto Mode が反応しない

原因: ターミナルアプリが Shift + Tab をショートカットとして横取りしている。

解決: ターミナル設定でキーバインディングを確認。iTerm2 / Windows Terminal の設定で重複していないか。

③ deny rules が効かない

原因: settings.json の構文エラー・ファイル配置場所の間違い。

解決: ファイルを ~/.claude/settings.json（ユーザー全体）か .claude/settings.json（プロジェクトローカル）に正しく置く。JSON 構文を JSONLint で検証。

④ コードが提案されるけど実行されない

原因: Claude Code が「これを実行していいか」確認待ちになっている。

解決: 「実行してください」を明示的に追加するか、Auto Mode を有効化。

⑤ 同じエラーが繰り返し出る

原因: Claude Code が局所的な修正だけで根本原因に踏み込んでいない。

解決: このエラーの根本原因を 3 つ以上の視点で分析してから、最も確からしい修正を提案してください と投げ直す。

⑥ コード生成が途中で止まる

原因: 出力長制限・コンテキスト長超過。

解決: 「ファイルを 3 つに分けて、1 つずつ提案してください」と分割指示。

⑦ 設定ファイルが反映されない

原因: Claude Code を再起動していない。

解決: 設定変更後は必ず Claude Code を一度終了して再起動。

⑧ Remote Control が PC 側に繋がらない

原因: PC 側の Claude Code が起動していない・スリープ中。

解決: PC 側で Claude Code を起動した状態にしておく。スリープ防止設定を入れる。

⑨ Python のライブラリが見つからない

原因: 仮想環境（venv）が有効化されていない・別の Python バージョンを参照している。

解決: which python / which pip で参照先を確認。venv を再有効化。

⑩ クライアントの環境で動かない

原因: 自分の環境に依存した設定・パスがハードコード。

解決: 設定はすべて環境変数化。パスは相対パスで書く。クライアントに README.md の動作確認手順を試してもらう。

⑪ メッセージ送信時に文字化け

原因: 文字コード非対応の特殊記号（エンダッシュ等）。

解決: 半角の通常記号に置換してから送信する。長文を CW の入力欄に直書きしない。

⑫ 案件の検索結果が出ない

原因: クラウドワークス側の検索条件が厳しすぎる。

解決: カテゴリ未指定で全件取得 → キーワードを 1 〜 2 個だけ追加 → 並び替えを「新着順」に。
