Claude Code、毎回ゼロからコード書き続けてますか？

プラグインを入れる前の自分は、TypeScript案件のたびにターミナルで tsc --watch を回しながら作業してた。型エラーが出るたびにコンパイル待ち。1案件あたりの作業時間のうち、たぶん1〜2割はそれだけで持っていかれてた。

しかも正直、プラグインの存在は知ってた。「設定めんどくさそう」「あとで入れよう」と思い続けて、3ヶ月ほったらかしてた。

先週、やっと重い腰を上げて /plugin を叩いた。Discoverタブを開いたら、公式プラグインが並んでいた。全部タダで使える。

typescript-lsp、github、commit-commands。この3本を入れただけで、週の副業作業時間が5時間から2時間になった。

きっかけは「なんとなく /plugin を叩いた」だけ

副業でCW（クラウドワークス）のTypeScript案件を週2〜3本こなしてる。コードを書く量は増えてきたのに、作業時間はあまり減っていなかった。

Claude Codeのバージョンを確認しようと claude --version を叩いたとき、ふと /plugin コマンドを試してみた。Discoverタブに公式マーケットプレイス（claude-plugins-official）のプラグインが並んでいた。「あ、これが最初から使えたのか」と知った。

インストールはコマンド1行だった。

/plugin install typescript-lsp@claude-plugins-official

これだけ。3秒で終わる。

入れてよかった3本

① typescript-lsp — 型エラーが自動で出てくる

LSP（Language Server Protocol）というプロトコルを通じて、コードを編集するたびにClaude Codeが型エラー・欠落インポート・構文ミスを自動で拾ってくれるようになる。

導入前
コードを書く → tsc を手動実行 → エラーを確認 → 修正 → また tsc
導入後
コードを書く → 編集した瞬間にClaude Codeがエラーを表示・自動修正

TypeScript案件では1サイクル（コード → テスト → PR）にかかる時間が1.5時間から50分になった。型エラーの確認だけで20〜30分消えてたと気づいた。

インストール後に1つはまったポイントがある。Executable not found in $PATH というエラーが出た。プラグイン本体とは別に、Language Serverのバイナリを自分でインストールしないといけないらしい。

npm install -g typescript-language-server typescript

これを走らせたら動いた。/plugin のErrorsタブにも案内が出るので、エラーが出たらそこを確認すると早い。

② github — ブラウザを開かなくなった

GitHubのMCP統合プラグイン。Issue確認・PR作成・コードレビューがターミナルから完結する。

/plugin install github@claude-plugins-official

CWのWebアプリ案件では、納品時にGitHubのPRをクライアントに確認してもらうフローが多い。今まではブラウザでGitHubを開いて、UIをポチポチしながらPRを作ってた。

今は「このコードの差分でPRを作って」とClaude Codeに頼むと、確認なしで作ってくれる。PR説明文も変更内容を見て自動生成される。毎回15分かかってたPR作業が3分になった。

③ commit-commands — git操作が1コマンド

git add → git commit → git push の3手順を /commit-commands:commit の1コマンドで完結させる。

/plugin install commit-commands@claude-plugins-official

コミットメッセージも変更内容を見て自動で作ってくれる。「feat: ログイン画面のバリデーション追加」みたいなの、毎回考えるのが地味に面倒だったので助かった。

1つはまりポイントがあった。インストール直後にコマンドが出てこない。/reload-plugins を叩かないとプラグインが有効にならなかった。5分ほど「なんで動かないんだ」と悩んだ。

週5時間 → 2時間

正直「使えなかった」プラグインも2本ある

全部がうまくいったわけじゃないので書いておく。

vercel
フロントエンドのデプロイを自動化するやつ。ただCW案件では「デプロイはクライアント側でやる」というケースが大半で、自分が触る場面がなかった。受注形態によっては刺さると思う
pr-review-toolkit
PRレビューを専門エージェントが担当するやつ。1人で案件をやってる自分には「誰かにレビューしてもらう」という状況がなくて使いどころがなかった。チームで動いてる人には合いそう

自分の副業スタイル（1人でCW案件を複数こなす）には上記2本はミスマッチだった。入れるプラグインは自分のワークフローに合わせて絞ると無駄がない。

最初に入れるなら typescript-lsp か commit-commands から

プラグインを全部試す必要はない。自分の案件で使っている言語に対応したLSPプラグインを1本入れるだけで、作業感がかなり変わる。

TypeScript / JavaScript案件なら
/plugin install typescript-lsp@claude-plugins-official
その後 npm install -g typescript-language-server typescript も必要
Python案件なら
/plugin install pyright-lsp@claude-plugins-official
その後 pip install pyright も必要
どの言語でも使えるやつ
/plugin install commit-commands@claude-plugins-official
インストール後に /reload-plugins を忘れずに

インストール後は /plugin の Installed タブで入れたプラグインを確認できる。

自分はこれだけで週3時間の作業時間が減った。月換算で12時間。その分で別の案件を1〜2本追加で受注できるようになった。

プラグインを入れてなかった3ヶ月が本当にもったいなかった。次はプラグインを自作して副業ワークフローに組み込む話を書く予定。
