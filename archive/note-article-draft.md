# Claude Codeで「社内プレゼン品質」のスライド生成器を作った話

社内向けプレゼン資料を3時間で仕上げるための、HTML+Playwrightベースの自動生成プロジェクトを Claude Code で構築した記録。

PowerPointのテンプレ沼に毎回ハマる人向け。

---

## なぜHTMLスライドなのか（技術選定の話）

最初に検討した選択肢と、却下した理由。

### 却下した選択肢

**Marp** — Markdownでスライドが書けるやつ。
- pptx出力すると画像埋め込みが崩れる既知問題
- レイアウトの自由度が低い
- 「社内プレゼン品質」には届かない

**python-pptx / PPTAgent** — pptxを直接生成する系。
- Windows環境でのセットアップが地雷
- pptxの仕様が複雑でレイアウト崩壊しやすい
- そもそも編集するならpptxにする意味がない

**Gamma / Beautiful.ai 等のSaaS** — AI系スライドサービス。
- 情報が外に出る（社内機密NG）
- ブランドカラーや独自レイアウトの再現が弱い
- 月額課金でロックイン

### 採用した構成: 梶谷式 HTML スライド

**1枚の長いHTMLページに16:9のセクションを縦積みする方式。**

- 出力: HTML（発表時） + PDF（配布時） + PNG（サムネ用）
- スクロールスナップ + キーボードナビ（矢印・Space・F でフルスクリーン）
- Playwright で 1920×1080 のPDF/PNGを書き出し
- CSS変数でブランドカラーを差し替え

pptxを捨てる = レイアウト崩壊ゼロ。HTMLなら CSS で完全にコントロールできる。

## プロジェクトの骨格

```
slide-generator/
├── CLAUDE.md                   # プロジェクト指示書
├── LESSONS.md                  # 指摘の蓄積ログ（自己改善用）
├── .claude/skills/create-slide/
│   └── SKILL.md                # ヒアリング→構成案→HTML生成の4ステップ
├── templates/
│   ├── base.html               # HTML雛形
│   ├── design-tokens.css       # デザイントークン（色・フォント・余白）
│   └── layouts/README.md       # 5レイアウトの使用例
├── scripts/
│   └── export-pdf.js           # Playwrightエクスポート
└── output/                     # 生成物
```

### 5つの定型レイアウト（これ以上増やさない）

| レイアウト | 使い所 |
|---|---|
| `slide-hero` | タイトル・セクション区切り・結論 |
| `slide-contrast` | Before/After、課題/解決 |
| `slide-icon-grid` | アジェンダ、3-5点の列挙 |
| `slide-data-callout` | 大きな数字ひとつ |
| `slide-process` | ステップ図（3-5段階） |

**レイアウトを固定する = ダサさの発生源を減らす。** 自由にレイアウトを作らせると必ず崩壊する。

## Claude Code が自動でやる4ステップ

SKILL.md に記述してある生成フロー。

### Step 1: ヒアリング

対話で最低限聞くこと:
1. 目的（聴衆に何を決めてほしいか）
2. オーディエンス（経営層 / エンジニア / 社外 等）
3. 枚数・時間（目安: 5分=5枚、15分=10-15枚）
4. キーメッセージ（1文）
5. 素材（使いたいデータ・エピソード）

固定の質問リストではなく、**案件に応じて省略・追加する**。

### Step 2: 構成案の提示

ヒアリング結果から Markdown で構成案を出す:

```
## 構成案: <タイトル>
想定時間: 5分 / 全6枚 / ブランドカラー: #2cb696

1. [Hero] 表紙
2. [DualCallout] Before/After の数字対比
3. [IconGrid] 時間の内訳
4. [Contrast] 運用の違い
5. [Process] 導入ステップ
6. [Hero] Call to Action
```

**ここで必ずユーザー確認を取る。** 承認されるまでHTML生成に進まない。

### Step 3: HTML生成

`output/<案件名>/index.html` を一発で書き出す。

デザイン制約（厳守）:
- タイトルは日本語15文字以内
- 箇条書きは1行25文字以内、最大5項目
- アイコンは Lucide のみ（他セット混在禁止）
- 色はCSS変数で定義した3色＋グレースケールのみ

### Step 4: PDF/PNG エクスポート

```bash
node scripts/export-pdf.js output/<案件名>/index.html --png
```

Playwrightが各セクションを16:9で書き出し。PDFは印刷用、PNGはサムネ用。

## 自分用のテスト: note-pipeline のクライアント提案資料

プロジェクトが動くか試すため、自分の note 運用パイプラインを売り込むクライアント提案資料を作ってもらった。

**最初の成果物の問題点:**
- 情報量が少なすぎる。抽象的すぎて刺さらない
- 「3.4時間 → 5分」の数字が別々のスライドに散っている
- デザインが素朴すぎる

ここから **修正指示 → 改善 → 再出力** のループが始まる。

### 修正1: 情報密度を上げる

「もっと実際の動作フローとかもろもろ情報量を増やしたい。必要な情報は実際に content-pipeline を見に行って埋めて」

→ サブエージェントに別プロジェクト `content-pipeline/` を読みに行かせ、実データを抽出:
- 100点ルーブリック（30+25+20+15+10の配点）
- 7スウィープ編集プロセス
- AIDA / Star Story Solution フレームワーク
- voice 6要素（語尾・一人称・改行・感情・修辞・噛み砕き）

これを各スライドに流し込んだら一気に説得力が出た。

### 修正2: スライドをマージ

「2枚目と10枚目は合わせて1枚にした方が見やすい」

→ `DualCallout` という新レイアウトを追加。Before/After の数字を1枚で対比できる構成にした。

```
BEFORE — 手作業        AFTER — パイプライン
  3.4h          →         5分
```

### 修正3: デザインをゴージャスに

→ CSS にデコレーションを足す:
- コーナーに `radial-gradient` のblob（opacity 0.08-0.1）
- カード天面に 4px のグラデーションバー
- 大数字は `background-clip: text` でグラデーション文字
- 重要ピルは primary→accent グラデ + white文字 + shadow-lg

平坦な白背景から、存在感のある資料に変わった。

### 修正4: Processレイアウトの下半分がスカスカ

「フローのページ、上側はテキストあるけど下側スカスカで情報量が薄い」

→ 各ステップに追加したCSS:
```css
.slide-process .step ul {
  margin-top: auto;        /* 下部に押し込む */
  padding-top: var(--sp-3);
  border-top: 1px dashed var(--color-border);
}
.slide-process .step .tag {
  background: color-mix(in srgb, var(--color-primary) 14%, transparent);
  padding: 6px 14px;
  border-radius: 999px;
}
```

HTML側は各stepに `<span class="tag">` ピル + `<ul><li>` 3点の具体アクションを追加。

### 修正5: 文字が小さすぎる

→ `--fs-small` (18px) を詳細テキストに流用したら潰れた。1920×1080では最小20pxに揃える。タグピルも13px → 18pxに修正。

### 修正6: 差別化ポイントを専用スライド化

「事実確認スウィープ（全クレームをWebSearch/WebFetchで実照合する仕組み）はうちの差別化ポイントだから、これを紹介するためのスライドを追加して」

→ rubricスライドのfooterに埋めていた要素を、専用スライドに昇格。バッジ + 大見出し + 4カードグリッド + ルールピルで構成。

## 一番重要だった学び: 自己改善ループを作る

6回の修正指示を受けた後、気づいた。

**「同じミスを次回も繰り返す気がする」**

Claude Codeは毎回セッションがリセットされる。今日学んだ「文字は20px以上」「Processは下半分埋める」を明日には忘れている。

そこで `LESSONS.md` を追加した。

```markdown
# スライド生成の教訓ログ

## 追記トリガー
- 「〜が小さい」「〜スカスカ」「〜追加して」等の修正指摘
- デザイン・情報量・レイアウト選択への否定的フィードバック

## 教訓一覧
- [2026-04-14] 情報密度: 初回叩き台は必ず「情報量が少なすぎる」と指摘される
- [2026-04-14] 文字サイズ: 1920×1080で最小20px。--fs-small (18px) は潰れる
- [2026-04-14] Process下半分対策: 各stepに tag + ul 3点を必ず追加
- [2026-04-14] 差別化ポイント: footer埋めせず専用スライド1枚で大きく見せる
- ...
```

そして `CLAUDE.md` と `SKILL.md` に以下を追加:

```
- IMPORTANT: セッション開始時に必ず LESSONS.md を読む
- IMPORTANT: ユーザーから指摘を受けたらその場で LESSONS.md に追記する
```

これで **指摘が資産化される**。次回の作業開始時、Claude Code は過去の教訓を読み込んでから叩き台を作る。同じ指摘を2回受けなくて済む。

## 使い方（完成形）

```
> このプロジェクトで、4月の全社会議用スライドを作って
```

と話しかけるだけで、Claude Code が:

1. LESSONS.md を読み込む
2. 目的・オーディエンス・枚数をヒアリング
3. 構成案を Markdown で提示 → 承認待ち
4. 承認後にHTML生成
5. Playwright でPDF/PNG書き出し

途中で「ここ情報薄い」「文字小さい」と言えば、その場で修正 + LESSONS.md に追記される。

## 得られたもの

- PowerPointを開かない生活
- レイアウト崩壊ゼロ
- 修正指示が蓄積されていく仕組み（= 使うほど賢くなる）
- 40枚デッキが2日 → 3時間

Claude Code は「ツールを作るツール」として使うのが一番強い。
定型業務を1回ちゃんとプロジェクト化しておけば、あとは話しかけるだけになる。

## この記事で触れなかったこと（続編予定）

- Playwright でPDFを1920×1080でズレなく書き出すコツ
- CSS変数設計（`--fs-hero: 96px` / `--fs-h1: 64px` / ...）の考え方
- 梶谷式スライドの構成論（1スライド=1メッセージ）
- Lucide アイコンを inline SVG で統一する理由
