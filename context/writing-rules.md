# ライティングルール

affaan-m/everything-claude-code の article-writing スキル設計思想を content-pipeline 向けに実装。

---

## 非エンジニア向け執筆ルール（全記事必須）

ターゲットは「副業を始めたい非エンジニア・初心者」。エンジニアが読者の前提で書かない。

### 基本姿勢

- **「稼いだ結果→使ったツール」の順番を守る**。ツールの説明から入らない
- 読者は「Claudeって何？」「ターミナルって何？」から始まる人を想定する
- 「エンジニアじゃなくてもできた」が常に伝わるように書く

### 技術用語の扱いルール

初出の技術用語は必ず1文で説明する。2回目以降は略称でOK。

| 用語 | 説明のしかた |
|---|---|
| Claude Code | 「Anthropicが作ったAIアシスタント。チャット感覚で指示すると、コードを書いたり作業を自動化してくれるツール」 |
| ターミナル / コマンドライン | 「テキストで命令を打ち込む黒い画面。怖そうに見えるけど、コピペするだけでいい」 |
| API | 「外部サービスと連携するための接続口。設定は1回やればあとは自動で動く」 |
| MCP / エージェント | 「Claudeに複数の作業を自動でやらせる仕組み」 |
| プロンプト | 「Claudeへの指示文。うまく書くほど出力が良くなる」 |
| スキル / SKILL.md | 「Claudeに覚えさせた手順書みたいなもの」 |

**説明のNG例:**
> Claude CodeはAnthropicが開発したCLIツールで、AIエージェントを活用したコード生成・ファイル操作が可能です。

**説明のOK例:**
> Claude Codeは、チャット感覚でAIに作業を頼めるツール。「この文章を整えて」「この案件に応募文を書いて」と指示するだけで動く。インストールは最初の1回だけ。

### コードブロックの扱い

コードを記事に載せる場合は必ず「**このコードが何をしているか**」を日本語で先に説明する。
コードを見せることが目的ではなく、「こういうことが自動でできる」を伝えることが目的。

```
NG: コードブロックだけ貼って終わり
OK: 「以下のコードを貼り付けると、〇〇が自動で〇〇されます。コピペするだけでOK。」
    → コードブロック
```

### 「難しそう」を先取りして潰す

技術的な工程が出てくる場面では、**事前に「難しくない」と明言してから手順を書く**。

```
NG: 「次に、APIキーを環境変数に設定します。」（難しそうに見える）
OK: 「APIキーの設定は1回だけやればOKで、コピペで終わります。」
    → 手順
```

### 読者への語りかけ方

- 「〜じゃないですか」「自分も最初そうだった」で共感を作ってから解決策を出す
- 「エンジニアの友達に聞かなくていい」「調べなくていい」という解放感を与える
- 失敗・詰まりポイントを隠さず出す（「ここで自分は1週間詰まった」等）

## 証拠優先ルール

セクションは「数字・具体例・コード」から書き始める。

**NG（印象先行）:**
> SNSマーケティングは非常に重要です。積極的に活用することで、多くのメリットが得られます。

**OK（証拠先行）:**
> Xで1万フォロワーを持つアカウントの平均エンゲージメント率は3.2%。同じ情報でも冒頭に数字があると読了率が1.8倍になる（自社計測データ）。

ルール:
- 各セクションの1文目は「数字 / 具体例 / コードスニペット / スクリーンショット説明」で始める
- 「〜は重要です」「〜が必要です」で始めない
- 抽象的な主張をする場合は、その直後に必ず根拠（数字・事例・引用元）を置く

## 禁止フレーズリスト（AI定型表現）

以下のフレーズはそのまま使用禁止。使いたい場合は別の表現に置き換えること。

### 曖昧・逃げ表現
- 「〜と言えるでしょう」 → 断言するか、根拠を示す
- 「〜ではないでしょうか」 → 「〜です」または「〜だと考えます（理由: 〜）」
- 「〜かもしれません」 → 確認できない場合は「〜は未確認だが、〜の観点からは〜が考えられる」
- 「〜と考えられます」 → 「〜です」または根拠付きで使う
- 「〜でしょう」 → 断言するか根拠を示す
- 「〜といっても過言ではありません」 → 言い切れないなら書かない

### 過剰な褒め言葉・誇張
- 「非常に重要」 → 「重要」で十分。または「〇〇の点で重要」と具体化
- 「素晴らしい」 → 何がどう良いかを具体的に
- 「有益な」 → 何がどう有益かを具体的に
- 「革新的な」 → 数字や比較で示す
- 「画期的な」 → 同上
- 「驚くべき」 → 数字で驚かせる

### AI感の強い定型句
- 「ぜひ」 → 「試してみてください」「やってみると〇〇です」
- 「積極的に活用」 → 具体的な使い方を書く
- 「シームレスに」 → 何が繋がっているかを具体的に
- 「効率的に」 → どれだけ効率化されるかを数字で
- 「スムーズに」 → 同上
- 「幅広い」 → 具体的に列挙する
- 「様々な」 → 具体的に列挙する
- 「多岐にわたる」 → 同上
- 「〜することが大切です」 → 「〜してください」または理由付きで
- 「〜を意識することが重要です」 → 「〜してください（なぜなら〜）」
- 「ご活用ください」 → 「使ってください」
- 「お役に立てれば幸いです」 → 削除
- 「参考になれば幸いです」 → 削除
- 「最後に」 → そのまま削除してまとめに入る
- 「いかがでしたか？」 → 削除
- 「まとめると」 → 「結論：〜」か削除
- 「〜について解説しました」 → 削除
- 「〜について説明してきました」 → 削除
- 「以上が〜です」 → 削除

## Voice Extraction Method（affaan-m/article-writing から）

Hyui の声を再現する際、以下の5要素を必ず抽出してからメモに書き起こすこと。
「なんとなく合わせる」ではなく、**5要素を明文化してから書き始める**。

```
[抽出メモ]
① 文のリズム: 1文は何語くらい？短文多用か長文か？段落は何文？
② 形式レベル: どれくらいくだけてるか（口語度）。「〜じゃん」「〜かも」の頻度
③ 修辞的癖: 問いかけ（〜じゃないですか）、括弧使い、体言止め、「みたいな」の多さ
④ 意見の出し方: 「自分はこう思う」をどれくらいはっきり言うか。強め/控えめ
⑤ 技術用語の扱い: どこまで説明するか。初出のみ説明してあとは略称か、毎回説明か
```

## ハイプ語リスト（誇張表現の扱い）

以下の語を使う場合は、**必ず数字・比較・根拠で裏付けること**。

| ハイプ語 | 使う場合の条件 |
|---|---|
| 最強 | 比較対象を明示して数値で証明 |
| 最高 | 何に対して最高かを明示 |
| 完璧 | 使わない。「〇〇の問題を解決した」に置き換え |
| 完全自動 | 人手ゼロで動く工程を具体的に列挙 |
| 爆速 | 具体的な時間を示す（「3分で〇〇」） |
| 劇的に | 「〇〇が△△% 改善した」に置き換え |
| 圧倒的に | 比較数値を示す |

## 声クローンフロー

Hyui の文体を再現するための手順。記事執筆前に必ず実行する。

### Step 1: 文体特徴の抽出（コンテキスト分離が最重要）

`context/voice-samples.md` を読む。

**⚠️ CRITICAL: voice-samples.md のサンプルは「トーン分析の対象のみ」として扱う。**
サンプルの内容・話題・事例はそのまま使用禁止。抽出するのは以下のみ:

```
1. 語尾パターン — 「〜した」「〜だった」「〜ってかんじ」「〜みたいな」etc.
   どの語尾が何%くらい使われているか
2. 一人称 — 「自分」で統一（「僕」「俺」は使わない。Hyuiは女性キャラ）
3. 改行リズム — 1文の平均的な長さ、1段落が何文か、空白行のタイミング
4. 感情の出し方 — 喜び・困惑・驚きをどんな言葉で表すか、強さはどれくらいか
5. 修辞的癖 — 括弧の多さ、問いかけ形式、体言止め、「〜みたいな」等の口癖
6. 専門用語の噛み砕き方 — 初出でどこまで説明するか、略称は使うか
```

サンプルの具体的な話題（「朝起きて〜」「情報系の大学生〜」等）を見て「こういう話を書こう」と考えるのは禁止。あくまで「こういう書き方をしよう」の参考。

### Step 2: 記事前メモ（3行で書く）

執筆前に以下を3行でメモしてから書き始める:

```
今回の声の特徴:
1. 語尾は「〜した」系で統一、「〜です/ます」は最小限
2. 改行は短め（2〜3行で改行）、テンポ感を出す
3. 技術用語は初出だけ説明、以降は略称で
```

### Step 3: 声クローンチェック（具体的フィルター）

生成後、voice-samples.md の任意のサンプル1段落と並べて読む。
以下のチェック項目を**全部確認**して、1つでもNGがあれば該当箇所を書き直す:

**構造AIパターン検出:**
- [ ] 「まず」「次に」「最後に」の三段構成がそのまま使われていないか
- [ ] 「〜ステップ1:」「〜ステップ2:」という見出し付きリストが本文中にあるか（あったら本文に溶け込ませる）
- [ ] 「論理接続詞」（したがって/このように/つまり/よって）が段落冒頭に2回以上あるか（あったら削除）
- [ ] 1段落に「〜こと」が3回以上あるか（あったら言い換え）
- [ ] 「〜です/ます」で終わる段落が4つ以上連続していないか

**感情・テンポチェック:**
- [ ] 文末が「〜です/ます」で連続する段落が3つ以上続いていないか
- [ ] 一番感情が動く文（驚き・共感・発見）が最初の3行以内にあるか
- [ ] 1000字に1回以上、読者に語りかける文（「〜じゃないですか」「〜だと思う」）があるか

**Hyuiテスト（最終判定）:**
声クローンチェック後、記事の冒頭3段落を読んで「Hyui（文系女子大学生）が書いたっぽいな」と思えるかどうかを確認する。男性寄りの表現（「僕」「〜だぜ」等）が1つでもあればNG。思えなければ全体を書き直す。

## 日本語AI感フレーズリスト（affaan-m/article-writing の日本語版）

英語記事でよく使われる AI感フレーズの日本語版。以下が出たら即削除・言い換え。

### 大げさ系（英語の "game-changer" / "rapidly evolving" 相当）
- 「急速に進化している」 → 「〇〇が〇月にリリースされた」と事実で書く
- 「パラダイムシフト」 → 具体的に何が変わったかを書く
- 「ゲームチェンジャー」 → 「〜が変わった。具体的には〜」
- 「新時代の幕開け」 → 削除
- 「革命的な」 → 数字・比較で示す
- 「AIの民主化」 → 誰でも使えるようになった経緯を具体的に

### 弱い接続詞系（英語の "Moreover" / "Furthermore" 相当）
- 「また、〜」が段落頭に3回以上 → 構造を見直す
- 「さらに、〜」が連続 → 事実を並べるか削除
- 「加えて、〜」 → 「〜もある」に短縮
- 「なお、〜」 → 必要なら残すが多用NG

### 論文的接続詞系（本文で使うと堅くなる）
- 「したがって」→ 段落頭では使わない
- 「このように」→ 段落末で使うと説教っぽい。削除が基本
- 「以上のことから」→ 削除
- 「〜という観点から」→ 「〜の点では」に短縮

### 未検証の確信系（英語の "it's worth noting" 相当）
- 「注目すべきは」 → 「〜が面白い」
- 「見逃せないのは」 → 削除して事実を書く
- 「ポイントは」 → 3回以上の連発NG

## Elements of Style 原則（簡潔さのための18ルール要約版）

### 削除のルール
1. 不要な言葉を削る — 「という」「こと」「もの」の多用禁止
   - NG: 「Claudeを使うということは非常に有効なことだと思います」
   - OK: 「Claudeを使うと有効です」
2. 修飾語の削除 — 「とても」「非常に」「かなり」は削る。強調したいなら数字で
3. 冗長な言い回しを短縮 — 「〜することができます」→「〜できます」
4. 受動態の重複を避ける — 「〜によって行われます」→「〜します」

### 構造のルール
5. 能動態を使う — 「Claudeによって記事が生成される」→「Claudeが記事を生成する」
6. 1段落1アイデア — 段落が変わったら話題が変わっている
7. 重要な情報を段落の先頭に置く — 読者は最初の1文で続きを読むか判断する
8. 並列構造を揃える — 箇条書きは同じ文法形式で統一

### 表現のルール
9. 具体的・断言的に書く — 「多くの場合」→「80%のケースで」
10. 抽象名詞を動詞に変える — 「〜の実装」→「〜を実装する」
11. 専門用語は初出で説明 — ただし同じ説明を2回繰り返さない
12. 比喩は1回使ったら変えない — 混合比喩を避ける

### 構成のルール
13. 冒頭で読者に「なぜ読むべきか」を伝える — 最初の3文が全て
14. 結論を先に書く — 理由・根拠・補足の順で続ける（倒置法）
15. 各セクションの最後に次セクションへのブリッジを置く
16. 見出しは「内容の要約」ではなく「読者への問いかけ」または「結論」にする

### 校正のルール
17. 声に出して読む — 詰まった箇所が直すべき場所
18. 書いてから1時間以上空けて再読する — 直後に読むと見逃す

## noteエディタ装飾パターン

過去記事（https://note.com/moyuchi_aistu/n/nb5bffb185a04）から抽出した視覚的レイアウト手法。
記事本文のHTML生成時に以下のパターンを使うこと。

### 冒頭アイランド問いかけ（CRITICAL・全記事必須）

**記事の1行目は必ずこのパターン。`<p>` から始めることは禁止。**

```html
<hr>
<p style="text-align: center;"><b>CWの案件チェック、まだ自分でやっていますか？</b></p>
<hr>
```

問いかけの作り方:
- 読者の「あ、自分のことだ」を引き出す1文
- 「まだ〇〇していますか？」「〇〇、毎回ゼロからやっていませんか？」形式が効きやすい
- 文言は**記事ごとに変える**（固定文言はNG。慣れられる）
- noteの一覧プレビューにこの問いかけが表示される → フックとして機能する

---

### 「アイランド強調」（最重要）

キーとなる数字・事実・フローを際立たせる。

```html
<hr>
<p style="text-align:center"><strong>1本あたりの台本作成時間：約10分</strong></p>
<p style="text-align:center">（従来比 1/10 以下）</p>
<hr>
```

パターン: `<hr>` → 中央揃え+太字の主張 → `<hr>`
効果: 前後と切り離された「島」として視線が止まる

### フロー図（中央揃え+矢印）

工程の流れを示すときに使う。

```html
<p style="text-align:center"><strong>ネタ探し → リサーチ → 構成 → 台本執筆</strong></p>
```

- `→` でステップを繋ぐ
- 太字にして視認性を上げる
- 中央揃えで「図」として見せる

### h2見出しの使い方

セクション区切りはh2（##）を使う。箇条書きで始めず、h2直後に1〜2文の導入を置く。

```
## SKILL.mdの中身はこんな感じ

設定ファイルは1つだけ。チャンネルごとに「誰向けか」「どんな口調か」を書くだけでいい。
```

### 強調パターンまとめ

| 用途 | 書き方 |
|---|---|
| 最重要な数字・結果 | `<hr>` で囲む + 中央揃え + 太字 |
| 工程フロー | 中央揃え + 太字 + `→` |
| セクション導入 | h2 + 直後に1行説明 |
| 補足・注意書き | （括弧）で添える。改行で1段落独立 |
| Before/After | 2段落で並べる。数字を先頭に出す |

### hr連続禁止ルール

アイランドブロックの直後に別の `<hr>` が続く場合、アイランドの**下hrを省略**する。

```html
<!-- NG: hrが2本連続 -->
<hr>
<p style="text-align: center;"><b>30分 → 8分</b></p>
<hr>

<hr>
<p style="text-align: center;"><b>ここから先で手に入るもの</b></p>

<!-- OK: 下hrを省略して繋げる -->
<hr>
<p style="text-align: center;"><b>30分 → 8分</b></p>

<p style="text-align: center;"><b>ここから先で手に入るもの</b></p>
```

### 禁止レイアウト

- 見出し直後にすぐ箇条書きを並べない（導入1文を挟む）
- 箇条書きを3レベル以上ネストしない
- コードブロック（```）内に説明文を書かない。コードと説明は分ける
- `<hr>` を2本連続で並べない（上記hr連続禁止ルールを参照）
