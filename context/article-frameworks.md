## AIDAフレームワーク（記事・ツイートのCTA設計）

### 基本構造

| フェーズ | 目的 | 実装 |
|---|---|---|
| **Attention** | 「自分ごと」にさせる | 冒頭1文で読者の状況・悩みを直撃 |
| **Interest** | 「なぜこれが重要か」を伝える | 読者の言葉で課題を言語化 |
| **Desire** | 「読んだら得をする」と思わせる | 具体的ベネフィット（数字・事例） |
| **Action** | 次の行動を促す | 読み続ける / 購入 / フォロー |

### 記事タイプ別実装例

**速報記事:**
```
A: 「Claude Codeが今日アップデートした。これ見逃すと3日分の損」
I: 新機能でできることが変わった。具体的に何が変わったか
D: この設定1つで作業時間が半分になる
Action: 「全部載せてます、続き読んでください」
```

**実録記事:**
```
A: 「副業0円から3ヶ月で月5万円になった話。再現性あります」
I: 自分と同じ状況（大学生・非エンジニア寄り）の人に向けて
D: やったこと・失敗したことを全部書いた。同じ手順でできます
Action: 「CWのプロフィール文テンプレ、記事の最後で配布」
```

**ノウハウ記事:**
```
A: 「Claudeに記事を書かせると秒でバレる。バレない方法を見つけた」
I: AI記事がスキを取れない理由（と取れる理由）
D: moyuchiの声クローン手法を公開
Action: 「プロンプトテンプレは有料パートで」
```

## Star Story Solutionフレームワーク（実録記事に最適）

### 基本構造

**Star（主人公と目標の設定）:**
- 読者が感情移入できる主人公（自分 = moyuchi）の設定
- 具体的な目標を数字で表す
- 「なぜその目標が必要だったか」背景を短く

**Story（挑戦と失敗の過程）:**
- 最初に試したこと → うまくいかなかった理由
- 感情の起伏を入れる（焦り・諦めそうになった瞬間）
- 読者が「わかる〜」と感じるポイントを1つ以上

**Solution（どう解決したか）:**
- ターニングポイントを1つに絞る
- 読者が真似できるステップ（3〜5ステップ）で説明
- 結果を数字で示す

### 「Claude Codeで月5万円受注した話」への適用例

```
Star:
- 主人公: moyuchi（情報系大学生、プログラミングはできるが実務経験なし）
- 目標: CWで月5万円を副収入にする
- 背景: 就活前にポートフォリオと実績が欲しかった

Story:
- 最初: 普通に応募文を書いて投稿 → 3週間で0件受注
- 失敗1: プロフィールが「大学生です。よろしくお願いします」レベル
- 失敗2: 応募文が「やります！」だけで具体性ゼロ
- 感情: 「才能ある人だけが稼げるのかと思った」

Solution:
- ターニングポイント: Claude Codeで応募文を構造化し始めた
- Step 1: 案件詳細を全文読んでClaude Codeに要件整理させる
- Step 2: クライアントの「本当の悩み」を推測させる
- Step 3: 解決策を具体的な手順で提示する応募文を生成
- Step 4: moyuchiの言葉に書き直す（声クローン）
- 結果: 2週間後に初受注、翌月5万円達成
```

## Before/After構造（数字で示す）

### 基本構造

**Before（問題の状態）:**
- 時間: 「1記事に〇時間かかっていた」
- コスト: 「外注費が月〇万円かかっていた」
- 手間: 「毎回〇〇の作業を手動でやっていた」

**After（解決後の状態）:**
- 具体的数字で示す（「50%削減」より「30分→10分」の方が伝わる）
- 副次的なベネフィットも添える

**Bridge（どうやって変えたか）:**
- 読者が再現できるステップ（多くても5つ）
- 各ステップに所要時間の目安を書く

### note記事での視覚的表現（blockquoteを使用）

```html
<!-- Before/After を視覚的に見せるパターン -->

<blockquote>
<strong>Before:</strong> 1記事の執筆に平均4時間。調査2時間 + 構成1時間 + 執筆1時間
</blockquote>

↓ Claude Code + content-pipeline 導入後

<blockquote>
<strong>After:</strong> 1記事45分。調査15分 + 構成5分 + 執筆25分（3.5倍速）
</blockquote>
```

## Picture-Promise-Prove-Pushフレームワーク（ノウハウ記事の説得力強化）

### 基本構造

| フェーズ | 目的 | 実装 |
|---|---|---|
| **Picture** | 読者の共感を得る | 「そう！まさにこれ！」と思わせる状況を描写 |
| **Promise** | 期待を作る | 「この記事を読めば〇〇できる」を明言 |
| **Prove** | 信頼を作る | 証拠（データ・事例・スクリーンショット） |
| **Push** | 行動を促す | note購入 / フォロー / コメント |

### 実装テンプレート

```
Picture:
「AIで記事書いてもスキがつかない。何度直してもなんかAI臭い。
そもそも自分の文体って何なんだろうって迷子になる。」
（← 読者が「わかる」と思う状況を1段落で）

Promise:
「この記事を読めば、Claudeに自分の文体を完全に真似させる
プロンプト設計ができます。最後まで読めばテンプレートも持って帰れます。」

Prove:
・before/afterの文体比較スクリーンショット
・「moyuchiが書いた」と思われた読者のコメント
・スキ数の変化グラフ

Push:
「全テンプレートは有料パートに置いてます（500円）。
無料パートだけでも骨格は理解できます。」
```

## note SEO 基本原則

### タイトル設計
- 検索キーワードを**タイトルの前半30文字以内**に含める
- 数字を入れる（「5つの」「3ヶ月で」「月5万円」）
- 読者の悩みを直接表現する（「〜できない」→「〜できるようになる方法」）
- 禁止: 抽象的なタイトル（「Claudeの使い方」より「Claude Codeで副業受注した方法」）

タイトル生成パターン:
```
[数字] + [対象ユーザー] + [具体的な結果/手段]
例: 「大学生が3ヶ月でCWから月5万円受注した全手順」

[悩み] + [解決方法]
例: 「AI記事がスキを取れない理由と声クローンで解決した話」

[意外性/数字] + [内容]
例: 「45分で1記事。Claude Codeのコンテンツパイプライン全公開」
```

### 本文SEO
- **冒頭200字**でキーワードを自然に使う（詰め込みNG）
- **h2見出し**にもキーワードを含める
- 内部リンク: 関連する自分の記事へのリンクを2〜3本入れる
- 文字数: noteのSEOは最低1500字推奨（ノウハウ記事は3000字〜）

### タグ戦略
- タグは5〜10個（多すぎると分散する）
- 競合調査: 上位5記事のタグをリストアップして重複が多いタグを選ぶ
- 大タグ（「副業」「AI」）と小タグ（「Claude Code」「コンテンツ制作」）を混ぜる
- 自分の記事シリーズ用のオリジナルタグを1つ育てる

### 競合調査チェックリスト
- [ ] 同じキーワードで上位5記事を読む
- [ ] 共通して使われているタグを記録
- [ ] 上位記事にない情報・視点を1つ以上用意する
- [ ] スキ数・コメント数から「何が刺さっているか」を分析
- [ ] タイトルパターンを参考に（コピーNG、パターンを参考に）
