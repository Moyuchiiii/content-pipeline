副業でClaude Codeを長時間動かしていたら、突然「さっきの設計、何でしたっけ」と返ってきた経験ありませんか？

先週、クラウドワークスの案件作業中にやられた。

3時間かけて要件を整理して、DBスキーマを一緒に設計して、コードを書かせて—4時間目に入ったとき、Claude Codeが自動でコンテキストを圧縮した。そのあと「さっき実装したDBのusersテーブルに合わせて〜」と言ったら「DBの構成を教えていただけますか」と返ってきた。

30分かけて同じ説明をし直した。同じ案件でこれが2回あった。

最初は「記憶力が悪いな」くらいに思っていたけど、原因は別にあった。コンテキストの上限に達したとき、Claude Codeが何を圧縮するかを完全に自動判断していたせいだった。これは設定でコントロールできる。

## コンテキスト切れって、何が起きてるのか

Claude Codeは会話の全履歴をトークンとして保持している。長時間使うと上限（20万トークン前後）に近づいてきて、95%を超えたタイミングで「auto-compact」が自動実行される。

auto-compactは今までの会話を要約して圧縮する仕組みで、コンテキストを軽くすることで作業を継続できるようにしてくれる。問題は「何を要約に残すか」をClaudeが自動判断するという点で、作業に必要なDBスキーマや変数名、ファイルパスが要約から抜け落ちることがある。

副業で3〜4時間の長いセッションをやると、この問題は必ずどこかで発生する。「さっき決めたこと」がなかったことになるので、作業効率がガタ落ちする。

## 最初にやっていた（ダメな）対策

コンテキスト切れが起きるたびに `/clear` で全リセットしていた。

これは最悪の対策だった。クリアすると全部消えるので、毎回「案件の概要は〜で、使っているスタックはNext.js+PostgreSQLで、今はXXフェーズです」と一から説明し直さないといけない。1回15〜20分のロスが出る。

同じ案件で3〜4回繰り返すと、1回の作業セッションで1時間近くが「説明し直し」に溶ける。これで副業の時給を計算したら、かなり非効率だった。

## CLAUDE.mdに「compact instructions」を書いたら変わった

Claude Codeの公式ドキュメントにひっそり書いてあったのがこれ。auto-compactが動くとき、何を要約に残すかをCLAUDE.mdで事前指示できるらしい。

CLAUDE.mdの末尾に追記するのはこれだけ。

```
# Compact instructions

When you are using compact, please focus on:
- Current task and its status
- DB schema and data structure
- Variable names and file paths in use
- Decisions already made (and why)
```

CLAUDE.mdはauto-compact後もリセットされずに残るので、この指示が毎回読まれる。auto-compactが走るとき、Claudeはこの指示に従って「要約に残すべき情報」を優先してくれる。

案件ごとに内容を変えてもいい。バックエンドの案件なら「APIのエンドポイント仕様」を、フロントなら「コンポーネントの設計方針」を優先するように書く。

設定してから1週間、同じ副業案件で試した。コンテキスト圧縮後に「DBスキーマが何か」を忘れることがなくなった。圧縮後もClaude Codeが文脈を保ってくれるので、作業の流れが途切れない。

## /compact を自分で打つタイミング

auto-compactを待つだけでなく、機能単位の区切りで手動実行するのも効果的だった。

`/compact` コマンドは、指示付きで実行できる。

```
/compact Focus on the current task progress, DB schema, and file structure
```

自分がやってるのは大きく2パターン。

**機能の区切りで打つ:** 機能Aの実装が完了したタイミングで手動compact「現在の完了状況とコードの状態を残して」→ 機能Bの詳細をインプットする

**2時間に1回打つ:** 長時間セッションの場合、コンテキスト使用率が低くても2時間に1回は手動compactを打つ。コンテキストが重くなる前にリセットした方が要約の精度が上がる

## Before / After

**対策前:** 4時間の副業作業で2〜3回コンテキストリセット。説明し直しで計60分ロス。

**対策後（CLAUDE.md + 手動/compact）:** 同じ4時間作業でコンテキスト圧縮後の「覚えていない問題」がゼロ。説明し直し時間がほぼ消えた。作業効率が体感で15〜20%改善。

## 4/1に出たAPI版の新機能

実は4月1日にAnthropicがAPIベータとして「Context Compaction API」を公開した。

これはClaude Code内のauto-compactとは別物で、自分でAPIエージェントを組んでいる場合に使える機能。リクエストに `compact-2026-01-12` というベータヘッダーを追加すると、コンテキストが設定したトークン数を超えたタイミングで自動要約が走る。

対応モデルはClaude Opus 4.6のみ。ベータ機能のため追加課金なし。要約に残す内容を `instructions` パラメータでカスタマイズ可能。デフォルトトリガーは15万トークン（最低5万以上で設定可）。

副業でClaude Codeをそのまま使っている場合は、前述のCLAUDE.md設計 + 手動compactで十分対応できる。MCPサーバーやAPIエージェントを自作している場合は、こっちを使えばコンテキスト管理をより細かく制御できる。

## つまずいたポイント

**CLAUDE.mdが長すぎると逆効果:** Compact instructionsを書くついでにCLAUDE.mdにいろいろ詰め込んだら、CLAUDE.md自体がコンテキストを大量消費するようになった。公式推奨は500行以下。必要最小限に絞ること。

**/compactに指示なしは「なんとなく圧縮」になる:** ただ `/compact` と打つだけだとClaudeが自動判断で要約してしまう。「コード変更の記録を残して」「決定事項を優先して」など、何を残してほしいかを毎回一緒に書くようにした。

**Compact instructionsは英語で書く:** 日本語で書いたら要約の精度が落ちた気がした（体感）。compactの処理はモデルが英語で考えているためか、英語指示の方が安定した。

CLAUDE.mdの設定は一度やれば終わりで、あとは作業区切りに手動compactを打つ習慣だけ。副業でClaude Codeを長時間使うなら、先に入れておいた方がいい設定だった。

フォローしてもらえると、Claude Codeで稼ぐための実録をこれからも届けます。
