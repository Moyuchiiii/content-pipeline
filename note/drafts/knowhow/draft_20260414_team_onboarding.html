自分がClaude Codeで何をどう使ってるか、ちゃんと説明できますか？

副業で毎日触ってるのに、いざ「どんな設定で、どのコマンドを、どういう順で使ってるか」を聞かれると、意外と答えられない。自分がそうだった。

CLAUDE.mdもdeny rulesもskillも、全部「その時の気分」で足していった結果、頭の中にしか記録がない。新PCに移ったら再現に30分。弟に「Claude Codeで副業始めたい」と言われた時は、説明だけで1時間半かかった。

4/10に出た/team-onboardingを、自分に向けて使ってみた。副業は1人プレイだから本来の用途とは違うけど、「自分のワークフローを自分に説明し直す」ツールとして使える気がしたから試した。

結果、CLAUDE.mdに書き忘れてた3つの抜けが出てきた。これはマジで発見だった。

/team-onboardingが何をやるのか

Claude Code v2.1.101（2026-04-10リリース）で追加された新コマンド。ローカルのClaude Code使用履歴を分析して、チームメンバー用のオンボーディングガイドを自動で生成する。

入力にするもの
ローカルのセッション履歴・CLAUDE.md・memory・設定（settings.json等）
出力されるもの
「この人はこういう使い方をしてる」という棚卸しドキュメント。よく使うコマンド・skill・hook・deny rulesの傾向が整理される

本来はチームで共有するためのツールだけど、出力ドキュメントを自分で読むと「自分が無意識にやってること」が文字になって出てくる。この効果が予想外に大きかった。

実際に動かしてみた

やることはClaude Codeのターミナルで1行打つだけ。

/team-onboarding

30秒ほど待つと、Markdownのガイドが生成される。自分の場合、長さは約1,800字。中身は大きく4ブロック。

① よく使うコマンドとskill
頻度順で整列。自分の場合は /loop が圧倒的1位、次に /compact、3位が自作の note-run skill
② CLAUDE.mdの要約
何を禁止してて、何を自動許可してるか、どのツールと連携してるか
③ 典型的なセッションの流れ
「プロジェクト起動 → /compact → skill呼び出し → コミット」みたいな実際の作業順序
④ 新規メンバーが最初にやるべきこと
CLAUDE.mdを読む、deny rulesを確認する、skill一覧を見る、の順で書かれてた

見えた「3つの抜け」

一番の収穫はここ。生成されたドキュメントを自分で読んで、「そうだ、これCLAUDE.mdに書いてなかった」と気づいた項目が3つあった。

抜け1: deny rulesの「理由」が書いてない

自分のCLAUDE.mdには rm -rf や git push --force がdenyリストに入ってる。でもなぜdenyなのかがどこにも書いてない。

onboarding出力では「このdeny rulesが何を守ってるのか説明がない」と遠回しに指摘してきた。実際、弟に副業を教える時にdeny rulesの話をしたら「なんでこれダメなの？」と聞かれて説明に5分使った。理由が書いてあれば、その5分は丸ごと消える。

抜け2: skillの組み合わせパターンが暗黙知

自分はnote記事を書く時、/collect-stats → /note-run → /content-engine → /x-run の順で動かす。これがワンパターン化してるのに、CLAUDE.mdには書いてなかった。

onboarding出力はセッション履歴からこの順序を拾って「この4つのskillは連続で使われる傾向がある」と書いてきた。これはそのままCLAUDE.mdの「典型ワークフロー」セクションに貼れる。

抜け3: 案件タイプ別の分岐が記録されていない

CW案件なのかococナラ案件なのかで、使うskillが微妙に違う。CWの時は応募文テンプレ、ococナラの時はヒアリング優先。これも頭の中にしかなかった。

onboarding出力は「案件タイプごとに違う動きをしている」ことまでは検出してくれた。ただ「なぜ違うのか」の判断基準は人間が書くしかないので、ここは自分で追記した。

Before/After

Before
・新PCセットアップ時、ワークフローの再現に30分
・弟・友人に教える時、1時間半の説明
・CLAUDE.mdは約900字、「禁止事項の羅列」のみ

After
・棚卸しドキュメント生成30秒
・人に渡す時はガイドを投げるだけ（最初の5分で質問がほぼ消える）
・CLAUDE.mdは約2,100字、「なぜ」「典型フロー」「案件分岐」が追加された

つまずいたこと

セッション履歴が少ないと出力が薄い

最初に別のプロジェクトで試したら、5セッション分しか履歴がなかったせいで出力が300字しかなかった。メインで使ってるプロジェクトで動かすのが正解。目安は30セッション以上。

出力をそのままチーム共有するのは危険

自分の場合、履歴にクライアント名がうっすら混ざってた（ファイルパスの一部に含まれてた）。人に渡す前にClaude Codeに「固有名詞を全部伏せて」と指示して1パス通すのが安全。

/team-onboardingはチームプラン限定ではない

コマンド名で勘違いしそうだけど、個人プランでも普通に動く。チームじゃなくても使えるのがポイント。

副業1人プレイでも使う価値がある3つの理由

本来はチーム向けだけど、1人副業でも以下3つの使い道がある。

① CLAUDE.md改善の材料になる
自分の使い方を客観的に文字化してくれる。何が書かれてないかが浮き彫りになる
② 新PC・新プロジェクトへの移植が速くなる
生成されたドキュメントをそのまま次の環境の初期設定ガイドにできる
③ 将来の「教える」素材になる
副業を誰かに教える時、事例付きのガイドをゼロから書かずに済む

個人的には①が一番大きい。自分のワークフローを「書かれた形」で見ると、足りないものが一瞬でわかる。これはコーディングでいうlinterに近い感覚。

まだ手を出してないこと

出力ガイドを定期的に再生成して差分を見れば、「最近の自分は何が変わったか」も追えるはず。月1くらいで/team-onboardingを回して、前回との差分をCLAUDE.mdに反映するフローを次の1ヶ月で試す予定。

副業を続けるほど設定は増える。増えた分を頭の中だけで管理するのは限界がある。このコマンドは「自分のClaude Code資産を棚卸しする仕組み」として使えると思う。

副業を2ヶ月で月14万まで持っていった時の全ルートは、別記事にまとめた。CLAUDE.md設計・応募文テンプレ・案件タイプ別フローまで全部公開してるので、/team-onboardingで見えた「抜け」を埋めたい人はこっちを読むと早い。
