副業の案件、PR待ちの間ずっと手が空いていませんか？

CWで2〜3本の案件を並行してるとき、こういう状況よくあった。

案件Aのコードを書き終えてPRを出す。CI待ち。この間、案件Bに集中したいのに「CIどうなったかな」が頭をちらついて集中できない。あるいは単純に、CI失敗してたのに30分後まで気づかなくて無駄に時間が経つ。

週で合計すると、PR待ち・確認・CI再実行みたいな作業で3時間くらい飛んでた。

Claude Code Desktopに本格的に切り替えてから1ヶ月。今は「PR待ち時間」という概念がほぼなくなった。

Claude Code Desktopって何が違うの

最初に言っておくと、Claude Code DesktopはターミナルのClaude Codeと中身は同じ。デスクトップアプリのUIから操作できることと、それに付随していくつかの追加機能がある点が違う。

副業で使い始めて「これは使える」と感じた機能が4つある。

Server Preview
devサーバーをデスクトップ内で自動起動して、ClaudeがUIを見ながらコーディングする
Visual Diff Review
プッシュ前に「Review code」ボタンを押すだけでClaudeがコードをチェック、指摘をインラインで入れてくれる
PR Monitoring
PRを出した後、ClaudeがバックグラウンドでCI結果を監視して、失敗したら自動で修正を試みる
Parallel Sessions
複数のセッションを独立して走らせられる。Gitのworktreeが自動で切られるのでコードも干渉しない

副業ワークフローがどう変わったか

PRを出したら別案件に全集中できるようになった

いちばん効いたのはPR Monitoring。PRを出した後にセッション上にCIステータスバーが出てくる仕組みで、ここで「Auto-fix」をONにしておくと、CIが失敗したときにClaudeが自動でエラーを読んで修正を試みてくれる。

「Auto-merge」まで入れると、全チェックが通ったら自動でマージまでやってくれる。自分は今のところAuto-fixだけ使って、Auto-mergeは手動にしてる。最終確認だけは自分でやりたいので。

CIが走ってる間に別のセッションに切り替えて案件Bを進める。案件AのCIが通ったらデスクトップ通知が来る。失敗しててもClaudeが直してるから、自分は通知を受け取るだけ。

デザインの実装確認でブラウザを切り替えなくなった

CW案件はフロントのUI実装をやることが多い。以前は「実装 → ブラウザで確認 → 問題あったら修正 → また確認」のループを全部手動でやってた。

今はデスクトップのPreviewパネルにブラウザが埋め込まれてて、ClaudeがUIを見ながら「ここレイアウト崩れてる」を自分で気づいて直してくれる。ClaudeがスクリーンショットとってDOMをチェックして、問題があれば自分で修正する。

「Auto-verify」という設定がデフォルトでONになってて、Claudeがファイルを編集するたびに自動でPreviewを確認してくれる。自分が「確認して」と言わなくてもいい。

プッシュ前のコードチェックが5分で終わる

Visual Diff Review。Claude Codeが変更を加えた後、diffの右上に「Review code」ボタンが出てくる。これを押すとClaudeが変更全体をレビューして、問題のある行にインラインでコメントを入れてくれる。

指摘されるのは「コンパイルエラー、ロジックのバグ、セキュリティ問題、明らかなバグ」に絞られてる。スタイルとかフォーマットは指摘しない。クライアントに渡す前の最終チェックとして使ってる。

3案件を同時に進める構成

Parallel Sessions。「+ New session」を押すと新しいセッションが立ち上がる。Gitリポジトリの案件だと、セッションごとにworktreeが自動で切られるので、別のセッションで変更しても干渉しない。

今の自分の使い方はこんな感じ。

セッションA
メインの作業案件。コードを書いてPRを出す
セッションB
別の案件。セッションAのCI待ち中に進める
セッションC
定型化できたタスクをAuto Modeで流しっぱなし

3セッションが同時に動いてても、それぞれのコードが干渉しない。案件Aの変更が案件Bに混入することがない。これが地味にストレスフリー。

使い始めてつまずいたこと

PR MonitoringはGitHub CLIが必要

最初、PR Monitoringを使おうとしたら動かなかった。ドキュメントを読んだら「GitHub CLI（gh）がインストール&認証済みであること」が条件だった。

自分はghを使ってなかったので、

brew install gh
gh auth login

してから再試行したら動いた。Windowsの場合は

winget install --id GitHub.cli

で入る。最初に「なんで動かないんだ」となったのはここだけ。

Parallel SessionsはGitなしプロジェクトで使えない

GitリポジトリじゃないフォルダでParallel Sessionsを試したら、worktreeが使えずセッション間でファイルが干渉した。セッション分離にはGitが必要という仕様。

CW案件は最初からGit管理してれば問題ない。既存プロジェクトでGitを使ってなかったら、git initしてから使うといい。

1ヶ月使ってみた結果

PR待ち時間 週3時間 → 0

同時進行案件数 1〜2本 → 3本

PR待ちがゼロになったというより、「待つ」という行為がなくなった感覚に近い。CIが走ってる間に別の案件を進めて、通知が来たら確認するだけ。

以前は「複数案件の同時進行は自分には無理」と思ってたんだけど、Parallel Sessions + PR Monitoringで3案件が同時に動くようになった。案件数が増えると単純に月収が増える構造なので、これは地味に大きい。

セットアップは10分以内で終わる。claude.comからDesktopアプリをダウンロードして、GitHub CLIを入れてgh auth loginするだけ。
