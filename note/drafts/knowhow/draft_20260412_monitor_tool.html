CIが終わるの、毎回ターミナル見ながら待ってませんか。

副業で案件を2〜3本並行してると、地味にしんどいのがCI待ちの時間。

PRを送って、GitHub Actionsが走り始める。完了まで大体8〜12分。その間、別の案件の作業に移るんだけど、なんか気になってターミナルを確認しに行く。まだ終わってない。戻る。また5分後に確認。終わってる。マージしてクライアントに報告。

これが1案件で1日4〜5回。2案件並行だと1日2時間近く「CI気にしながら作業する時間」に使ってた。週で換算すると5時間前後。

4月9日にClaude Code v2.1.98がリリースされて、Monitorツールが追加された。2日使ってみたら、この「CI気にしながら時間」がほぼなくなった。

Monitorツールって何者か

一言で言うと、「バックグラウンドで動くスクリプトの出力を、リアルタイムでClaudeに流す仕組み」。

Claudeに「devサーバーを監視して、ビルドエラーが出たら教えて」と頼む。Claudeがバックグラウンドスクリプトを書いて実行する。そのスクリプトのstdout（標準出力）が1行ずつClaudeに届く。エラーのキーワードを検知したら、Claudeがこちらに教えてくれる。

/loopが「定期的に同じことをさせる」機能なら、Monitorは「何か起きたときに反応する」機能。ポーリング（定期確認）じゃなくてイベント駆動なので、何も起きてない間はトークンをほぼ消費しない。

使い方のイメージ
特別なコマンドはない。Claudeとの会話の中で「〇〇を監視して、〇〇が起きたら教えて」と指示するだけ。ClaudeがMonitorツールを使ってバックグラウンドスクリプトを動かす仕組みになってる。

副業で試した3パターン

GitHub Actions CI監視

一番よく使ってるのがこれ。PRを送った直後にClaudeに頼む。

CIが完了（pass/fail問わず）したタイミングでClaudeが教えてくれる。passなら即マージ。failなら内容を確認してすぐ修正。今まで「CIどうなったかな」と気にしながら別作業してたのが、「通知が来たら対応」に変わった。集中が途切れなくなった。

devサーバーのビルドエラー監視

Next.jsの開発中に使うパターン。devサーバーを立ち上げてる間、コード変更のたびにビルドが走る。エラーが出たときだけ教えてもらえれば、成功してる間は別作業に集中できる。

「devサーバーを起動して、エラーが出た場合のみ教えて。ビルド成功はスルーでいい」と伝えると、本当にエラーのときだけ割り込んでくれる。

PRへのレビュー到着監視

クライアントがPRをレビューするのを待つシーンで使った。新しいコメントがついたタイミングで教えてくれるので、そのまますぐ返答できる。GitHub CLIのインストールが前提になるが、副業でCWのPRを受け取るフローなら入れておいて損はない。

使う前と後

導入前
CIの完了を確認するため、1案件あたり1日4〜5回ターミナルに戻る。2案件並行で週5時間前後が「確認作業」に消える
↓ Monitorツール導入後
導入後
通知が来るまで別作業に集中。週5時間のうち4.5時間が返ってきた。0.5時間はMonitorの設定と確認対応に使う

数字で言うと「週4.5時間の削減」。もっと大事なのは、集中が途切れる回数が激減したこと。確認のたびにコンテキストが切れてたのがなくなって、コードを書く密度が体感で上がった。

つまずいたこと（正直に書く）

条件指定が曖昧だと全部通知が来る

最初に「CIの状況を監視して」とだけ伝えたら、ログの1行ずつに反応してきた。「Step X passed」のたびに通知が飛んできて逆に邪魔になった。

「CIが完了したとき（successかfailure）だけ教えて」と明示する必要がある。通知の粒度を自分で指定しないと、全出力に反応する設定になる。

v2.1.98未満では動かない

Monitorツールはv2.1.98から追加された機能。それより古いバージョンだと「MonitorToolが見つかりません」みたいなエラーになる。

ターミナルで claude --version を確認して、2.1.98未満なら claude update を先に実行する。これだけで動く。

実際にClaudeに渡したプロンプト

GitHub Actions CI監視プロンプト

MonitorツールでGitHub ActionsのCIを監視してください。
リポジトリは現在のディレクトリです。
直近のPRのCIが完了（success または failure）したタイミングだけ教えてください。
進行中のステップは通知不要です。
gh コマンドを使って10秒ごとにステータスを確認してください。

ポイントは「完了したときだけ」という条件と「進行中はスルー」という明示。これがないと全ステップに反応してくる。

devサーバーのビルドエラー監視プロンプト

MonitorツールでNext.jsのdevサーバーを起動して監視してください。
コマンドは「npm run dev」です。
標準出力に「Error」か「Failed」が含まれる行が出たときだけ教えてください。
ビルド成功・ホットリロード成功は通知不要です。

「"Error"か"Failed"が含まれる行」というキーワード指定がコツ。stdout全体をClaudeに判断させると反応が遅くなるし、トークンも消費する。キーワード絞り込みを自分で指定するのが正解。

PRレビュー到着監視プロンプト

MonitorツールでPR番号 {PR番号} へのレビューを監視してください。
「gh pr view {PR番号} --comments」を30秒ごとに実行して、
新しいコメントやレビューが追加されたか確認してください。
現在のコメント数は0件です。
この数が増えたときだけ教えてください。

「現在のコメント数は〇件」と初期状態を伝えるのが重要。伝えないと既存のコメントを毎回通知してくる。PRを送るタイミングで「今0件です」と添えてセットで使う。

/loopとの組み合わせ（2段構成）

自分が今使ってる一番強い設定が、Monitorで監視して、対応処理を/loopで動かす組み合わせ。

Monitorで「何か起きた」を検知 → /loopで「対応タスク」を自動実行

CIのpassを検知したら自動でPRのマージコメント下書きを生成させる設定にしてる。CI完了 → 自分が確認 → マージ → クライアントへのコメント下書きを見て微調整 → 送信、という流れが半自動になった。

# /loop との組み合わせのイメージ
1. Monitor: CI完了（success）を検知
2. Claudeが通知「PRのCIが通りました。マージしますか？」
3. 「はい」と答えるとマージ実行 + クライアントへのコメント下書きを生成
4. 自分は内容を確認して送信するだけ

個別に動かすより確認ステップが1回減る。案件が増えるほど効いてくる。

向かなかった2シーン

Slackメッセージの監視
ClaudeにSlack APIを叩かせてメッセージを監視しようとしたが、OAuth認証の設定が面倒すぎて断念。Slackの通知をそのまま使う方が速い。Monitor経由でSlackを監視するメリットは薄い。
ディレクトリのファイル変更監視
「新しいファイルが来たら処理して」という用途で試したが、Monitorが頻繁に誤検知した。ファイル変更監視はfswatchやinotifywaitなど専用ツールの方が安定する。MonitorはCIログ・サーバーログのようなテキスト出力との相性が良い。

要するにMonitorが強いのは「テキストストリームから条件に合う行を拾う」ユースケース。ファイル変更やAPIポーリングはほかのツールに任せた方がいい。
