ネタ出し・リサーチ・執筆・サムネ作成、毎回ゼロからやっていますか？

このnote記事、Claude Codeが書いた。

正確には「/note-run」と打ってGoを出したら、ネタ探しからリサーチ・HTML生成・Notionへの登録まで全部勝手に終わった。自分がやったのはトピックの確認と最後のnoteへのコピペだけ。

「その結果どうなったか」はひとつ前の記事に書いた。今回は「どうやって作ったか」を全部見せる。SKILL.mdの設計思想、Notionの2DB構造、9フェーズの処理順序——ぜんぶ実物ベースで解説する。

仕組みの全体像

核心は3つ。Claude CodeのSKILL.md、Notionの2DB設計、そして9フェーズの処理順序。これを組み合わせた自作パイプラインが「content-pipeline」という名前でローカルで動いてる。

トレンドスキャン → トピック選定 → 競合調査 → 執筆 → 品質チェック → 保存 → Notion更新

このフローをSKILL.mdに書いておくと、「/note-run」を打つだけでClaude Codeが上から順に実行してくれる。1コマンドで完結する仕組みはここから来てる。

SKILL.mdは「AIへの仕事の説明書」

Claude CodeのSKILL.mdは、スラッシュコマンドが何をすべきかを書いたMarkdownファイル。「/note-run」を打つとClaude CodeはこのSKILL.mdを読み込んで作業を始める。

最初は「note記事を書いて。文体はこのサンプルに合わせて」みたいな1行プロンプトで試してた。結果は毎回ブレブレで、どこかで見たようなAI文章しか出てこなかった。

変わったのは「考える順番」を書いたとき。「最初にパフォーマンスデータを読む → 次にトレンドをスキャンする → その後でスコアリングする」という処理の流れをSKILL.mdに明文化した。それだけで出力の一貫性が段違いになった。

SKILL.mdに書くべきもの
このコマンドの目的と完了条件
各フェーズで「何を・どの順番で・どのファイルを参照して」実行するか
出力の形式と保存先
エラー時の対処方法

「細かく書きすぎでは？」と思うくらいが正しい。曖昧さを残すとAIは最もラクな選択肢、つまり汎用的で個性のない内容を選ぶ。指示を具体化するほど、出力が人間らしくなる。

9フェーズの設計と、処理順序が重要な理由

/note-runは9つのフェーズで動く。順番には意図があって、「先に過去データを読まないとスコアリングが機能しない」「先にリサーチしないと執筆がブレる」という依存関係がある。

Phase 0
過去記事のパフォーマンスデータを読み込み、「何が刺さったか」を自動分析する。スキ数・転換率・記事タイプのパターンを抽出して、Phase 2のトピック選定スコアリングに反映させる
Phase 1〜2
WebSearchで直近24〜48時間のClaude関連ニュースを収集し、5項目スコアリングでトピックを自動選定する。20点以上なら有料、15〜19点なら無料、14点以下は見送り。バランス制御（直近5本で有料が多い場合は今回無料に切り替え等）も自動で入る
Phase 3〜4
note.comで競合記事を調査して差別化ポイントを特定。タイトルを5候補生成して20点満点のスコアリングで最適なものを選ぶ
Phase 5〜6
HTML形式で執筆後、7スウィープ編集と100点ルーブリックで品質チェック。60点未満なら自動rewrite（最大2回）。さらにすべての事実クレームをWebFetchで一次情報確認する
Phase 7〜9
SEO最適化・Canva MCPでサムネ自動生成・Notionに記録・PDCAフィードバック更新

この流れをSKILL.mdに書いておくと、毎回「次に何をすべきか」を指示しなくていい。Claude Codeが順番通りに動いて、最後にファイルが保存されて終わる。

Notionが「自動学習する編集部」になる

2つのDBを使ってる。ネタ帳と記事管理DB。

ネタ帳はスキャンで見つかったトピック候補を「未使用/採用済み/見送り」で管理するもの。見送った理由も記録するので、同じネタを次回また候補に上げることがなくなる。

記事管理DBは公開後のPV・スキ数・転換率を記録して、Phase 0の「過去データ分析」に使う。

このループが回ると起きること
記事公開 → PV・スキ数をNotionに記録 → Phase 0が成功パターンを抽出 → 次の記事のトピック選定スコアリングの精度が上がる

最初の頃は「これで記事になるの？」というトピックを選んでいた。今は自分が思いつかなかった切り口を提案されることもある。積み上げが機能してきてる感じがする。

実際につまずいた2つのこと

最初の数ヶ月で何度も引っかかった問題がある。

1つ目はコンテキスト枯渇。SKILL.mdを1セッションで全実行しようとするとPhase 6あたりで文脈保持が限界に来る。解決策は/compactコマンドを中間に挟むことと、CLAUDE.mdに「セッション再開時に引き継ぐべき情報」を書いておくこと。この仕組みをどう設計するかはまた別記事で書く予定。

2つ目は事実確認の甘さ。WebSearchで拾った情報を鵜呑みにして、存在しない機能について書いてしまったことがある。今のPhase 6には「記事内のすべてのクレームをWebFetchで一次情報確認する」というステップがある。これで事実確認のmissが激減した。

まとめと次回予告

SKILL.mdの設計思想は「AIへの仕事の説明書を書く」こと。「何を・どの順番で・どのファイルを参照して」を書けば書くほど出力が安定する。1行プロンプトで使うフェーズはとっくに終わった。

NotionのDB設計は記事を管理するためだけでなく、「過去データを読み込んでスコアリングに反映させる学習ループ」を作るためにある。この設計が入って初めて「毎週打ち込むだけで動く」が成立する。

次回はnote記事からXスレッドへの自動変換の仕組みを書く予定。x-runスキルの設計もcontent-pipelineの一部で、noteと同じフレームワークで動いてる。

SKILL.mdをさらに進化させた「副業ワークフローの自作プラグイン化」もまとめてある。git clone 1回で別PCに10秒で移植できる仕組みで、content-pipelineとも相性がいい。
