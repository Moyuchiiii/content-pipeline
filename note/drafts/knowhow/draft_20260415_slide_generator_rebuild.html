なぜPowerPointを捨ててHTMLで作ることにしたか

最初にMarp・python-pptx・Gamma・Beautiful.aiを全部検討した。全部却下した理由がある。

Marp

Markdownでスライドが書けるやつ。pptx出力すると画像埋め込みが崩れる既知問題があって、社内配布用には使えなかった。レイアウトの自由度も低い。

python-pptx / PPTAgent

pptxを直接生成する系。Windows環境でのセットアップが地雷、pptxの仕様が複雑でレイアウト崩壊しやすい。そもそも編集するならpptxにする意味がない。

Gamma / Beautiful.ai

情報が外に出る（社内機密NG）、ブランドカラー再現が弱い、月額課金でロックイン。却下。

採用した構成（梶谷式HTMLスライド）

1枚の長いHTMLページに、16:9のセクションを縦積みにする方式

出力はHTML（発表時）+ PDF（配布時）+ PNG（サムネ用）の3つ。スクロールスナップ＋キーボードナビ（矢印／Space／F でフルスクリーン）。Playwrightで1920×1080のPDF/PNGを書き出し。CSS変数でブランドカラーを差し替えるだけで案件ごとに使い回せる。

pptxを捨てる = レイアウト崩壊ゼロ。HTMLならCSSで完全にコントロールできる。これが核心。

つまずき1 情報量が少なすぎる

抽象的なスライドしか出てこない。「3.4時間 → 5分」みたいな数字が別々のスライドに散らばっていた。Claude Codeに「必要な情報は実際に別プロジェクトを見に行って埋めて」と指示したら、サブエージェントが実データを抽出してきて一気に説得力が出た。

つまずき2 Before/Afterが1枚にまとまらない

対比数字は1枚に並べないと刺さらない。DualCalloutという派生レイアウトを追加した。

つまずき3 文字が潰れて読めない

詳細テキストを18pxで書いたら1920×1080では読めなかった。最小20pxに統一。タグピルも13px → 18pxに修正。

つまずき4 Processレイアウトの下半分がスカスカ

番号＋タイトル＋説明だけだと情報密度が足りない。各stepに「タグピル」＋「ul li3点」を必ず追加するルールに変更。margin-top: autoで下部に押し込むCSSも入れた。

つまずき5 平坦な白背景はダサい

コーナーにradial-gradientのblob（opacity 0.08〜0.1）、カード天面に4pxのグラデーションバー、大数字はbackground-clip: textでグラデ文字。これで一気に「社内配布して恥ずかしくない」見た目になった。

つまずき6 差別化ポイントがfooterに埋もれる

一番見せたい差別化要素は専用スライド1枚にする。バッジ＋大見出し＋4カードグリッド＋ルールピルの構成で大きく見せる。

# slide-generator

社内プレゼン品質のスライドをClaude Codeで自動生成するプロジェクト。

## 方針

- 梶谷式準拠: PowerPoint/Keynoteではなく、**1枚のHTMLページに16:9セクションを縦積み**する方式
- 出力は HTML（発表時）+ PDF（配布時）+ PNG（サムネ用）
- pptx編集は**サポートしない**（Marpの画像埋め込み問題・レイアウト崩壊を回避するため）

## 絶対ルール

- IMPORTANT: セッション開始時に必ず `LESSONS.md` を読み、過去の教訓を確認する
- IMPORTANT: ユーザーから修正指摘を受けたら、その場で `LESSONS.md` に1行追記する
- IMPORTANT: スライド生成は `.claude/skills/create-slide/SKILL.md` のフローに従う
- IMPORTANT: 生成前に必ず `templates/design-tokens.css` のCSS変数でデザイン統一する
- IMPORTANT: レイアウトは `templates/layouts/` の5種から選ぶ。自由記述レイアウトは禁止（ダサくなる原因）
- IMPORTANT: ヒアリングで構成案を作ったら、**必ずユーザー確認を取ってから**HTML生成に進む
- IMPORTANT: 日本語タイトルは15文字以内、本文箇条書きは1行25文字以内
- IMPORTANT: 配色は3色＋グレースケールのみ。CSS変数を上書きして使う

## 出力

- `output/<案件名>/index.html` — 本体
- `output/<案件名>/slides.pdf` — 配布用（`scripts/export-pdf.js` で生成）

## エクスポート

```bash
node scripts/export-pdf.js output/<案件名>/index.html
```

## デザイン原則（強制）

- 1スライド = 1メッセージ
- タイトル + 本文3〜5点 + ビジュアル1点
- アイコンは Lucide (https://lucide.dev) のSVGを inline で埋め込む。他のアイコンセットとの混在禁止
- フォント: 見出し Inter、日本語 Noto Sans JP、コード JetBrains Mono
- font-weight: 見出し 700、本文 400。中間値禁止
- 背景グラデーションはコーナーのみ。テキスト背後に敷くの禁止

# create-slide スキル

Claude Codeで社内プレゼンスライドを作るための手順書。

## 起動トリガー

- 「スライド作って」「プレゼン資料作って」「slide-generatorでXXの資料を」等の依頼

## セッション開始時（必須）

- IMPORTANT: 作業を開始する前に必ず `LESSONS.md` を読み込み、過去の教訓を確認すること
- 教訓は「情報密度」「文字サイズ」「レイアウト選択」等の判断に影響する。無視して同じ指摘を受けるのはNG

## 教訓の追記ルール（必須）

ユーザーから修正指摘を受けたら、**その場で** `LESSONS.md` に1行追記する。

**追記トリガー:**
- 「〜が小さい」「〜スカスカ」「〜追加して」「〜少なすぎる」「もっと〜」等の修正指摘
- デザイン・情報量・レイアウト選択への否定的フィードバック
- 非自明な判断が明示承認されたケース（例: 2スライドのマージ承認）

**追記フォーマット:**
```
- [YYYY-MM-DD] <カテゴリ>: <ルール>。<理由・適用条件>
```

- 同じ指摘が2回以上来たら先頭に ★ を付ける（優先度高）
- 既存教訓と重複しそうなら新規追加せず既存行を更新する
- memoryやCLAUDE.mdではなく **必ずこの `LESSONS.md` に書く**

## フロー（必ず順守）

### ステップ1: ヒアリング

ユーザーに以下を対話で聞く。すべて固定質問ではなく、案件に応じて省略・追加する。

**最低限確認すること:**
1. **目的** — 聴衆に何を決めて / 理解して / 行動してほしいか
2. **オーディエンス** — 誰向け（経営層 / エンジニア / 社外顧客 / 新入社員 等）
3. **枚数・時間** — 何分の発表か、枚数の目安（目安: 5分=5枚、15分=10-15枚、30分=20-25枚）
4. **キーメッセージ** — 聴衆に1つだけ覚えてほしいこと（1文）
5. **素材** — 使いたいデータ・エピソード・引用（あればテキストで貰う。なければ後で聞く）

**可能なら確認すること:**
6. トーン（フォーマル / カジュアル）
7. ブランドカラー（あれば HEX で。なければデフォルト: インディゴ #4f46e5 + ミント #10b981）
8. 締切

不明点があれば **勝手に解釈せず質問する**。

### ステップ2: 構成案の提示と確認

ヒアリング結果を踏まえ、以下フォーマットで**Markdownの構成案を提示**する。

```
## 構成案: <プレゼンタイトル>

想定時間: X分 / 全Y枚 / ブランドカラー: #xxxxxx

1. [Hero] タイトル・サブタイトル・発表者
2. [IconGrid] アジェンダ（3-5点）
3. [Hero] セクション区切り: 課題
4. [DataCallout] 現状の数字（◯◯%が△△）
5. [Contrast] Before vs After
6. [Process] 解決ステップ（3-5段階）
7. [IconGrid] 期待効果
8. [Hero] まとめ・Call to Action
```

各スライドに **[レイアウト名] キーメッセージ** を1行で書く。

**IMPORTANT: ここでユーザー確認を必ず取る**。「この構成で進めていい？」と聞いてOK貰うまでHTML生成に進まない。

### ステップ3: HTML生成

承認後、以下の規則で `output/<案件名>/index.html` を生成する。

**ファイル構成:**
- `output/<案件名>/index.html` — 全スライドを1ファイルに集約
- `output/<案件名>/style.css` — design-tokens.cssをコピー＋案件ごとのブランドカラー上書き

**HTML雛形:**
`templates/base.html` をコピーして使う。各 `<section class="slide">` は以下のクラスから選ぶ:
- `slide-hero` — タイトル・セクション区切り・結論
- `slide-contrast` — 2分割比較（Before/After、課題/解決）
- `slide-icon-grid` — アイコン+短文 3-5点（アジェンダ、メリット、ステップ概要）
- `slide-data-callout` — 大きな数字 + 補足（KPI、統計）
- `slide-process` — 矢印つきステップ図（3-5段階）

**デザイン制約（厳守）:**
- タイトル（h1/h2）: 日本語15文字以内
- 箇条書き: 1行25文字以内、1スライド最大5項目
- アイコン: Lucide（https://lucide.dev/icons）のSVGをinlineで埋め込む。他混在禁止
- 色は `style.css` のCSS変数（--color-primary / --color-accent / --color-text / --color-bg）のみ使用
- 背景画像・グラデーションをテキスト背後に敷かない（可読性低下）

### ステップ4: PDFエクスポート

生成後、以下を実行して配布用PDFを作る。

```bash
node scripts/export-pdf.js output/<案件名>/index.html
```

`output/<案件名>/slides.pdf` が生成される。

### ステップ5: 確認・調整

- ブラウザで `output/<案件名>/index.html` を開いて見た目を確認するようユーザーに依頼
- 「スライド3の本文をもっとシンプルに」等の修正指示を受けたら該当 `<section>` のみ編集
- 大幅修正が出たら構成案から再確認

## やってはいけないこと

- 構成案の確認をスキップしてHTML生成に進む
- 独自レイアウト（templates/layouts以外）を作る
- ブランドカラー以外の色を勝手に追加する
- 1枚に複数メッセージを詰め込む
- Lucide以外のアイコンセット（Font Awesome等）を混ぜる
- Marpやpython-pptxに切り替える（このプロジェクトはHTML専用）

このSKILL.mdが、スライド生成時にClaude Codeが参照する手順書になる。「ヒアリング→構成案→承認→生成」という順番を強制することで、叩き台がいきなり崩れるのを防げる。

Step 8 scripts/export-pdf.js を置く

PlaywrightでHTMLをPDFとPNGに書き出すスクリプト。一度置けば使い回し。

#!/usr/bin/env node
/**
 * PlaywrightでHTMLスライドをPDF/PNG出力する
 *
 * 使い方:
 *   node scripts/export-pdf.js output/案件名/index.html
 *   node scripts/export-pdf.js output/案件名/index.html --png
 */

const { chromium } = require('playwright');
const path = require('path');
const fs = require('fs');

async function main() {
  const args = process.argv.slice(2);
  const inputArg = args.find(a => !a.startsWith('--'));
  const exportPng = args.includes('--png');

  if (!inputArg) {
    console.error('使い方: node scripts/export-pdf.js <index.html のパス> [--png]');
    process.exit(1);
  }

  const inputPath = path.resolve(inputArg);
  if (!fs.existsSync(inputPath)) {
    console.error(`ファイルが見つからない ${inputPath}`);
    process.exit(1);
  }

  const outDir = path.dirname(inputPath);
  const fileUrl = 'file:///' + inputPath.replace(/\\/g, '/');

  console.log(`入力 ${inputPath}`);
  console.log(`URL ${fileUrl}`);

  const browser = await chromium.launch();
  const context = await browser.newContext({
    viewport: { width: 1920, height: 1080 },
    deviceScaleFactor: 2,
  });
  const page = await context.newPage();

  await page.goto(fileUrl, { waitUntil: 'networkidle' });
  await page.evaluate(() => document.fonts.ready);
  await page.waitForTimeout(500);

  // PDF出力
  const pdfPath = path.join(outDir, 'slides.pdf');
  await page.pdf({
    path: pdfPath,
    width: '1920px',
    height: '1080px',
    printBackground: true,
    preferCSSPageSize: true,
    margin: { top: 0, right: 0, bottom: 0, left: 0 },
  });
  console.log(`PDF出力完了 ${pdfPath}`);

  // PNG出力（--png指定時）
  if (exportPng) {
    const pngDir = path.join(outDir, 'png');
    if (!fs.existsSync(pngDir)) fs.mkdirSync(pngDir, { recursive: true });

    const slideCount = await page.evaluate(() => document.querySelectorAll('.slide').length);
    console.log(`PNG出力 ${slideCount}枚`);

    for (let i = 0; i < slideCount; i++) {
      await page.evaluate((idx) => {
        const slides = document.querySelectorAll('.slide');
        slides[idx].scrollIntoView({ behavior: 'instant', block: 'start' });
      }, i);
      await page.waitForTimeout(300);

      const pngPath = path.join(pngDir, `slide-${String(i + 1).padStart(2, '0')}.png`);
      await page.screenshot({
        path: pngPath,
        clip: { x: 0, y: 0, width: 1920, height: 1080 },
      });
    }
    console.log(`PNG出力完了 ${pngDir}`);
  }

  await browser.close();
}

main().catch((err) => {
  console.error(err);
  process.exit(1);
});

ここで構成案を承認する前にHTML生成に進ませないのが鉄則。構成で気になる点があれば全部ここで潰す。生成後に構造から直すのは高コスト。

Step 5 HTML生成依頼

承認した構成案でHTMLを生成してください。ブランドカラーは #xxxxxx / #xxxxxx で。既存データは <参照元プロジェクトパス> から引っ張ってきて埋めてください。

データ参照元があれば必ず指定する。「いい感じに埋めて」と言うと抽象的なプレースホルダーで埋めてくる。具体的な参照元を指示すると一次情報が入って説得力が段違いになる。

Step 6 PDF/PNG書き出し

export-pdf.js を実行して、PDFとPNGを書き出してください。

または自分で実行するなら以下。

node scripts/export-pdf.js output/<案件名>/index.html --png

Step 7 修正依頼（ここで教訓が溜まる）

ブラウザでoutput/<案件名>/index.htmlを開いて確認。気になる点を指摘する。

スライド3の文字が小さい。もっと大きく。あとスライド5のProcessレイアウト、下半分がスカスカなので情報量を増やしてください。

指摘を受けたClaude Codeは、SKILL.mdのルールに従ってLESSONS.mdに1行追記してから修正に入る。次回から同じミスをしなくなる。

Step 8 差分修正（構成は変えない）

スライド7のBefore/Afterの数字を「3.4時間 → 5分」に差し替えてください。他は触らないで。

数値や文言の差し替えは該当セクションだけを編集させる。全体を作り直させると別の場所で劣化が起きる。

つまずきポイントと回避策

「Playwrightが起動しない」

npx playwright install chromiumを忘れている可能性が高い。Step 0のセットアップコマンドを全部再実行。

「PDFにするとフォントが崩れる」

Google Fontsの読み込みが完了する前に書き出している。export-pdf.jsのawait page.waitForTimeout(500);を1500に増やす。オフラインで使う場合はフォントをローカルに落として@font-faceで指定する。

「Processレイアウトがスカスカに戻ってしまう」

LESSONS.mdに対策が書いてあっても、Claude Codeがセッション開始時にそれを読んでいない可能性。Step 1の「LESSONS.md を読み込んでください」を毎回必ず最初に投げる。CLAUDE.mdのIMPORTANT: セッション開始時に必ず LESSONS.md を読むを強調しておく。

「ブランドカラーを変えたら一部の色が残る」

design-tokens.cssをコピーしてoutput/案件名/style.cssを作るときに、--color-primaryと--color-accentだけ上書きすればOK。本体を触らない。

「1枚に文字を詰め込みすぎて読めない」

CLAUDE.mdの「1行25文字以内・最大5項目」を守らせる。守ってこなかったら「CLAUDE.mdの制約を守ってください」と1行投げれば戻る。

運用していくと何が起きるか

最初の1案件でLESSONS.mdに10〜20行たまる。2案件目以降は「最初の叩き台」の質が上がっている。同じ指摘を2回受けない。

自分の場合、3案件目くらいから初回生成物がそのまま使えるレベルになった。修正ラウンドが0〜1回で済む。

LESSONS.mdを育てている感覚は自分用のAIをチューニングしている感覚に近い。話しかけるたびに賢くなっていく。

応用 他の定型業務にも転用できる

同じ構造は他の業務にも転用できる。

提案書生成プロジェクト
CLAUDE.md＋提案書テンプレート＋LESSONS.md
議事録整形プロジェクト
CLAUDE.md＋議事録フォーマット＋LESSONS.md
ブログ記事草稿プロジェクト
CLAUDE.md＋記事フレームワーク＋LESSONS.md

コアの発想は「プロジェクト憲法 + スキル手順書 + 教訓ログ」の3点セット。これを作っておけば、話しかけるだけで動く自分専用のAIツールが量産できる。

おわりに

Claude Codeの一番強い使い方は、コードを書かせることではなく、ツールを作らせることだと思っている。

1回ちゃんとプロジェクト化しておけば、あとは話しかけるだけ。LESSONS.mdが育っていくほど賢くなる。
