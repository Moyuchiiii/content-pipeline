Claude Codeって結局なに？

最初に1つだけ説明しておく。

Claude Code（クロード・コード）は、Anthropicっていう会社が作ったAIアシスタント。チャット感覚で「この文章整えて」「このデータをこのフォーマットに変換して」って頼むと、コードを書いたり作業を自動化してくれるツール。

似たツールでChatGPTがあるけど、Claude Codeはターミナル（PCの黒い画面）の中で動いて、ファイルを読んだり書いたりできるのが一番の違い。ChatGPTは「質問に答える相棒」、Claude Codeは「手を動かしてくれる相棒」みたいな感じ。

ターミナルって聞くと難しそうじゃないですか？自分も最初そう思ってた。でも実際にやることは、黒い画面にコピペしたコマンドを貼るだけ。一度慣れたら普通のアプリと変わらない。

Claude Codeは「AIに指示を出して手を動かしてもらう」ツール

副業を始めた理由はシンプルに「お金がなかった」

文系大学生の自分がなんで副業を始めたかというと、バイトだけじゃ足りなかったから。

時給1,100円のカフェバイト週3で月5万くらい。そこから授業料の自己負担分を払うと手元にほぼ残らない。友達と遊びに行くお金もなくて、しんどかった。

2026年1月末、X（旧Twitter）で「Claude Codeで副業始めて月10万いった」みたいな投稿を見かけた。発信してたのはエンジニアじゃなかった。「へぇ、AIで仕事できるんだ」と思って、そこから興味を持った。

調べてると「クラウドワークス（CW）」っていうのが出てきた。これは企業や個人が「こういう仕事できる人いない？」って案件を出してて、登録してる人が応募できるサイト。案件を見てみると、確かに「AIで自動化してくれる人募集」みたいなのがちらほらある。

「これ、自分にもできるのかな」と思って、とりあえずCWに登録した。

最初の2週間 何を書いても応募が通らない

登録してから最初の2週間、本当に何も起きなかった。

1日1〜2件くらい応募文を送ってたけど、全部無視された。既読もつかない。自分は「やります！頑張ります！」くらいの応募文しか書けなくて、今思うと当然の結果だった。

このときのプロフィール、今読み返すと恥ずかしい。

当時のプロフィール（抜粋）
文系大学生です。AIに興味があって勉強中です。よろしくお願いします。

これで仕事来るわけない。自分が発注側だったら絶対選ばない。

2週間経っても0件受注。「やっぱり文系には無理なのかも」と思って、1回全部やめようとした。でもその前に、試しにClaude Codeに全部相談してみた。

Claude Codeに相談したら全部ひっくり返った

「応募が全然通らない。プロフィールと応募文を見直してほしい」って、プロフィールと直近5件の応募文を全部渡した。

返ってきたのが、発注側が何を見てるかっていう観点の指摘だった。

  • 「できます」だけじゃ何ができるか伝わってない
  • 自分が相手の案件をちゃんと読んだ証拠（具体的に言及）がない
  • 過去の実績がないなら「この案件をどう進めるか」の提案で埋めるべき
  • プロフィールに「文系」って書くとマイナスに働くので、代わりに何ができるかを書く

言われてみれば全部当たり前なんだけど、自分じゃ気づけなかった。

そこからは案件ごとにClaude Codeに「この案件文を読んで、クライアントが本当に困ってることを整理してほしい。そのうえで、自分ならこう進めますって応募文を書いてほしい」って投げるようにした。

Claude Codeが出してきた文章そのままじゃAI感が出るから、最後に自分の言葉で書き直した。「〜と考えます」を「〜だと思います」に変えたり、固すぎる言い回しを崩したり。

このやり方で応募文を出すようにしたら、その週のうちに初めての返信が来た。

初受注は2月末 ¥15,000の業務自動化スクリプト

最初に受注できたのは、スプレッドシート（Excelみたいな表計算アプリ）のデータを別フォーマットに自動変換するスクリプト作成の案件だった。金額は¥15,000。

案件文を読んだとき、正直「これ自分にできるのか？」って思った。でもClaude Codeに案件文を渡したら、「こういう構成で組めば解けます」って設計図を出してくれた。自分がやることは「その設計図どおりに動くコードを書いて」って頼むだけ。

実際、コードは一行も自分で書いてない。Claude Codeが全部書いてくれた。自分は「ここをこう変えて」「これだとテスト通らないからこうして」って指示を出し続けただけ。

納品した翌日に「問題なく動いてます、ありがとうございます」って返事が来た。初めて副業でお金が入った瞬間だった。¥15,000だけど、自分にとってはめちゃくちゃ大きかった。

案件Bはクライアントが自社の売上データを毎朝手動でExcelに転記してたのを、自動で日報に変換するスクリプト。

案件C 在庫管理の通知bot（¥20,000）

3件目は個人商店のクライアントが、商品の在庫が少なくなったらLINEに通知がくるbotを作る案件。

ここで初めて「LINE Messaging API」っていう、LINEから自動で通知を送るための仕組みを使った。APIって聞くと難しそうじゃないですか？

APIっていうのは、他のサービスと連携するための接続口みたいなもの。LINEの場合、ちょっとした登録作業をすると「ここに特定の文字列を送ると自分のLINEに通知がくる」っていう専用のURLをもらえる。あとはClaude Codeにそのインフラ設定を任せれば、在庫データを監視して自動でLINEに通知する仕組みができる。

設定は最初の1回だけ20分くらいかかる。2回目以降はコピペで終わる。自分はこの案件で初めてAPIを使ったけど、詰まった時間は正味15分くらい。Claude Codeが「ここでこう設定してください」って指示してくれる通りにやればOKだった。

納品後、クライアントから「LINEに通知来て感動しました」って言われた。たぶん発注側にとっても「AIで業務自動化」って初めての体験だったみたい。

2月のまとめ 単価を上げる余地しかなかった

2月が終わった時点で、自分の中に1つの問題意識があった。

案件単価が¥15,000〜¥20,000のレンジから抜けられてなかった。このまま同じ系統の案件を積み上げていっても、月5〜7万が上限。もっと上を狙うには、別ジャンルに広げる必要があった。

ここで3月の転換点に入る。

3月の転換点① Auto Modeで作業時間が1/3になった

3月に入って最初に変えたのが、Claude Codeの「Auto Mode」を使い始めたこと。

Auto Modeっていうのは、Claude Codeに作業を任せたときに、一つ一つの操作で「これをやっていいですか？」って確認を取らずに連続で進めてくれるモード。

2月までは毎回確認ダイアログが出てて、自分が「はい」「はい」って押してた。これが意外と時間を食う。1案件で50〜100回くらい確認が出ると、それだけで30分〜1時間かかってた。

Auto Modeにすると、Claude Codeがファイルの読み書き・コード生成・テスト実行までを自分で完結させる。人間に聞くのは、本当に危ない操作（ファイル削除とか外部への送信とか）だけ。

Auto Mode導入前
要件確認 → 指示 → 確認ダイアログ連打 → 手修正 → 再テスト／1案件 平均8〜10時間
Auto Mode導入後
要件確認 → 指示 → 完成物チェック → 納品／1案件 平均2.5〜3時間

同じ時間で受けられる案件数が、月2〜3件から4〜5件に増えた。これだけで売上が1.5倍以上になる計算。

Auto Modeを有効化する手順（実作業 約3分）

Auto Modeの有効化は本当に簡単。Claude Codeを起動した状態で、キーボードのShiftを押しながらTabキーを押すだけ。画面に「Auto Mode: ON」って出る。もう一度Shift+Tabで戻せる。

最初「え、これだけ？」って思った。これだけ。

自分の運用ルールとしては、以下のタイミングでAuto Modeを切る。

  • 本番環境のデータに触る作業のとき
  • クライアントから渡された未検証のコードを実行するとき
  • 大きな金額を動かす処理を含むとき

それ以外はずっとAuto Modeで回してる。安全に倒したいなら、Claude Codeの設定ファイルに「この操作だけは必ず確認する」っていうリストを書いておく方法もある。これは後半で書く。

失敗記録② テスト不足でv6まで修正が続いた話

もっと深刻な失敗がこっち。Chrome拡張機能の開発案件で、テスト範囲が甘くてクライアントから6回バグ報告を受けた。

この案件は、Amazonの商品ページを開いたら自動で商品説明を整形して、別のECサイト（カラーミーショップ）に登録するっていう拡張機能を作る内容。

最初の納品（v1）はすぐ動かなくなった。原因はOpenAI APIのクレジット不足で、自分はテスト環境では動くのを確認してたけど、クライアントの環境で初めて発覚した。

v2を納品したらカラーミー認証後にAmazonで動かないバグ。v3は書籍ページで商品説明が取れない。v4はカテゴリが正しく設定されない。v5は特殊記号（エンダッシュっていう長いハイフン）でエラー。v6でようやく安定。

最終的にクライアントから「こちらが提示した商品だけでテストされてます？」って直接指摘された。

完全にその通りで、自分はクライアントが最初に提示したサンプル5件だけでしかテストしてなかった。Amazonには本・食品・衣類・家電・書籍・ゲームとか、商品カテゴリごとにページ構造が全然違う。それを想定できてなかった。

この失敗以降、テスト観点をClaude Codeに生成させるようにした

以降、納品前に必ず投げるプロンプトができた。

この機能の受け入れテスト観点を、以下の切り口で網羅的に出してください。

1. 正常系（意図通りに動くケース）
2. 異常系（エラー・例外ケース）
3. 境界値（数値・文字数の上限下限）
4. 環境依存（OS・ブラウザ・ネット状況）
5. データのバリエーション（フォーマット差・特殊文字・空データ）
6. ユーザー操作の順序違い

このプロンプトを実装完了時に投げると、テスト観点が50〜100項目出てくる。全部やるのは無理だけど、見た瞬間に「あ、この観点抜けてた」って気づけるのが大きい。

これ以降、同じ系統のバグ報告で追加修正を食らうことはなくなった。

文系の自分が詰まらなかった3つの理由

正直、自分もやる前は「コード読めないのに副業で仕事受けれるの？」って思ってた。2ヶ月やってみて、詰まらなかった理由を3つ書く。

理由1 コードを理解しようとしなかった

最初の頃、自分はClaude Codeが出してきたコードを「理解してから進めよう」としてた。全然進まなかった。

あるタイミングから、「理解しなくていい、動けばOK」に切り替えた。コードを読まず、動作確認だけする。エラーが出たらそのままClaude Codeに貼り返す。これだけで作業時間が半分になった。

文系がコードを1行ずつ理解しようとするのは無理ゲー。エンジニアが1ヶ月かけて覚えることを、Claude Codeに「この処理を日本語で説明して」って投げるだけで30秒で済む。自分で覚える必要がない。

理由2 ターミナルは「コピペする場所」と割り切った

ターミナルって聞くと、プログラマーがカタカタ打ってる難しそうな画面を想像する。自分もそうだった。

でも実際、副業で使う範囲のターミナル操作は「決まったコマンドをコピペする」だけ。Claude Codeが「このコマンドを打ってください」って指示してくれるから、それをコピペして Enter を押すだけ。

自分がよく使うコマンドは4つくらい。

  • フォルダ移動のコマンド
  • Claude Codeを起動するコマンド
  • プログラムを実行するコマンド
  • ファイル一覧を見るコマンド

全部Claude Codeが指示してくれるから、覚える必要もない。

理由3 「わからない」を恥ずかしがらなかった

クライアントから専門用語が飛んできたとき、最初は「わからない」って言えなかった。プロのふりをしたくて、わかったフリをして進めて、あとで事故ってた。

途中から、わからない単語は即Claude Codeに「これ何？」って聞いて、5秒で理解してから返信するようにした。クライアント側は「ちゃんと確認してくれる人」と捉えてくれる。

1個例を出すと、クライアントから「このWebサイト、レスポンシブ対応できます？」って聞かれたことがある。レスポンシブって言葉、自分は知らなかった。Claude Codeに聞いたら「スマホでもPCでも見やすい画面にする技術」って返ってきて、「できます」って答えた。実際Claude Codeが実装してくれるから問題なかった。

今も詰まってるところ（正直に書く）

2ヶ月やってきて、全部うまくいってるわけじゃない。今でも詰まってる部分を正直に書く。

1. 複雑な要件のすり合わせ

単純な業務自動化は問題ないけど、複数のシステムが絡む案件だと、要件の整理段階で時間を食う。クライアント側も言語化できてないことが多い。ここはClaude Codeも万能じゃない。

2. 「自分のスキルが上がってるか」が見えにくい

Auto Modeで全部任せてると、半年後に自分が独立したときに何ができるようになってるのか、正直不安。どこを自分でやって、どこをClaudeに任せるか、今も模索中。

3. 単価交渉

¥65,000の案件で時給¥3,800は悪くないけど、Webアプリ案件の相場としては安い方。もっと単価を上げたいけど、交渉慣れしてなくて、毎回クライアントの提示額をそのまま受けてる。これは今月の課題。
