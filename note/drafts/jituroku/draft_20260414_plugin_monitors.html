CI完了の通知、毎回自分で「見に行って」いませんか？

先週、クラウドワークスの納品が1件遅れた。CIのテストが落ちてたのに、自分が気づいてなかっただけ。Claude Codeで作業してるうちにウィンドウを閉じて、Monitorツールを起動し忘れてた。

この「Monitorツールを毎回手動で起動する手間」が、4月13日リリースのClaude Code v2.1.105で消えた。自作プラグインに監視設定を埋め込んでおくと、Claude Code起動した瞬間に監視が走り出す。

1週間使ってみたら、取りこぼしが週2回→0になった。その全記録。

何が変わったか

Claude CodeのMonitorツールは、バックグラウンドの変化（CIの進捗・サーバーログ・PRコメント）を監視してClaudeに通知する機能。v2.1.98でリリースされた。ただし毎回セッションごとに手動で起動しないといけなかった。

v2.1.105で、プラグインの設定ファイル（plugin.json）に monitors という項目が追加された。ここに監視の設定を書いておくと、以下のタイミングで自動起動する。

セッション開始時
Claude Codeを起動した瞬間に監視が走る。常時必要な監視（CI・PR到着など）向け
スキル呼び出し時
プラグインのスキル（例/my-plugin:deploy）が起動したタイミングで監視が走る。作業中だけ必要な監視（devサーバー・ビルドログなど）向け

この2段階の使い分けが、副業ワークフローにめちゃくちゃ合う。

Monitorツール手動起動で何を取りこぼしてたか

先月、Monitorツールの記事を書いたとき、「週5時間のCI待ちが0.5時間になった」と書いた。実際そのとおりだった。ただ、正確には「Monitorを起動してる間は」という条件付きだった。

自分の副業の失敗談を正直に書くと、以下のパターンを週2回くらいやってた。

パターン1
CW案件の納品前、CIをpushして別作業に入った。Claudeのセッションを閉じた。翌朝見たらCIが落ちてた。気づいたのは納品予定の2時間前
パターン2
Webアプリ案件でdevサーバー起動したまま作業。Claudeのターミナル経由で見えないエラーがログに出てた。バグ報告がクライアントから来て初めて気づいた

どっちも「Monitorを起動するのを忘れてただけ」。でも人間は忘れる。

自作プラグインにmonitorsを書いた

自分が使ってる副業プラグイン（以前の記事で自作したやつ）に、今回 monitors 設定を追加した。

プラグインのルートに monitors/monitors.json を置いて、以下のように書く。何をしてるかというと、3種類の監視をプラグインに同梱してる。

{
  "monitors": [
    {
      "id": "ci-watch",
      "trigger": "session_start",
      "command": "gh run list --limit 1 --json status,conclusion,url",
      "notify_when": "conclusion changed",
      "poll_interval": "30s"
    },
    {
      "id": "dev-server",
      "trigger": "skill_invoke",
      "command": "tail -f ${CLAUDE_PLUGIN_DATA}/dev.log",
      "notify_when": "contains 'ERROR'"
    },
    {
      "id": "pr-review",
      "trigger": "session_start",
      "command": "gh pr view --json reviews --jq '.reviews[-1]'",
      "notify_when": "new review added",
      "poll_interval": "2m"
    }
  ]
}

そして plugin.json に1行だけ追加する。

{
  "name": "cw-fukugyou",
  "version": "1.3.0",
  "monitors": "./monitors/monitors.json"
}

これでGitHubリポジトリにpushして、副業で使ってるPC全部に/plugin updateかけるだけ。どの環境で起動しても、セッション開始の瞬間からCI監視とPRレビュー監視が走る。

3つの監視、実際にどう効いたか

CI監視（セッション開始時）

Claude Codeを起動した瞬間に gh run list が走り始める。最新のGitHub Actions実行結果が変わったら即通知が来る。

先週、朝コーヒー淹れてる間にClaude起動しといて、シャワー浴びて戻ったら「CIが失敗しました」と画面に出てた。10分前にpushしたやつ。気づく速度が変わった。

devサーバー監視（スキル呼び出し時）

WebアプリUI案件では /cw-fukugyou:dev-start というスキルを自作してる。これを起動するとViteのdevサーバーが立ち上がる。このスキル起動と同時に、devサーバーのログにERRORが出たらClaudeに通知する監視も走り出す。

クライアントから「この画面でエラー出ます」と言われる前に、自分が気づいて直せるようになった。

PRレビュー監視（セッション開始時）

PR投げっぱなしで寝落ちしても、朝Claude起動した瞬間に「新しいレビューが届いてる」と教えてくれる。2分に1回ポーリングしてるけど、普段の作業中は気にならない。

取りこぼし 週2回 → 0回

つまずいた2パターン

session_startで重い監視を走らせすぎて起動が遅い

最初、監視を6個くらい全部 session_start にしてた。Claude Code起動するのに毎回10秒以上かかるようになった。

常時必要な監視（CI・PR）はsession_start、作業中だけ必要な監視（devサーバー・ビルド・テストウォッチ）はskill_invokeに分けるのが正解。副業で触るリポジトリが増えても起動速度は落ちない。

${CLAUDE_PLUGIN_DATA}のパスがずれた

devサーバーのログを ${CLAUDE_PLUGIN_DATA}/dev.log に吐かせてたけど、プラグインをアップデートしたら古いログが消える問題にハマりかけた。

公式ドキュメントを読み返したら、${CLAUDE_PLUGIN_DATA}はアップデートしても残る永続ディレクトリで、${CLAUDE_PLUGIN_ROOT}はアップデートで消えるパスと書いてあった。ログを残したい場合はDATA側に書く。ここ、混同しやすい。

Before / After

導入前
Monitorを起動し忘れる週2回。CI失敗気づくの最短30分、最悪納品の2時間前
導入後
Monitor起動忘れ0回。CI失敗気づくの最長1分以内。プラグインを入れたPC全部で同じ挙動

週1回あった「あ、CIコケてた」の焦りが消えた。これだけで睡眠が良くなる。

誰に向いてるか

正直、まだプラグイン自体を自作してない人には遠い話。プラグインを作ってない状態で monitors だけ使う方法はない。

ただ、副業で2〜3案件を並行でさばいてて、1つでも「CI落ちた」「エラー気づかなかった」で納期リスクを感じたことがある人には、たぶん1週間で元が取れる。

自作プラグインの作り方はこっちに書いた。chmod忘れ・nameの付け方ミス・/reload-plugins忘れの3つのつまずき込みで、最初の1本が1時間以内に作れる手順になってる。

プラグイン自体の作り方（bin/ディレクトリ活用・GitHub管理・別PC移植10秒）はこっちにまとめた。今回のmonitorsはその拡張編として読むとスムーズ。
