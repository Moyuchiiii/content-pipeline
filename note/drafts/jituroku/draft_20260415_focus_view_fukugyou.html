副業を3案件並行したら、画面の情報量で頭がパンクしていませんか？

先週、Claude Codeの画面を眺めながら案件Aのバグを追ってた途中で、別タブの案件Bから修正依頼が来て、そのままCに戻ろうとしたら「今どこ見てたんだっけ」ってなって10分止まった。

これが1日に3回くらい起きてた。ツール呼び出しのログ、中間の思考、ファイル差分、ストリームで流れてくるログ、全部見えてる状態。自分がどこを見るべきかわからなくて、頭の中で案件が混ざる。

結論から言うと、Claude Code v2.1.97（2026-04-07頃リリース）で入ったFocus View（Ctrl+O）を1週間使ったら、この問題がほぼ消えた。

3案件並行時の切り替えロス 1日30分 → 5分

（Before/Afterの実測）

何に困ってたか

副業で動いてる案件は大体2〜4本。それぞれにClaude Codeのセッションがある。画面を切り替えながら「この案件どこまで進んだっけ」を追う時間が、地味に積み上がっていた。

ざっくり分解するとこう。

画面を切り替えるたびに起きていたこと
ログを上にスクロールして過去のやり取りを探す／中間の思考ブロックを読み飛ばす／最後のコードの差分だけ確認する

毎回3〜5分。1日6〜7回切り替えてたので、これだけで1日30分消えてた。時給換算したらちょうど1案件分の利益が溶けてる計算。

Focus Viewって何ができるのか

Claude Codeのターミナル画面でCtrl+Oを押すと、表示が3段階で切り替わる仕組み。通常表示、Transcriptモード、そしてFocus View。

Focus Viewに切り替えると、画面に残るのは3つだけ。

自分の最後のプロンプト
何を頼んだかを1行で確認できる
ツール実行の1行サマリー
どのファイルを何行編集したかだけ
Claude Codeの最終レスポンス
「何をやったか」の結論

途中の思考ログとか、ツールが何を出したかの生ログは全部畳まれる。画面がスッと静かになる感覚。最初に押した瞬間「え、これだけ？」ってなった。

使い始めて何が変わったか

案件を切り替えるときにまずCtrl+Oを押すのが習慣になった。前のセッションから戻ってきたとき、画面には「自分が最後に頼んだこと」「Claudeが何を変えたか」「どう終わったか」の3つしか映らない。

これだけで、脳が「続きから」モードに入れる。過去ログを上にスクロールして探す工程が消えた。

1週間計測した感じだと、切り替えロスが1日30分→5分。週で約3時間戻ってきた。この3時間は、別案件1本受注に使える時間量。

この先で手に入るもの

  1. 3案件並行時の具体的ワークフロー（朝の立ち上げから夜の片付けまで）
  2. 通常表示 / Transcript / Focus Viewの使い分けルール3本
  3. 自分が1週間でハマった3つのつまずきポイント
  4. Focus Viewが向かない作業タイプ（これを間違えると逆に遅くなる）

3案件並行時のワークフロー

この1週間でやってた実際の動き。コピペで真似できるように書く。

朝の立ち上げ（9時〜9時15分）

案件ごとのClaude Codeセッションを開く。昨日の続きから--resumeで再開して、最初にCtrl+OでFocus View。これで画面には昨日の最後のやり取りだけが残る。

15分以内で3案件の「どこまで進んだか」を把握できる。以前はこれに40分かかってた。

作業中（通常表示に戻す）

実際にClaudeに指示を出してる時間は通常表示。Ctrl+Oをもう1回押すと通常表示に戻るので、思考ログも見ながら作業できる。

「今書かせてる内容を監督してる」時間は通常表示、「確認だけして次に行く」時間はFocus View、という切り分け。

案件切り替え時（Ctrl+Oで即シフト）

別案件のタブに移ったら、条件反射でCtrl+O。前のセッションの状態を1秒で把握して、次の指示に進む。案件を跨ぐと頭がリセットされる感覚があって、これが一番効いた。

夜の片付け（23時前後）

Focus Viewのまま、3案件の最終レスポンスだけ見て「明日やること」をメモアプリに1行ずつ書く。通常表示だとログが多すぎて、どれを書き残すべきか迷ってた。

3つのビューの使い分けルール

通常表示を使う場面

Claudeが何を考えながら作業してるかを見ておきたい時。新しいプロンプトを試してる時、バグ調査で思考過程が必要な時。ここはログ全部見えてた方がいい。

Transcriptモードを使う場面

過去のやり取りを細かく追いたい時。納品前に「この仕様を前に何て言ったっけ」を探す時は、Ctrl+O 2回押してTranscriptに切り替える。

Focus Viewを使う場面

案件切り替え時、レビュー時、朝と夜の整理時。「結論だけ欲しい」全部これ。1週間使って気づいたけど、副業ワークフローの8割はFocus Viewで足りる。

使い分けの一言ルール
書かせてる時は通常表示・見るだけの時はFocus View・探す時はTranscript

つまずいた3パターン

1. NO_FLICKERモードじゃないと動かない

最初Ctrl+O押しても反応なくて「壊れてる？」ってなった。Focus ViewはNO_FLICKERレンダリングモードの中だけで動く。設定を確認したら、環境変数NO_FLICKERが無効になってた。ここで1時間溶かした。

Claudeを開いてる状態でclaude --versionを叩いて2.1.97以上か先に確認する。それ以下ならclaude updateで即アップデート。

2. レビュー作業以外で常用すると遅くなる

最初テンション上がってずっとFocus Viewで作業してたら、逆に遅くなった。Claudeが何を考えてるか見えないと、暴走しかけた時に止められない。2日目で「書かせてる時間は通常表示」ルールに戻した。

3. Transcriptモードと混乱する

Ctrl+Oは3ビュー循環なので、押しすぎると「今どのモードだっけ」ってなる。最初の3日間は何度も混乱した。対策として画面上部にステータスライン表示を入れた（refreshIntervalオプションも同じv2.1.97で追加されてて、これと組み合わせると現モードが常に見える）。

Focus Viewが向かない作業

1週間の実測で「これはFocus Viewだと逆に遅い」とわかったものを正直に書く。

新規案件のヒアリング初日
Claudeに要件整理させる初回は通常表示。思考ログが判断材料になる
バグ調査の初動
原因追跡は中間ログに答えが出てるパターンが多い
新しいMCPツール検証
ツールが期待通り動いてるかは生の出力を見ないとわからない

逆に言うと、この3つ以外はFocus Viewでよさそう。運用に乗せてから1週間、画面を静かにするだけで頭が動きやすくなるのを体感した。

この経験で変わった考え方

Claude Codeのアップデートって新しい機能ばかり注目されがちだけど、Focus Viewは「情報を減らす」タイプの機能。副業みたいに並行して複数プロジェクト動かす時、情報を増やす機能より情報を減らす機能のほうが効く。

案件が増えるたびに集中力が削られる感覚があった人には、1回試してみてほしい。Ctrl+O押すだけで、1週間後には戻れなくなる。
