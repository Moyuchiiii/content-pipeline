クライアント向けの提案資料、毎回ゼロから作ってませんか？

副業でクライアントワークをやるようになって、一番時間を食ってたのが提案資料作りだった。

PowerPointを開いて、テンプレを探して、ロゴを貼って、色を合わせて、文字サイズを揃えて。気づいたら2日経ってる。本題（提案内容を詰める）に入る前に力尽きる、みたいなことが何回もあった。

先週、思い切ってClaude Codeに「自分専用の資料生成ツール」を作ってもらった。結果、40枚の提案デッキが2日 → 3時間になった。

この記事は、その体験の全記録。技術の話というより「Claude Codeに道具を作らせる感覚」の話です。

そもそも何がしんどかったか

副業で提案資料を出すとき、毎回こういうことが起きてた。

毎回ハマってたポイント
テンプレ選びで30分。色の調整で1時間。箇条書きの位置がズレて修正で1時間。画像書き出しでレイアウト崩壊して最初からやり直し

本気で提案の中身を考える時間より、PowerPointをいじってる時間のほうが長い。これは副業としておかしい、と思った。

「Claude Codeに資料を作ってもらう」じゃなくて「作るツールを作ってもらう」

最初は普通に「提案資料の草稿を書いて」とClaude Codeに頼んでた。でも毎回ゼロからやり直しになる。今日の指示を明日には忘れてるし、デザインの好みも覚えてくれない。

そこで発想を変えた。

やったこと
資料そのものを毎回作ってもらうんじゃなく、「資料を作るための自分専用ツール」をClaude Codeに作らせて、それを毎回呼び出す形にした

要は「AIに作業させる」から「AIに道具を作らせて、その道具で作業する」に切り替えた。これがデカかった。

最初に投げた1文

プロジェクト用のフォルダを1つ作って、そこでClaude Codeを起動して、投げた最初の指示はこれだけ。

最初に送った指示
ブラウザで見れる形式（HTML）の提案資料を、話しかけるだけで作れるプロジェクトを作ってほしい。社内プレゼンに耐えるレベルの品質で。PowerPointは使わない

ターミナル（黒い画面に文字を打つやつ。エンジニアが使うコマンド窓みたいなもの）に1行打って送るだけ。あとは向こうがファイル構成を組んでくれる。

面白いのは、Claude Codeが「レイアウトは5種類に絞りましょう」と自分から提案してきたこと。理由を聞いたら「自由に作らせると毎回デザインが崩れるから」と。確かに過去の失敗そのものだった。

この時点で「あ、これは普通にAIに作業させるのとは違うな」と感じた。相手が「どうすれば毎回崩れないか」まで設計してくる。

この先で手に入るもの

  1. Claude Codeに渡した「プロジェクト指示書」の中身（そのまま転用可）
  2. レイアウトを5種類に縛るための具体ルール
  3. 毎回のヒアリングで聞く5項目のテンプレ
  4. 6回やった修正指示の全文と、Claude Codeがどう反応したか
  5. 一番効いた仕組み「教訓ノート」の運用ルールと実物
  6. これを資料以外（請求書・レポート・SNS運用）に横展開する考え方

全部、自分が実際に動かしてるプロジェクトから持ってきた実物です。コピペして自分用に書き換えれば、今日から使える。

ここから有料エリア

1. プロジェクト指示書（CLAUDE.md）に書いたこと

Claude CodeにはプロジェクトのルートにCLAUDE.mdというファイルを置いておくと、毎回セッション開始時に自動で読み込んでくれる仕組みがある（簡単に言うと「このプロジェクトで守ってほしいこと」を書いた指示書）。

自分が書いた中身はこんな感じ。そのまま載せます。

方針
1枚の長いHTMLページに16対9のセクションを縦積みする方式。出力はHTML（発表時）とPDF（配布時）とPNG（サムネ用）。PowerPointの編集はサポートしない
絶対ルール
セッション開始時に必ず教訓ノートを読む / 修正指摘を受けたらその場で教訓ノートに1行追記する / 生成フローはスキル定義に従う / 生成前に必ずCSS変数でデザイン統一する / レイアウトは5種から選ぶ（自由記述は禁止） / 構成案を作ったら必ずユーザー確認を取ってから本体生成に進む / 日本語タイトルは15文字以内、箇条書きは1行25文字以内 / 配色は3色＋グレースケールのみ

重要なのは「禁止」と「必ず」を大量に書いていること。AIは自由度を与えると崩れる。縛れば縛るほど安定する。

特に効いた3つを挙げるとこれ。

「構成案を作ったら必ずユーザー確認を取る」

これがないと、Claude Codeは勝手に全部作ってくる。2000行のHTMLを生成した後に「構成が違います」って言われても直すのが地獄。先に構成だけMarkdownで出させて、OKが出てから本体生成に進ませる。これで手戻りがほぼゼロになった。

「日本語タイトルは15文字以内、箇条書きは25文字以内」

人間が何も指定しないと、AIは長い文章を詰め込んでくる。文字数の上限を明示的に書いたら、勝手に削って要約してくれるようになった。

「配色は3色＋グレースケールのみ」

色を無制限にすると毎回カラフルで素人っぽい資料が出てくる。3色に縛ると、急にプロっぽくなる。

2. レイアウトを5種類に縛るということ

Claude Codeが提案してきた5レイアウトはこれ。

Hero（表紙・区切り）
タイトルスライド、セクションの節目、結論の一言に使う。文字を少なく、余白を大きく
Contrast（2分割比較）
Before/After、課題/解決、現状/理想を左右で対比。副業の提案資料だと一番使う
IconGrid（3〜5点列挙）
アジェンダ、機能紹介、メリット列挙。アイコン＋短文で視覚的に並べる
DataCallout（大きい数字）
「87%」「月14万」みたいな1枚で見せる数字。インパクト重視
Process（ステップ図）
手順、工程、ロードマップ。3〜5段階まで

この5種類以外は作らせないというルールをCLAUDE.mdに書いた。

「ステップと箇条書きを組み合わせた新しいレイアウトで」とか言われても拒否する。理由は、自由にレイアウトを組ませると必ずどこかがダサくなるから。資料全体の統一感も壊れる。

5種類の中から毎回選ぶようにすると、「どのレイアウトを使うか」の議論だけに集中できるようになる。これが想像以上に楽だった。

3. 毎回のヒアリングで聞く5項目

新しい資料を作るとき、Claude Codeが自分に質問してくる内容も固定化した。スキル定義ファイルに「最低限これは聞く」と書いた5つ。

  1. 目的：聴衆に何を決めてほしいか、理解してほしいか、行動してほしいか
  2. オーディエンス：経営層・エンジニア・社外顧客・新入社員のどれか
  3. 枚数・時間：何分の発表か（目安：5分＝5枚、15分＝10〜15枚）
  4. キーメッセージ：聴衆に1つだけ覚えてほしいことを1文で
  5. 素材：使いたい数字・エピソード・引用をテキストで貰う

最初はこれを毎回自分が口頭で伝えてた。毎回抜けが出るから、Claude Code側に「ユーザーに聞け」と書いておいた。向こうが勝手に質問してくれるので、答えるだけで済む。

オプションで「トーン（フォーマル or カジュアル）」「ブランドカラーのHEX値」「締切」も聞いてもらう。

4. 構成案の出力フォーマット

ヒアリングが終わったら、Claude Codeは構成案をこの形式で出してくる。

## 構成案: <プレゼンタイトル>

想定時間: 5分 / 全6枚 / ブランドカラー: #2cb696

1. [Hero] 表紙
2. [DualCallout] Before/Afterの数字対比
3. [IconGrid] 時間の内訳
4. [Contrast] 運用の違い
5. [Process] 導入ステップ
6. [Hero] Call to Action

各スライドに[レイアウト名] キーメッセージを1行で書く。これだけ。

ここで「この構成でいい？」とClaude Codeが確認してくる。OKを出すまで本体生成には進まない。「これだけ」な構成案が、一番重要な品質ゲートになる。

5. 実際にテストでやった6回の修正指示

プロジェクトが動くか確かめるため、自分の副業（このnote運用の仕組み）をクライアントに売り込む提案資料を、Claude Codeに作らせてみた。最初の成果物は正直ダメだった。

ここから修正指示ループが始まる。全部、自分が実際に送った日本語。

修正1「情報量が少なすぎる」

送った言葉
もっと実際の動作フローとかもろもろ情報量を増やしたい。必要な情報は実際にcontent-pipelineを見に行って埋めて

Claude Codeがサブエージェント（別の作業用AI）を起動して、自分の別プロジェクトを読みに行ってくれた。そこから実データを抽出して埋めてきた。

向こうが持ってきた実データ
100点ルーブリック（30＋25＋20＋15＋10の配点）・7スウィープ編集プロセス・AIDAとStar Story Solutionフレームワーク・voice 6要素（語尾・一人称・改行・感情・修辞・噛み砕き）

これを各スライドに流し込んだら説得力が一気に出た。初回の叩き台は必ず情報が薄いと想定しておくのが正解。

修正2「2枚を1枚にまとめて」

送った言葉
2枚目と10枚目は合わせて1枚にした方が見やすい

→ DualCallout（Before/Afterの数字を左右に大きく対比する型）という新レイアウトを1つだけ追加してもらった。5種類の原則に例外を1つ増やす判断。

BEFORE 手作業        AFTER パイプライン
  3.4h          →         5分

この1枚が資料全体で一番インパクトあるスライドになった。

修正3「デザインが素朴すぎる」

ここで初めて具体的なデザインリクエストを投げた。「もっとゴージャスに」とだけ。

Claude Codeが提案してきた変更点。

装飾の追加
コーナーに薄いグラデーション（不透明度0.08〜0.1のにじみ）・カード天面に4pxのグラデーションバー・大数字はグラデーション文字（CSSのbackground-clip: textで実現）・重要ピルはprimary色→accent色のグラデ＋白文字＋大きめの影

具体的なCSSは全部向こうが書いてくれた。自分は「ゴージャスに」と言っただけ。具体的な実装方法をAIに任せても、こっちの意図に沿うレベルの選択肢を持ってきてくれる。

修正4「下半分がスカスカ」

送った言葉
フローのページ、上側はテキストあるけど下側スカスカで情報量が薄い

→ Processレイアウトの各ステップに「タグピル（薄い背景の丸いラベル）」と「具体アクション3点」を追加してもらった。空白が埋まって圧が出た。

修正5「文字が小さい」

実際に1920×1080の画面で見たら、サブ情報用の18pxフォントが潰れて読めなかった。

向こうの修正
18pxを使ってた箇所を全部20pxに。タグピルも13pxから18pxに引き上げ

これ以降、「1920×1080基準で最小20px」というルールを教訓ノートに追記した。

修正6「差別化ポイントを専用スライドにして」

送った言葉
事実確認スウィープ（全クレームをWeb検索で実照合する仕組み）はうちの差別化ポイントだから、これを紹介するためのスライドを追加して

→ 元はフッターに埋めてた要素を、専用スライド1枚に昇格。構成はバッジ＋大見出し＋リード文＋4カードグリッド＋ルールピル。資料全体で一番記憶に残るスライドになった。

6. 一番デカかった学び「指摘を資産にする仕組み」

6回も修正指示を出してて、途中で気づいた。

「このままだと次の案件でも同じ指摘を6回やらされる」

Claude Codeは毎回セッションがリセットされる（会話の記憶は引き継がれない）。今日「文字が小さい」「下半分埋めて」と伝えたことを、次のプロジェクトでは忘れてる。

そこでLESSONS.md（教訓ノート）というファイルをプロジェクトに1枚追加した。実物はこれ。

## 教訓一覧

- [2026-04-14] 情報密度: 初回叩き台は必ず
  「情報量が少なすぎる」と指摘される。
  プロジェクト本体から実データを引っ張って
  埋める前提で作る
- [2026-04-14] 文字サイズ: 1920x1080基準で最小20px。
  --fs-small (18px) をサブ情報に流用すると潰れる
- [2026-04-14] Process下半分対策: 各stepに
  タグピル+ul 3点を必ず追加し、下部に押し込む
- [2026-04-14] 差別化ポイント: footerや箇条書きに
  埋めず、専用スライド1枚で大きく見せる
- [2026-04-14] スライドマージ判断: 関連する
  対比数字はDualCalloutで1枚に統合
- [2026-04-14] ゴージャス化要求時: コーナーに
  薄いグラデ、カード天面にグラデバー、大数字は
  グラデ文字、重要ピルはグラデ+shadow-lg

フォーマットもシンプル。日付・カテゴリ・1行ルール・理由の4項目だけ。

そして、CLAUDE.mdに以下の2行を追加した。

CLAUDE.mdに追記したルール
セッション開始時に必ずLESSONS.mdを読む / 修正指摘を受けたらその場でLESSONS.mdに1行追記する

これで指摘が資産化される。次回の作業開始時、Claude Codeは過去の教訓を読み込んでから叩き台を作る。同じ指摘を2回受けなくて済む。

地味だけど、これが一番効いた。使うたびにツールが賢くなっていく感覚。

7. これを資料以外に横展開する考え方

この仕組み、資料作成以外にも全部使える。自分が実際に今回のやり方を横展開してる作業を挙げる。

請求書発行
クライアントごとの項目・税区分・振込先のパターンをプロジェクト化。毎月「今月の請求書作って」で終わる
クライアント向け月次レポート
数字の抽出ソース・フォーマット・ハイライトの出し方をルール化。毎月コピペ作業がゼロになる
SNS運用（このnoteとX）
投稿フォーマット・文体ルール・禁止フレーズをプロジェクト化。「今日のネタは◯◯」と伝えるだけで下書きが上がる
競合リサーチ
調査観点・出力フォーマット・情報源の信頼度タグ付けルールを固定。毎回「競合5社調べて」で揃った形式の資料が出る

副業って、同じパターンの作業を何度もやることが多い。そのパターンを1回ちゃんとプロジェクト化しておけば、あとは話しかけるだけで済む。

エンジニアじゃない人ほど効く。コード書けなくても、「こういう作業を自動化したい」と日本語で伝えれば、Claude Codeがプロジェクトごと組んでくれるから。実際、自分は今回のスライド生成プロジェクトで1行もコードを自分で書いてない。全部Claude Codeが書いた。

やる順番のおすすめ

この記事を読んで「自分もやってみたい」と思った人向けに、取りかかる順番だけ書いておく。

  1. 繰り返してる作業を1つ選ぶ（週1回以上やってるやつがベスト）
  2. 専用フォルダを作る（名前は作業名でOK）
  3. Claude Codeに「この作業を自動化するプロジェクトを作って」と日本語で頼む
  4. 向こうが作った構成にOKを出す前に、必ず「守るルール」を3つ以上書かせる
  5. LESSONS.mdを最初から作らせる（後から作るより運用が定着する）
  6. 1回テスト実行して、修正指示を3〜6回出す（このフィードバックが資産）

大事なのは「最初から完璧を狙わない」こと。叩き台はダサいのが普通。修正指示を繰り返して、その指摘を教訓ノートに貯めていく。これだけ。
