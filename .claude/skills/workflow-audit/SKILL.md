# workflow-audit スキル

note-run / x-run / brain-run / source-run スキルの定期監査を実行する。

> **2026-04-27 大幅改訂**: 過去の監査で「記憶ベース提案」「出典不正確なクレームの放置」が発生したため、**Phase 0（過去未対応フォロー）**・**Phase 1.5（クレーム検証）**・**Phase 3.5（自己批判）**・**「却下した提案」セクション** を必須化。記憶ベースで提案を出すことを禁止する。

---

## Phase 0: 過去未対応提案のフォローアップ強制（必須・最初に実行）

`context/content-memory.md` の「workflow-audit 実施履歴」セクションを読み、**直近3回分の監査の「未対応」指摘**を抽出する。

各未対応項目について **3択強制で決定**:

```
- ✅ 実装する（今日中に着手・着手しない場合は「実装する」を選んではいけない）
- ❌ 廃案にする（理由を1行で記録 → 後述の「却下した提案」セクションへ）
- ⏸ 引き続き保留（次回再評価日を明記・無期限保留禁止）
```

**「未対応のまま放置」は禁止**。3択のいずれかで決着させる。決着しないとPhase 1に進めない。

## Phase 1.5: クレーム検証（出典付き数字の真贋チェック・必須）

> **2026-04-27 追加**: SKILL.md に「○○ベンチマーク」「○○調査」と書いてある具体数字（%・倍率等）を**全件 WebSearch / WebFetch で出典確認**する。Anthropic 公式ハルシネーション対策の「引用検証」テクニックを workflow-audit に組み込む形。

**手順:**

1. Phase 1 で読み込んだ全ファイルから、以下のパターンを `Grep` で抽出:
   - 「\d+%」（パーセント数値）
   - 「HubSpot」「Anthropic公式」「○○調査」「ベンチマーク」等の出典名
   - 「サンプル○件」「調査○名」等の規模表現
2. 各クレームについて WebSearch / WebFetch で原典を確認
3. 判定:
   - ✅ **検証済み**: 原典に該当数字が存在
   - ⚠️ **要修正**: 原典に存在するが文脈違い（noteには適用不可・小規模では再現不可 等）
   - ❌ **削除**: 原典に存在しない or 確認できない

**修正ルール:**

- ❌ 該当のクレームは即削除 or 「未検証だが ~~○○ロジック~~ で正当化」のような留保表現に書き換え
- ⚠️ 該当のクレームは文脈を補足（「大規模ブログ前提」「サイドバー比較なのでnoteには直接適用不可」等）
- 検証結果を出力レポートに必ず記載

**過去事例（2026-04-27 検証で判明）:**

`note-run` SKILL.md「mid-content CTA は末尾より121%高い（HubSpot）」→ **HubSpot公式に該当データなし**。「インラインCTA vs サイドバーCTA」の比較で、noteには適用不可。即修正済み。

## Phase 2: パフォーマンス分析

Notion の記事管理DB と context/*-performance.md からデータを取得して分析する。

**分析内容:**

## Phase 3: 改善提案生成（必須チェックリスト適用）

> **2026-04-27 追加**: 各提案を出す前に**必須チェックリスト**を全項目通過させる。1項目でも落ちたら「却下した提案」セクションへ移動する。

### 必須チェックリスト（提案ごとに全項目通過必須）

```
- [ ] 根拠が実データに紐づいているか（ファイルパス:行番号 / Notion DB / WebSearch URL のいずれか具体的に）
- [ ] 主観表現を使ってない or 定量データで補強済み
  （NG: 「過多」「気がする」「ぱっと見」「たぶん」「〜のはず」「〜と思う」）
- [ ] 改修コスト（分単位）の見積もりあり
- [ ] 期待効果が定量目標で書かれている（「PV ○→○」「○件回避」「ER ○%→○%」等）
- [ ] ROI 計算: コスト < 効果 が成立
- [ ] 現フォロワー段階での再現性が確認済み
  （hyui の現状: X 8人 / Threads 489人 / note PV2,406 全期間）
- [ ] 大規模アカウント前提のデータを小規模に当てはめていないか確認済み
- [ ] 代替案との比較が済んでいる
```

1項目でも落ちたら → **「却下した提案」セクションへ移動**

### 提案テンプレート（自由記述禁止・構造化必須）

```
【提案X】タイトル
判定: ✅ やる価値あり / 🟡 条件付き / ❌ やる価値なし

根拠データ:
- {データ1: ファイルパス:行番号 / Notion DB / WebSearch URL + 具体的数値}
- {データ2: 同上}

改修コスト: {分} / 難易度: {低・中・高}
期待効果: {定量目標} / 達成見込み: {高・中・低}
ROI: {コスト < 効果 → ✅ / コスト > 効果 → ❌}

8〜500フォロワー段階での再現性: {確認済 / 大規模前提で要警戒}
代替案: {なし / 別案: ○○}
```

→ 自由記述では記憶ベース提案が混入する。**構造化テンプレで書きづらくする**ことで強制的に検証させる。

## Phase 3.5: 提案の自己批判（Phase 3 の直後に実施・必須）

Phase 3 で出した提案リストの全件について、以下3問に YES と答えられるか検証:

```
Q1. これは大規模アカウント前提のデータを小規模に当てはめてないか？
    （HubSpot等の大手調査データ・「○億PV規模のブログ」のベンチマーク等）
Q2. これは記憶ベースの印象論じゃないか？データで実際にカウント・確認したか？
Q3. ROI 計算したか？コストと効果を「分単位 × 件数 × 確率」で計算したか？
```

**3問全部 YES → 提案リストに残す**
**1問でも NO → 「却下した提案」セクションへ移動**

→ Phase 3.5 を経た提案だけが最終アウトプットの「🟡 改善提案」に進める。

## 実行手順

1. このスキルは `/workflow-audit` で呼び出す
2. 全フェーズを順番に実行する（**Phase 0 / Phase 1.5 / Phase 3.5 はスキップ禁止**）
3. 出力フォーマットの内容を Discord リードWebhookに通知する
4. 「🔴 要対応」「🟡 改善提案」がある場合は、ユーザー承認後に実装する
5. 「❌ 削除」のクレームは即修正（ユーザー承認不要・出典不正確の事実は確定）
6. 修正完了後、`context/content-memory.md` に監査結果と「却下した提案」を追記する

---

## 注意事項（必須遵守）

- Notionデータが取得できない場合は「Phase 2: データ取得失敗」と明記して Phase 3 へ
- ルールの削除・変更は提案のみ行う。実際の修正はユーザー承認後（ただし Phase 1.5 の「❌ 削除」は除く）
- 「記録なし（初期状態）」のデータが多い場合は「まだ学習データが不足している」と明記して次回に回す
- **Phase 3 の必須チェックリスト・Phase 3.5 の自己批判はスキップ禁止**。スキップして「🟡 改善提案」に出すのは規定違反
- **「却下した提案」を空欄にしない**: 検討して却下したものは必ず明記（同じ提案が次回再浮上することを防ぐため）
- 出典付きクレームを新たに SKILL.md / context に追記する場合、その時点で WebSearch / WebFetch で裏取りする義務あり
