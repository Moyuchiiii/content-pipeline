# workflow-audit スキル

note-run / x-run スキルの定期監査を実行する。週1回を目安に実行し、スキルの品質を維持する。

---

## Phase 1: スキル定義スキャン

以下のファイルを読んで陳腐化・問題点を検出する。

**読み込むファイル:**
- `.claude/skills/note-run/SKILL.md`
- `.claude/skills/x-run/SKILL.md`
- `context/writing-rules.md`
- `context/article-frameworks.md`
- `context/content-memory.md`
- `context/note-strategy.md`
- `context/x-strategy.md`

**検出対象:**

1. **最終更新から30日以上経過したルール**
   - ファイルの更新日時を確認
   - 「競合サービスの仕様」「プラットフォームのアルゴリズム」に関するルールは特に陳腐化しやすい
   - 日付付きのコメント（例: `<!-- 2025-01-01 時点の情報 -->`）を探す

2. **Notionの実データと乖離している設定**
   - `context/note-performance.md` と `context/x-performance.md` の実績データを確認
   - スキルのスコアリング基準と実際のパフォーマンスが一致しているか検証
   - 「スキ数が多いとされているフォーマット」と実際のスキ数TOP記事を照合

---

## Phase 2: パフォーマンス分析

Notionの記事管理DBからデータを取得して分析する。

```
# Notion DB から記事データ取得
mcp__notion__notion-fetch または WebFetch で記事管理 DB を参照
```

**分析内容:**

1. **スキ数・売上・PVのトレンド変化**（直近30日 vs 3ヶ月前）
   - 上昇傾向のある記事タイプを特定
   - 下降傾向のある記事タイプを特定

2. **記事タイプ別のパフォーマンス比較**
   - 速報 / 実録 / ノウハウ / 有料記事 のスキ数・売上の平均値を比較
   - 最も ROI が高い記事タイプを特定

3. **フック型別のエンゲージメント**
   - `context/content-memory.md` のフック型別成功率テーブルを参照
   - エンゲージメントが高いフック型の傾向を分析

---

## Phase 3: 改善提案生成

Phase 1・2 の結果をもとに以下を出力する。

**削除すべき陳腐化したルール（理由付き）:**
- 30日以上更新されていないプラットフォーム仕様に関するルール
- 実データと逆の結果になっているルール（例: 「短いタイトルが良い」と書いてあるが実際は長いタイトルの方がスキが多い）
- 重複しているルール（別ファイルと同じ内容が書かれている箇所）

**追加すべき新ルール（根拠付き）:**
- 直近30記事で高パフォーマンスだったパターン
- `context/content-memory.md` の改善メモから抽出したルール

**スコアリング基準の調整提案:**
- note-run / x-run のスコアリング（A〜D等）と実際のパフォーマンスの乖離を修正する提案

**記事タイプ比率の最適化提案:**
- 現在の記事タイプ比率（速報:実録:ノウハウ = X:Y:Z）
- 推奨比率（パフォーマンスデータに基づく）

## 実行手順

1. このスキルは `/workflow-audit` で呼び出す
2. 全フェーズを順番に実行する（スキップ不可）
3. 出力フォーマットの内容を Discord リードWebhookに通知する
4. 対応が必要な項目がある場合は、ユーザーに確認を取ってから修正する
5. 修正完了後、`context/content-memory.md` に監査結果を追記する

---

## 注意事項

- Notionデータが取得できない場合は「Phase 2: データ取得失敗」と明記してPhase 3に進む
- ルールの削除・変更は提案のみ行う。実際の修正はユーザー承認後に実施する
- 「記録なし（初期状態）」のデータが多い場合は「まだ学習データが不足している」と明記して次回に回す
