# /note-run スキル

## オプション: 深掘りリサーチモード

`deep_research: true` を指定すると、Phase 3（競合調査）とPhase 6（品質ゲート・根拠スウィープ）が強化版になる。

- デフォルト: true（常に深掘りリサーチを実行）
- オフにしていい場面: 速報記事で時間優先の場合、事実関係が明確でソースが公式のみの場合

note.com向けの記事を**完全自動で**生成するスキル。トピック選定から最終稿まで人間の介入不要。

## Phase 0: コンテキスト読み込み + PDCA自動学習

以下を読み込んで現状を把握する：
1. `context/note-profile.md` — プロフィール・ポジショニング
2. `context/existing-articles.md` — 過去記事（テーマ重複回避）
3. `context/voice-samples.md` — 文体サンプル
4. `context/note-strategy.md` — 収益化戦略・ノウハウ
5. `context/note-performance.md` — 過去記事のパフォーマンスデータ（PV・スキ数・売上）
6. Notion「note記事管理」DB（data_source: `collection://812aa728-8d3e-42e4-a9cd-6a91c303b2c2`）
   - 過去記事のパフォーマンス（PV・スキ数・売上）
   - 現在進行中の記事があるか
   - 直近のトレンド（どんなテーマが反応良かったか）
7. Notion「ネタ帳」DB（data_source: `collection://1a603b4f-d1e4-4ed7-8c75-c7c0a5b7e595`）
   - ステータスが「未使用」のネタ候補を取得
   - 過去に見送ったネタの理由も確認（同じネタを再スキャンしないため）
8. `context/writing-rules.md` — 証拠優先ルール・AI禁止フレーズリスト・声クローンフロー
9. `context/article-frameworks.md` — AIDA・Star Story Solution・Before/Afterフレームワーク
10. `context/content-memory.md` — 過去の成功・失敗パターン蓄積データ（あれば参照）

### PDCA自動学習（過去データから成功パターンを抽出）

Notion記事管理DBの全レコードを取得し、以下を自動分析する：

**パフォーマンス分析:**
- スキ数が最も多かった記事のテーマ・構成・タイトルパターン
- PVが最も多かった記事の特徴
- 有料記事の売上ランキング
- 無料記事のうちスキ数が多かったもの（有料化候補）

**成功パターンの抽出:**
- タイトルの型: 「〇〇してみた」「〇〇の最強の使い方」等、どの型がスキ数多いか
- 記事の長さ: 何文字の記事が最も反応良いか
- 記事タイプ: 速報/実録/ノウハウのどれが反応良いか
- タグ: どのタグの記事が反応良いか

**直近の出稿状況:**
- 直近5本の記事タイプの内訳（速報○本/実録○本/ノウハウ○本）
- 直近5本の無料/有料比率
- 最後に投稿した日からの経過日数

これらの分析結果を Phase 2 のスコアリングに反映する。

- "Claude" 新機能 OR リリース OR アップデート

## Phase 2: トピック選定（自動判定）

Phase 1の結果と `context/note-strategy.md` のノウハウを使って、最適なトピックを自動選定する。

### まずネタ帳をチェック
Phase 1で新規スキャンする前に、ネタ帳DBの「未使用」ネタを確認する。
- mcp__notion__notion-fetch で `collection://1a603b4f-d1e4-4ed7-8c75-c7c0a5b7e595` を参照
- 未使用のネタの中に、今日出すべきもの（緊急度が上がったもの等）がないか確認
- ある場合は新規スキャン結果と合わせてスコアリング
- ない場合はPhase 1のスキャン結果のみでスコアリング

### 今日すでに書いた記事を除外する
Notion「note記事管理」DB（`collection://812aa728-8d3e-42e4-a9cd-6a91c303b2c2`）から今日（実行日）作成されたレコードを取得し、そのトピック・キーワードに重複する候補をスコアリング対象から除外する。
同じ日に複数回 `/note-run` を実行しても、同じネタを二重に書かないようにするため。

### 判定基準（スコアリング）

各候補に以下の5項目で1〜5点をつけ、合計スコアが最も高いものを選ぶ：

| 項目 | 判定基準 | 根拠 |
|---|---|---|
| **市場の大きさ** | そのトピックに関心がある人の推定数。X検索やnote検索で関連投稿数を確認 | bookの3要素 |
| **悩みの深さ** | 「お金を払ってでも解決/知りたい」レベルか。HARM法則（Money/Ambition）に当てはまるか | bookのHARM法則 |
| **緊急度** | 「今すぐ知りたい」か。新機能リリース直後は緊急度が高い | bookの緊急度 |
| **差別化可能性** | 既にnoteで同テーマの記事が溢れていないか。自分の実体験で語れるか | 競合調査 |
| **過去実績との相性** | 過去記事で反応が良かったテーマとの類似性。`context/note-performance.md`を参照 | PDCAフィードバック |

### スコアに基づく判定 + バランス制御

**基本判定:**
- スコア20点以上 → 有料記事（500円）として作成
- スコア15〜19点 → 無料記事として作成
- スコア14点以下 → 見送り。次の候補に進む

**バランス制御（基本判定を上書きする場合あり）:**
Phase 0で分析した直近の出稿状況に基づき、以下のルールで無料/有料を調整する：

1. **有料連投防止**: 直近5本のうち有料が2本以上 → 今回はスコアに関係なく無料にする
2. **無料連投時の有料チャンス**: 直近5本が全部無料 → スコア18点以上なら有料にする（閾値を2点下げる）
3. **初期フォロワー制限**: Notionの記事管理DBの総記事数が10本未満 → 有料率を20%以下に制限（5本に1本まで）
4. **記事タイプ分散**: 直近5本が同じ記事タイプ → 別タイプを優先（速報ばかりにならないように）

**PDCAフィードバック加算:**
Phase 0で抽出した成功パターンに該当する場合、スコアにボーナスを加算：
- 過去最もスキ数が多かったテーマと同系統 → +2点
- 過去最もスキ数が多かったタイトルパターンと同じ型 → +1点
- 過去反応が悪かったテーマと同系統 → -2点

### 記事タイプの自動判定
- Claude/Anthropicの新機能リリースから48時間以内 → **速報記事（sokuho）**
- Claude Codeを実務で使った体験がある → **実録記事（jituroku）**
- Claude/Claude Codeの使い方・活用法 → **ノウハウ記事（knowhow）**

### エラーハンドリング

**ニュースが見つからない場合:**
Phase 1でスキャン結果が0件の場合：
1. ネタ帳DBの「未使用」ネタを確認
2. 未使用ネタがあれば、その中からスコアリングして最適なものを選ぶ
3. 未使用ネタもない場合 → 「今日は速報ネタがありません。ノウハウ記事を書きますか？」とユーザーに確認

**全候補がスコア14点以下の場合:**
→ 「本日は投稿を見送ります。理由: {各候補のスコアと見送り理由}」と報告して終了

**スコアリング結果の表示:**
採否に関わらず、全候補のスコア内訳を表示する：
```
📊 トピック候補スコアリング結果:
1. [トピック名] — 合計: 22点 (市場5/悩み4/緊急5/差別4/実績4) → ✅ 採用（有料500円）
2. [トピック名] — 合計: 16点 (市場3/悩み3/緊急4/差別3/実績3) → 📋 ストック
3. [トピック名] — 合計: 12点 (市場2/悩み2/緊急3/差別3/実績2) → ❌ 見送り
```

### Notion記録: 記事レコード作成
トピックが決まったら、Notionに新しいレコードを作成する:
- mcp__notion__notion-create-pages で以下を登録:
  - parent: { "data_source_id": "812aa728-8d3e-42e4-a9cd-6a91c303b2c2" }
  - 記事名: （仮タイトル）
  - ステータス: "進行中"
  - 記事タイプ: "速報" / "実録" / "ノウハウ"
  - スコア: （算出したスコア）
  - トピック: （選定理由・元ネタ情報）
  - 価格: （推奨価格）
- 作成されたページURLを記録しておく（Phase 8で更新するため）

### ネタ帳のステータス更新
- 採用したネタ → ステータスを "採用済み" に更新（mcp__notion__notion-update-page）
- スコアが低くて見送ったネタ → ステータスを "見送り" に更新、見送り理由を記入
- それ以外 → "未使用" のまま残す（将来使う可能性あり）

## Phase 3: 競合調査 + Deep Research

選定したトピックについて、note上の競合調査と多角的なリサーチを行う。
詳細なリサーチ手順は `context/research-protocol.md` を参照すること。

### 基本調査（常に実行）

**YOU MUST: 以下を必ず実行してから構成設計に進むこと。**

1. WebSearchで `site:note.com {トピックのキーワード}` を検索
2. 上位5記事のタイトル・スキ数・価格を確認
3. 差別化ポイントを特定:
   - 競合が書いていない角度は何か
   - 自分の実体験で追加できる情報は何か
   - 「最強の使い方」を実演できるか

**売上推測:** スキ数 × 記事価格 × 1.5 ≒ おおよその売上。スキ100以上の有料記事があれば「売れるテーマ」と判断。

### deep_research: true の場合（強化版 — `context/research-protocol.md` に従う）

**Step 1: サブクエスチョン分解**
トピックを 3〜5 つのサブクエスチョンに分解する（例: 仕組み・事例・競合比較・注意点・差別化）

**Step 2: クエリバリエーション生成（合計 15〜30 本）**
各サブクエスチョンに 2〜3 バリエーションのクエリを作成して実行:
- 一次情報: `"{キーワード}" site:anthropic.com OR site:github.com`
- 日本語コミュニティ: `"{キーワード}" site:note.com` / `site:zenn.dev`
- 実践事例: `"{キーワード}" 使い方 OR 活用 OR やってみた`
- 批判・制限: `"{キーワード}" 問題 OR 注意 OR limitation`

**Step 3: ソース収集と信頼度タグ付け**

**NEVER: ⚠️参考情報のみを根拠に事実として記述すること。**

- ✅ 一次データ: 公式発表、自分の実験結果
- 🔶 二次データ: 信頼できるテックブログ・Zenn 記事
- ⚠️ 参考情報: 未検証の口コミ・匿名情報・推測

**Step 4: 核心 URL の全文取得**
最も信頼度が高い 3〜5 URL を `WebFetch` で全文取得する。スニペットだけで判断しない。

**MCP強化オプション（インストール済みの場合）:**
- `Exa MCP` が利用可能なら WebSearch の代わりに使う（AI最適化検索で精度向上）
- `Firecrawl MCP` が利用可能なら WebFetch の代わりに使う（失敗率が低くフル本文取得に強い）
- `YouTube Transcript MCP` が利用可能なら、競合ニュースの解説動画があれば字幕テキストも取得する

**Step 5: クロス検証**
- 同じ情報が複数の独立ソースで確認できるか確認する
- 単一ソースの主張は「〇〇によると」と出典を明示し断定しない
- 収集結果を `drafts/research/research_{日付}_{トピック}.md` に保存する

収集・検証した情報は Phase 5（執筆）と Phase 6（品質ゲート・根拠スウィープ）で参照する。

## Phase 4: 構成設計 + タイトル最適化

### タイトルA/Bテスト

記事のタイトル候補を**5本**生成し、以下の基準でスコアリングして最適なものを選ぶ。

**タイトル生成の型（過去の成功パターンから選択）:**
- 数字型: 「〇〇したら毎日2時間浮いた話」
- 方法型: 「〇〇の最強の使い方」
- 体験型: 「大学生が〇〇してみたら」
- 疑問型: 「〇〇って何ができるの？」
- 速報型: 「〇〇が出たので即日試してみた」
- 比較型: 「〇〇 vs △△、実際に使って比較」

**タイトルスコアリング（各20点満点）:**

| 項目 | 判定基準 |
|---|---|
| **検索キーワード** | ターゲットが検索しそうなワードが含まれているか（0-5点） |
| **クリック欲求** | 「読みたい」と思わせる好奇心・ベネフィットがあるか（0-5点） |
| **具体性** | 数字・固有名詞が入っているか（0-5点） |
| **過去成功パターン類似度** | Phase 0で抽出した成功タイトルの型に合致するか（0-5点） |

**AIDAフレームワーク適用:**
タイトル生成時に `context/article-frameworks.md` の AIDA原則を参照すること。
- Attention（注意）: 冒頭で読者の目を止める
- Interest（興味）: 「これは自分ごとだ」と思わせる
- Desire（欲求）: 「読んだら得をする」と感じさせる
- Action（行動）: クリックという行動を引き起こす

**表示フォーマット:**
```
🏷️ タイトル候補:
1. [タイトルA] — 16/20点 ✅ 採用
2. [タイトルB] — 14/20点
3. [タイトルC] — 13/20点
4. [タイトルD] — 11/20点
5. [タイトルE] — 10/20点
```

### 構成テンプレート

以下のテンプレートから記事タイプに合った構成を選び、トピックに合わせてカスタマイズする。

### 速報記事の構成（800〜1,500字）

**⚠️ テンプレートはあくまで参考。「番号付きの手順列挙」は避けること。**

以下の要素を含める（順番・見出し名は記事の流れに合わせて自由に変える）:
- 何が出たか・何が変わるか（読者が最初に知りたいこと）
- 公式情報の要点
- 自分が実際に使ってみて「これは使える/使えない」と思ったこと（一次情報）
- 副業や日常でどう使えるか（ユースケースは1〜2つに絞る。「3つ」に機械的に合わせない）
- 締め（一言の価値要約）

記事全体で「自分が試してみた人」として語ること。機能説明の羅列にしない。

### 実録記事の構成（2,000〜4,000字）

以下の要素を含める（順番・見出し名は記事の流れに合わせて自由に変える）:
- 読者の共感を得る導入（「自分もこういう状況だった」から入る）
- 何をしたかったか・なぜやったか
- Claude Codeでどう解決したか（体験談として語る）
- Before/After（数字で示す）
- つまずいたこと・失敗談（必須。完璧な成功話はAI感が出る）
- この経験から得た知見（「〜だとわかった」のような発見の形で）
- 次の読者へ（フォロー誘導・次回予告）

### ノウハウ記事の構成（2,500〜5,000字）

以下の要素を含める（順番・見出し名は記事の流れに合わせて自由に変える）:
- 読者が抱えている問題（「こういうことで困ったことない？」から入る）
- この記事で得られること（約束）
- 具体的な手順・方法（「ステップ1:」という形式は極力避ける。体験談として流れで説明する）
- つまずきポイント（自分が実際に詰まったことのみ）
- 応用（他にどう使えるか、1〜2個に絞る）
- 締め・次回予告

## Phase 5: 執筆

### 絶対ルール
- **CRITICAL — 非エンジニア向け執筆**: `context/writing-rules.md` の「非エンジニア向け執筆ルール」を必ず守る。技術用語は初出で平易な日本語で説明。コードを載せる場合は「何をするか」を先に1文で説明。「難しそう」な工程には「コピペするだけ」「1回やればOK」を添える。
- **CRITICAL — 冒頭アイランド問いかけ（必須）**: 記事の一番最初は必ず以下のパターンで始める。`<p>` で始めることは禁止。
  ```html
  <hr>
  <p style="text-align: center;"><b>（記事テーマに合わせた読者の痛みを突く問いかけ）</b></p>
  <hr>
  ```
  問いかけの作り方: 「まだ〇〇していますか？」「〇〇、毎回ゼロからやっていませんか？」のように、読者が「あ、自分のことだ」と感じる1文。文言は記事ごとに変える（固定文言禁止）。
- **証拠優先**: セクションは必ず「数字・具体例・コード・スクリーンショット」から始める。「〜は重要です」「〜が便利です」から始めない。`context/writing-rules.md` の証拠優先ルールを参照
- **禁止パターン厳守**: `context/writing-rules.md` の禁止フレーズリストを見ながら書く。生成後に必ずスキャン
- 文体は `context/voice-samples.md` に合わせる
- 一人称: 「自分」「僕」
- AI定型フレーズ禁止: 「〜と言えるでしょう」「〜ではないでしょうか」「重要なのは」の連発
- 括弧（）での補足説明の多用禁止
- 数字・実績は正確に（盛らない）
- クライアント情報は匿名化
- 有料記事の場合: 無料エリア（冒頭30〜40%）で十分な価値を見せてから有料ライン

### 出力フォーマット: noteエディタ対応HTML

記事はnoteエディタにHTMLペーストで入力するため、以下のHTML形式で出力すること。

**使用するHTMLタグ:**
- `<h2>` — メインセクションの見出し
- `<h3>` — サブセクション（具体例、手順の各ステップ等）
- `<p>` — 通常の段落
- `<b>` — 強調（キーポイント、Before/After等）
- `<blockquote>` — 引用（公式情報の引用、ユーザーへの指示文、印象的な一文）**＋ 箇条書き的な列挙にも使う**（noteの引用ブロックは視認性が高いため、ポイント列挙・できること一覧・特徴リスト等は `<blockquote>` で1項目ずつ囲む）
- `<ul><li>` — **原則使わない**。箇条書き的な列挙は `<blockquote>` を使うこと。`<ul><li>` は「あわせて読みたい」のリンクリスト等、装飾不要の短いリストにのみ使用
- `<ol><li>` — 番号付きリスト（手順、「この先で手に入るもの」等）。ステップ形式で順序が重要な場合のみ使用
- `<pre><code>` — コードブロック（コマンド、設定値、コード例）
- `<hr>` — 区切り線（セクション間、有料ラインの前後）
- `<p style="text-align: center;">` — 中央寄せ（有料ラインの境界メッセージ）
- `<br>` — 段落内改行（Before/Afterの並列表示等）

**装飾の使い分けガイド:**
- h2は記事全体で3〜6個。大きなセクション区切り
- h3はh2の下に2〜4個。具体例や手順の各ステップ
- 引用（blockquote）は「ユーザーがAIに送る指示文」「公式情報の要約」「印象的な一言」に加え、**箇条書き的な列挙（ポイント・特徴・できること一覧等）にも積極的に使う**。noteでは引用ブロックが視覚的に映えるため、`<ul><li>`より`<blockquote>`を優先する
- **blockquote内でラベルをつける場合: `ラベル: テキスト` の形は禁止。`<b>ラベル</b><br>テキスト` の形にする。例: `<blockquote><b>CLAUDE.mdなしの場合</b><br>セッションのたびに説明し直す</blockquote>`**
- コードブロック（pre>code）は「コマンド」「設定値」「コード例」に使う
- 中央寄せは有料ラインの境界メッセージにのみ使う（多用禁止）
- 区切り線（hr）は有料ラインの前後、大きなセクション転換時に使う
- 番号付きリストは「この先で手に入るもの」や手順に使う

**有料記事の構成:**
- 有料ライン前に `<hr>` + 中央寄せメッセージ + 「この先で手に入るもの」リスト + `<hr>`
- **コロン（:）禁止ルール: 記事本文・見出し・中央寄せテキスト等、コードブロック外ではコロンを使わない。「この先で手に入るもの:」→「この先で手に入るもの」のようにコロンを削除する。AI感が出る最大の原因の一つ。**

### 有料記事の無料エリア設計（bookのノウハウ）
- 無料エリアだけでも「役に立った」と思わせる
- 有料エリアには「実際のコード/プロンプト全文」「具体的な数字」「テンプレート」等の実践的価値を入れる
- 無料→有料の境界に「この先で手に入るもの」リストを入れる

## Phase 9: PDCAフィードバック記録

`context/note-performance.md` に以下を追記するよう促す:
- 記事タイトル・URL・公開日
- 投稿後1週間のPV・スキ数
- 売上（有料記事の場合）
- 学び（何が良かった/悪かったか）

このデータは次回のPhase 2（トピック選定）で自動参照される。

### Notion記録: パフォーマンス更新の案内
記事投稿後、以下をNotionに記録するよう促す:
- noteURL: 投稿後のURL
- 公開日: 投稿日
- 投稿1週間後に以下を更新:
  - PV: ページビュー数
  - スキ数: スキの数
  - 売上: スキ数 × 価格 × 1.5
  - メモ: 学び・次回への示唆

### PDCAメモの記録
記事管理DBのメモ欄に、以下のフォーマットでPDCA記録を残すよう促す:

**投稿直後に記録:**
- 投稿日時
- noteURL
- 使ったタグ一覧

**1週間後に記録:**
- PV数
- スキ数
- コメント数
- 高評価数（購入手続きバナー下に表示）
- 売上推計（スキ数 × 価格 × 1.5）

**振り返り:**
- 何が良かったか（タイトル？内容？タイミング？）
- 何が悪かったか
- 次回への改善アクション（具体的に1つ）

このPDCAデータは次回の /note-run 実行時にPhase 0で自動参照される。

### voice-samples.md 自動更新

Phase 0で取得したNotionの記事管理DBから、**スキ数上位3記事**を特定し：
1. 該当記事の草稿ファイル（drafts/内）を読み込む
2. 導入部・まとめ部・印象的な一文を抽出
3. `context/voice-samples.md` の「サンプル文」セクションに追記する
4. 既に登録済みのサンプルと重複する場合はスキップ
5. サンプルが10個を超えたら、スキ数が低い記事のサンプルを削除（最新・高パフォーマンスを優先）

これにより、成功した記事の文体が自動的に学習される。
