# /note-run スキル

## オプション: 深掘りリサーチモード

`deep_research: true` を指定すると、Phase 3（競合調査）とPhase 6（品質ゲート・根拠スウィープ）が強化版になる。

- デフォルト: true（常に深掘りリサーチを実行）
- オフにしていい場面: 速報記事で時間優先の場合、事実関係が明確でソースが公式のみの場合

note.com向けの記事を**完全自動で**生成するスキル。トピック選定から最終稿まで人間の介入不要。

- 直近5本の無料/有料比率

これらの分析結果を Phase 2 のスコアリングに反映する。

ネタ探し機能は `/source-run` に統合済み。note-run はネタ帳から候補を読み込む。

各レコードから以下を抽出:

### Step 2: 候補ゼロ時のフォールバック

     - `非エンジニア AI 副業 OR 稼ぐ`
   - 取得した候補をスキャンメモ用にローカル変数に保持（ネタ帳には投入しない）

### Step 3: Phase 2 への引き渡し

取得した候補リストを Phase 2 のスコアリング対象として渡す。

## Phase 2: トピック選定（自動判定）

Phase 1の結果と `context/note-strategy.md` のノウハウを使って、最適なトピックを自動選定する。

### 今日すでに書いた記事を除外する
Notion「記事管理」DB（`collection://812aa728-8d3e-42e4-a9cd-6a91c303b2c2`）から今日（実行日）作成されたレコードを取得し、そのトピック・キーワードに重複する候補をスコアリング対象から除外する。
同じ日に複数回 `/note-run` を実行しても、同じネタを二重に書かないようにするため。

### 判定基準（スコアリング）

各候補に以下の6項目で1〜5点をつけ、合計スコアが最も高いものを選ぶ：

| 項目 | 判定基準 | 根拠 |
|---|---|---|
| **市場の大きさ** | そのトピックに関心がある人の推定数。X検索やnote検索で関連投稿数を確認 | bookの3要素 |
| **悩みの深さ** | 「お金を払ってでも解決/知りたい」レベルか。HARM法則（Money/Ambition）に当てはまるか | bookのHARM法則 |
| **緊急度** | 「今すぐ知りたい」か。新機能リリース直後は緊急度が高い | bookの緊急度 |
| **差別化可能性** | 既にnoteで同テーマの記事が溢れていないか。自分の実体験で語れるか | 競合調査 |
| **過去実績との相性** | 過去記事で反応が良かったテーマとの類似性。`context/note-performance.md`を参照 | PDCAフィードバック |
| **非エンジニア適合度** | (a) 主題が非エンジニアの副業課題か（Claude Code機能解説・プロジェクト構造・CLAUDE.md運用・プラグイン等が主役の記事は低スコア）。(b) エンジニア知識なしで読めるか。(c)「自分にもできそう」と思えるか。技術寄りトピックは上限3点 | ターゲット層（最重要） |

### スコアに基づく判定 + バランス制御

**CRITICAL — ハードフロア（全項目より優先）:**
- **非エンジニア適合度が 4 点未満の候補は合計点に関わらず問答無用で不採用**。Claude Code機能解説・プロジェクト構造話・CLAUDE.md運用ノウハウ等、主題がエンジニア向けの候補はここで弾く
- 非エンジ適合度を満たすまで次の候補にドロップダウンする。全候補が3点以下なら「本日は非エンジニア向けネタなし」で見送り、スキャンBの副業ネタを再生成する

**基本判定（ハードフロア通過後に適用）:**
- スコア24点以上 → 有料記事（500円）として作成
- スコア18〜23点 → 無料記事として作成
- スコア17点以下 → 見送り。次の候補に進む

**バランス制御（基本判定を上書きする場合あり）:**
Phase 0で分析した直近の出稿状況に基づき、以下のルールで無料/有料を調整する：

1. **有料連投防止**: 直近5本のうち有料が2本以上 → 今回はスコアに関係なく無料にする
2. **無料連投時の有料チャンス**: 直近5本が全部無料 → スコア22点以上なら有料にする（閾値を2点下げる）
3. **初期フォロワー制限**: Notionの記事管理DBの総記事数が10本未満 → 有料率を20%以下に制限（5本に1本まで）
4. **記事タイプ分散**: 直近5本が同じ記事タイプ → 別タイプを優先（速報ばかりにならないように）

**PDCAフィードバック加算:**
Phase 0で抽出した成功パターンに該当する場合、スコアにボーナスを加算：
- 過去最もスキ数が多かったテーマと同系統 → +2点
- 過去最もスキ数が多かったタイトルパターンと同じ型 → +1点
- 過去反応が悪かったテーマと同系統 → -2点

**CRITICAL — PDCA加算の制約:**
過去実績の加算は**非エンジニア適合度のハードフロアを迂回しない**。PDCA加算でスコアを釣り上げて有料化するのは禁止。過去の勝ち筋が技術寄り記事だったとしても、それを理由にエンジニア向けトピックを採用してはいけない。PDCA加算は「非エンジ視点で書ける副業・実録ネタの中で、どれが一番過去実績と近いか」の判定にのみ使う。

### 記事タイプの自動判定
- Claude/Anthropicの新機能リリースから48時間以内 → **速報記事（sokuho）**
- 副業・フリーランスでAI/Claudeを使った体験・実績がある → **実録記事（jituroku）**
- AI/Claudeを使って副業・作業を効率化する方法・手順 → **ノウハウ記事（knowhow）**

### エラーハンドリング

**ニュースが見つからない場合:**
Phase 1でスキャン結果が0件の場合：
1. ネタ帳DBの「未使用」ネタを確認
2. 未使用ネタがあれば、その中からスコアリングして最適なものを選ぶ
3. 未使用ネタもない場合 → 「今日は速報ネタがありません。ノウハウ記事を書きますか？」とユーザーに確認

**全候補がスコア17点以下の場合:**
→ 「本日は投稿を見送ります。理由: {各候補のスコアと見送り理由}」と報告して終了

**スコアリング結果の表示:**
採否に関わらず、全候補のスコア内訳を表示する：
```
📊 トピック候補スコアリング結果:
1. [トピック名] — 合計: 26点 (市場5/悩み4/緊急5/差別4/実績4/非エンジ4) → ✅ 採用（有料500円）
2. [トピック名] — 合計: 19点 (市場3/悩み3/緊急4/差別3/実績3/非エンジ3) → 📋 ストック
3. [トピック名] — 合計: 14点 (市場2/悩み2/緊急3/差別3/実績2/非エンジ2) → ❌ 見送り
```

### Notion記録: 記事レコード作成
トピックが決まったら、Notionに新しいレコードを作成する:
- mcp__notion__notion-create-pages で以下を登録:
  - parent: { "data_source_id": "812aa728-8d3e-42e4-a9cd-6a91c303b2c2" }
  - 記事名: （仮タイトル）
  - ステータス: "進行中"
  - 記事タイプ: "速報" / "実録" / "ノウハウ"
  - トピック: （選定理由・元ネタ情報）
  - note価格: （推奨価格）
- 作成されたページURLを記録しておく（Phase 8で更新するため）

### ネタ帳のステータス更新
- 採用したネタ → ステータスを "採用済み" に更新（mcp__notion__notion-update-page）
- スコアが低くて見送ったネタ → ステータスを "見送り" に更新、見送り理由を記入（**source-run 由来のネタの場合のみ。Step 2 フォールバックで取得した自前スキャン由来は更新不要**）
- それ以外 → "未使用" のまま残す（将来使う可能性あり）

## Phase 3: 競合調査 + Deep Research

選定したトピックについて、note上の競合調査と多角的なリサーチを行う。
詳細なリサーチ手順は `context/research-protocol.md` を参照すること。

### 基本調査（常に実行）

**YOU MUST: 以下を必ず実行してから構成設計に進むこと。**

1. WebSearchで `site:note.com {トピックのキーワード}` を検索
2. 上位5記事のタイトル・スキ数・価格を確認
3. 差別化ポイントを特定:
   - 競合が書いていない角度は何か
   - 自分の実体験で追加できる情報は何か
   - 「最強の使い方」を実演できるか

**売上推測:** スキ数 × 記事価格 × 1.5 ≒ おおよその売上。スキ100以上の有料記事があれば「売れるテーマ」と判断。

### deep_research: true の場合（強化版 — `context/research-protocol.md` に従う）

**Step 1: サブクエスチョン分解**
トピックを 3〜5 つのサブクエスチョンに分解する（例: 仕組み・事例・競合比較・注意点・差別化）

**Step 2: クエリバリエーション生成（合計 15〜30 本）**
各サブクエスチョンに 2〜3 バリエーションのクエリを作成して実行:
- 一次情報: `"{キーワード}" site:anthropic.com OR site:github.com`
- 日本語コミュニティ: `"{キーワード}" site:note.com` / `site:zenn.dev`
- 実践事例: `"{キーワード}" 使い方 OR 活用 OR やってみた`
- 批判・制限: `"{キーワード}" 問題 OR 注意 OR limitation`

**Step 3: ソース収集と信頼度タグ付け**

**NEVER: ⚠️参考情報のみを根拠に事実として記述すること。**

- ✅ 一次データ: 公式発表、自分の実験結果
- 🔶 二次データ: 信頼できるテックブログ・Zenn 記事
- ⚠️ 参考情報: 未検証の口コミ・匿名情報・推測

**Step 4: 核心 URL の全文取得**
最も信頼度が高い 3〜5 URL を `WebFetch` で全文取得する。スニペットだけで判断しない。

**MCP強化オプション（インストール済みの場合）:**
- `Exa MCP` が利用可能なら WebSearch の代わりに使う（AI最適化検索で精度向上）
- `Firecrawl MCP` が利用可能なら WebFetch の代わりに使う（失敗率が低くフル本文取得に強い）
- `YouTube Transcript MCP` が利用可能なら、競合ニュースの解説動画があれば字幕テキストも取得する

**Step 5: クロス検証**
- 同じ情報が複数の独立ソースで確認できるか確認する
- 単一ソースの主張は「〇〇によると」と出典を明示し断定しない
- 収集結果を `drafts/research/research_{日付}_{トピック}.md` に保存する

収集・検証した情報は Phase 5（執筆）と Phase 6（品質ゲート・根拠スウィープ）で参照する。

## Phase 4: 構成設計 + タイトル最適化

### タイトルA/Bテスト

記事のタイトル候補を**5本**生成し、以下の基準でスコアリングして最適なものを選ぶ。

**タイトル生成の型（過去の成功パターンから選択）:**
- 数字型: 「〇〇したら毎日2時間浮いた話」
- 方法型: 「〇〇の最強の使い方」
- 体験型: 「大学生が〇〇してみたら」
- 疑問型: 「〇〇って何ができるの？」
- 速報型: 「〇〇が出たので即日試してみた」
- 比較型: 「〇〇 vs △△、実際に使って比較」

**タイトルスコアリング（各20点満点）:**

| 項目 | 判定基準 |
|---|---|
| **検索キーワード** | ターゲットが検索しそうなワードが含まれているか（0-5点） |
| **クリック欲求** | 「読みたい」と思わせる好奇心・ベネフィットがあるか（0-5点） |
| **具体性** | 数字・固有名詞が入っているか（0-5点） |
| **過去成功パターン類似度** | Phase 0で抽出した成功タイトルの型に合致するか（0-5点） |

**AIDAフレームワーク適用:**
タイトル生成時に `context/article-frameworks.md` の AIDA原則を参照すること。
- Attention（注意）: 冒頭で読者の目を止める
- Interest（興味）: 「これは自分ごとだ」と思わせる
- Desire（欲求）: 「読んだら得をする」と感じさせる
- Action（行動）: クリックという行動を引き起こす

**表示フォーマット:**
```
🏷️ タイトル候補:
1. [タイトルA] — 16/20点 ✅ 採用
2. [タイトルB] — 14/20点
3. [タイトルC] — 13/20点
4. [タイトルD] — 11/20点
5. [タイトルE] — 10/20点
```

### 構成テンプレート

以下のテンプレートから記事タイプに合った構成を選び、トピックに合わせてカスタマイズする。

### 速報記事の構成（800〜1,500字）

**⚠️ テンプレートはあくまで参考。「番号付きの手順列挙」は避けること。**

以下の要素を含める（順番・見出し名は記事の流れに合わせて自由に変える）:
- 何が出たか・何が変わるか（読者が最初に知りたいこと）
- 公式情報の要点
- 自分が実際に使ってみて「これは使える/使えない」と思ったこと（一次情報）
- 副業や日常でどう使えるか（ユースケースは1〜2つに絞る。「3つ」に機械的に合わせない）
- 締め（一言の価値要約）

記事全体で「自分が試してみた人」として語ること。機能説明の羅列にしない。

### 実録記事の構成（2,000〜4,000字）

**フレーミングの順番（必須）: 困りごと → 結果 → 手段（Claude/AI）の順で語る。Claude Codeから入らない。**

以下の要素を含める（順番・見出し名は記事の流れに合わせて自由に変える）:
- 読者の共感を得る導入（「自分もこういう状況だった」「こういうことで困ってた」から入る）
- 何に困っていたか・何を達成したかったか（非エンジニアでも「あるある」と思える状況を先に出す）
- どうなったか・Before/After（数字で示す）← 結果を早めに出す
- 手段として「AIに頼んだ」「Claudeを使った」が登場（技術説明より「こう指示したらこうなった」体験談として）
- つまずいたこと・失敗談（必須。「自分もここで詰まった」が非エンジニアの安心感になる）
- この経験から得た知見（「〜だとわかった」のような発見の形で）
- 次の読者へ（「エンジニアじゃなくてもできた」を締めで強調する）

### ノウハウ記事の構成（2,500〜5,000字）

以下の要素を含める（順番・見出し名は記事の流れに合わせて自由に変える）:
- 読者が抱えている問題（「こういうことで困ったことない？」から入る）
- この記事で得られること（約束）
- 具体的な手順・方法（「ステップ1:」という形式は極力避ける。体験談として流れで説明する）
- つまずきポイント（自分が実際に詰まったことのみ）
- 応用（他にどう使えるか、1〜2個に絞る）
- 締め・次回予告

## Phase 5: 執筆

### 絶対ルール
- **CRITICAL — 段落細分化**: 1段落は60〜150字、1段落1メッセージ。400字超の段落禁止。印象的な一文・感情的な転換点・決定的な結論は独立1段落1文で切り出し、リズムを作る。例: `<p>でも、眠れなかった。</p>` `<p>それだけ。</p>`
- **CRITICAL — 太字は段落全体型のみ**: note エディタも tiptap 系のマーク継続挙動リスクあり。`<b>` / `<strong>` は `<p><b>全文</b></p>` の形で段落全体を囲む。段落中間の部分強調は禁止。強調したい一文は独立段落に分離してから太字化する。`grep -nE '<p>[^<]{5,}<b>' {file}` で0件必須
- **CRITICAL — 読者への問いかけ**: 各h2セクションに「あなたも〜ですか？」「〜じゃないですか？」「〜したことありますか？」等の問いかけを1回以上入れる。1記事で「あなた」が10回以上出るのが目安
- **CRITICAL — 非エンジニア向け執筆**: `context/writing-rules.md` の「非エンジニア向け執筆ルール」を必ず守る。技術用語は初出で平易な日本語で説明。コードを載せる場合は「何をするか」を先に1文で説明。「難しそう」な工程には「コピペするだけ」「1回やればOK」を添える。
- **CRITICAL — 冒頭アイランド問いかけ（必須）**: 記事の一番最初は必ず以下のパターンで始める。`<p>` で始めることは禁止。
  ```html
  <hr>
  <p style="text-align: center;"><b>（記事テーマに合わせた読者の痛みを突く問いかけ）</b></p>
  <hr>
  ```
  問いかけの作り方: 「まだ〇〇していますか？」「〇〇、毎回ゼロからやっていませんか？」のように、読者が「あ、自分のことだ」と感じる1文。文言は記事ごとに変える（固定文言禁止）。
- **証拠優先**: セクションは必ず「数字・具体例・コード・スクリーンショット」から始める。「〜は重要です」「〜が便利です」から始めない。`context/writing-rules.md` の証拠優先ルールを参照
- **禁止パターン厳守**: `context/writing-rules.md` の禁止フレーズリストを見ながら書く。生成後に必ずスキャン
- 文体は `context/voice-samples.md` に合わせる
- 一人称: 「自分」「僕」
- AI定型フレーズ禁止: 「〜と言えるでしょう」「〜ではないでしょうか」「重要なのは」の連発
- 括弧（）での補足説明の多用禁止
- 数字・実績は正確に（盛らない）
- クライアント情報は匿名化
- 有料記事の場合: 無料エリア（冒頭30〜40%）で十分な価値を見せてから有料ライン

### 出力フォーマット: noteエディタ対応HTML

記事はnoteエディタにHTMLペーストで入力するため、以下のHTML形式で出力すること。

**使用するHTMLタグ:**
- `<h2>` — メインセクションの見出し
- `<h3>` — サブセクション（具体例、手順の各ステップ等）
- `<p>` — 通常の段落
- `<b>` — 強調（キーポイント、Before/After等）
- `<blockquote>` — 引用（公式情報の引用、ユーザーへの指示文、印象的な一文）**＋ 箇条書き的な列挙にも使う**（noteの引用ブロックは視認性が高いため、ポイント列挙・できること一覧・特徴リスト等は `<blockquote>` で1項目ずつ囲む）
- `<ul><li>` — **原則使わない**。箇条書き的な列挙は `<blockquote>` を使うこと。`<ul><li>` は「あわせて読みたい」のリンクリスト等、装飾不要の短いリストにのみ使用
- `<ol><li>` — 番号付きリスト（手順、「この先で手に入るもの」等）。ステップ形式で順序が重要な場合のみ使用
- `<pre><code>` — コードブロック（コマンド、設定値、コード例）
- `<hr>` — 区切り線（セクション間、有料ラインの前後）
- `<p style="text-align: center;">` — 中央寄せ（有料ラインの境界メッセージ）
- `<br>` — 段落内改行（Before/Afterの並列表示等）

**装飾の使い分けガイド:**
- h2は記事全体で3〜6個。大きなセクション区切り
- h3はh2の下に2〜4個。具体例や手順の各ステップ
- **太字 `<b>` は段落全体型のみ**: `<p><b>全文</b></p>` の形で独立段落を丸ごと太字化する。段落中間・末尾の部分強調 `<p>...<b>重要</b>と続く</p>` は禁止（tiptap マーク継続バグで以降の段落も強制太字化する）。強調したい一文は独立段落に分離してから太字化すること
- **段落細分化**: 1段落は60〜150字。長い論理展開を1段落に詰め込まない。独立1行段落（「でも、眠れなかった。」「それだけ。」等）でリズムを作る
- 引用（blockquote）は「ユーザーがAIに送る指示文」「公式情報の要約」「印象的な一言」に加え、**箇条書き的な列挙（ポイント・特徴・できること一覧等）にも積極的に使う**。noteでは引用ブロックが視覚的に映えるため、`<ul><li>`より`<blockquote>`を優先する。ただし**連続3個以上の blockquote は禁止**（目次的な章リスト等は `<p><b>タイトル</b></p>` + `<p>概要</p>` の段落化で代替）
- **blockquote内でラベルをつける場合: `ラベル: テキスト` の形は禁止。`<b>ラベル</b><br>テキスト` の形にする。例: `<blockquote><b>CLAUDE.mdなしの場合</b><br>セッションのたびに説明し直す</blockquote>`**
- コードブロック（pre>code）は「コマンド」「設定値」「コード例」に使う
- 中央寄せは有料ラインの境界メッセージにのみ使う（多用禁止）
- 区切り線（hr）は有料ラインの前後、大きなセクション転換時に使う
- 番号付きリストは「この先で手に入るもの」や手順に使う

**有料記事の構成:**
- 有料ライン前に `<hr>` + 中央寄せメッセージ + 「この先で手に入るもの」リスト + `<hr>`
- **コロン（:）禁止ルール: 記事本文・見出し・中央寄せテキスト等、コードブロック外ではコロンを使わない。「この先で手に入るもの:」→「この先で手に入るもの」のようにコロンを削除する。AI感が出る最大の原因の一つ。**

### 有料記事の無料エリア設計（bookのノウハウ）
- 無料エリアだけでも「役に立った」と思わせる
- 有料エリアには「実際のコード/プロンプト全文」「具体的な数字」「テンプレート」等の実践的価値を入れる
- 無料→有料の境界に「この先で手に入るもの」リストを入れる

## Phase 6: 品質ゲート（100点ルーブリック + 7スウィープ編集）

### Step 0: 事実確認スウィープ（全件Web検索・必須）

**IMPORTANT: 記憶・学習データで判断することを禁止する。以下の全項目を必ずWebSearch/WebFetchで実際に調べてから判定すること。**

#### Anthropic 公式ハルシネーション対策テクニック（2026-04-23 追加・7手法）

[Anthropic 公式ガイド](https://platform.claude.com/docs/en/test-and-evaluate/strengthen-guardrails/reduce-hallucinations) の推奨テクニックを本 Step 0 に全て適用:

1. **「わからない」の許可** — 公式ソースで確認できなかったクレームは無理に書かず、「未確認のため本文から削除」と明示して削除する。虚偽の断言よりも削除が優先
2. **直接引用の強制** — 長文ドキュメント（20k トークン超）を読む場合、先に該当箇所を抜粋引用してから判定する
3. **引用検証** — 記事生成後、各クレームの裏付け引用を探して本文内に明示。見つからないクレームは撤回
4. **Chain-of-Thought 検証** — 最終判定前に「このクレームが正しいと言える根拠」を段階的に書き出してから判定
5. **外部知識制限** — 提供されたソース（WebFetch で取得した公式ページ）のみを使用。一般知識・記憶ベースの追加は禁止
6. **Best-of-N 検証（重要クレーム限定）** — 月収等の根幹数字は fact-checker エージェントで複数回検証
7. **Iterative Refinement** — 検証結果を受けて本文を修正し、修正後にもう一度 Step 0 を走らせる

記事全文を読んで、以下の種別に該当する「事実の主張」をすべて列挙する：

## Phase 8: 保存・出力

### 保存形式

**Step 1: 前回の `today/` を正規フォルダに移動**
`today/` フォルダの中身を確認し、ファイルがあれば正規フォルダへ移動してから新規保存を行う：
- `today/*.html` → `note/drafts/{記事タイプ}/`
- `today/*.md` → `note/drafts/{記事タイプ}/`
（ファイル名のタイプ部分から記事タイプを判定。`today/` が空ならスキップ）

**Step 2: 今回のファイルを2箇所に保存**

正規フォルダ（バックアップ・Git管理用）:
1. `note/drafts/{記事タイプ}/draft_{日付}_{トピック要約}.md`
2. `note/drafts/{記事タイプ}/draft_{日付}_{トピック要約}.html`

**`today/` フォルダ（投稿用・すぐ開ける場所）:**
3. `today/draft_{日付}_{トピック要約}.html` ← noteにペーストするのはこれ
4. `today/draft_{日付}_{トピック要約}.md`
5. `today/delivery_{日付}_{トピック要約}.md` ← **投稿セット永続化ファイル（下記「delivery ファイル」参照・必須）**

HTMLファイルはメタ情報なし、`<h2>`〜`</p>`のみの純粋なHTMLボディ。
`<html>`や`<body>`タグは不要。noteエディタにそのままペーストできる形式。

Markdownファイルのフォーマット（冒頭メタ情報）:
```
---
title: 記事タイトル
type: sokuho/jituroku/knowhow
price: 0 or 500
tags: [タグ1, タグ2, ...]
created: YYYY-MM-DD
status: final
score: {Phase 2のスコア}
---
```

**目的:** 完了サマリーの内容（サムネプロンプト・タグ・URL・投稿推奨・スコア・TODO）はチャット応答のみに存在し、セッション圧縮や次セッション遷移で失われる。記憶ベースで再生成するとスキル規定違反（例: サムネプロンプトが英語化）や情報欠落が起きる。これを防ぐため、完了サマリーをチャットに表示する**前**に、同じ内容を Markdown ファイルとして書き出して永続化する。

**書き出し先:** `today/delivery_{日付}_{トピック要約}.md`

### サムネイル生成プロンプト（Nano Banana / Gemini）

Gemini（Nano Banana Pro）でサムネイルを生成するためのプロンプトを出力する。

**記事タイプ別デザイン方針:**

| 記事タイプ | 背景 | アクセントカラー | 雰囲気 |
|---|---|---|---|
| 速報（sokuho） | オフホワイト+ソフトベージュ | コーラル #FF6B6B | 新鮮・速報感 |
| 実録（jituroku） | オフホワイト+ライトグレー | ディープミント #4FB99F | 実績・信頼感 |

### メインコピー / サブコピーの作り方（必須ロジック）

タイトルから以下の手順で2行コピーを抽出する。完了サマリーに出力する際は抽出結果をプロンプトに埋め込む。

**Step 1: メインコピー（7〜12文字）を作る。優先順位:**

1. **数字 × Before/After 型**（最強）: 記事に数字があればそれを軸にする
   - 例: `2日→3時間` `月14万` `0→3万円` `8時間→30分`
2. **強い動詞 × 結末型**: 結果を一言で
   - 例: `副業で詰んだ話` `CWで初受注` `全部AIに任せた`
3. **パワーワード単体型**: 感情を動かす1ワード
   - 例: `やらかした` `月収14万` `限界突破`

**Step 2: サブコピー（10〜18文字）を作る。メインコピーの補足:**

- メインが数字なら「何が / どうやって」を補足（例: `提案資料を自動化した話`）
- メインが動詞型なら「誰が / いつ」を補足（例: `文系女子大学生の2ヶ月`）
- タイトルの一部を切り出してもいいが、必ず18文字以内に圧縮する

**Step 3: NGパターン:**

- タイトル全文をコピーする → 親指サイズで読めない
- 「〜話」「〜記録」「〜について」等の冗語を残す
- 2段で合計25文字超え
- コピー内に「、。」等の句読点を入れる

**Step 4: 変換例:**

- タイトル: `提案資料づくりが2日→3時間になった。Claude Codeに「自分専用の資料ツール」を作らせた話`

  - メイン: `月14万` / サブ: `文系女子大学生の副業2ヶ月`

### pending_cta 保存（x-run 連携用）

記事の告知候補メタ情報を `x/pending_cta/` に JSON で保存する。
**告知文本体の生成は x-run に委譲**（このスキルでは生成しない）。x-run が daily-auto 実行時に自動判定して夜前半枠に組み込む。

ファイル名: `x/pending_cta/note_{YYYYMMDD}_{slug}.json`

保存内容:

```json
{
  "type": "note",
  "created_at": "2026-04-23T22:00:00+09:00",
  "source": "note/published/jituroku/20260423_xxx.md",
  "title": "{記事の最終タイトル}",
  "url": null,
  "atomic_ideas": [
    "{核心1: 記事の一番刺さる発見・体験}",
    "{核心2: Before/After or 数字}",
    "{核心3: 読むべき人・解決する悩み}"
  ],
  "image_path": "{noteサムネの絶対パス}",
  "article_type": "jituroku",
  "price": "free",
  "target_fit": "high",
  "suggested_priority": "normal"
}
```

**フィールドの判定基準:**

- `url`: 記事公開前は `null`。投稿後にユーザーまたは x-run が追記
- `atomic_ideas`: 本文から「独立して成立するXツイートになり得るアイデア」を3〜5個抽出
- `target_fit`: hyui（非エンジ副業初心者）ターゲットとの合致度
  - `high`: 副業・実録・初心者向け体験記
  - `medium`: AI活用Tips・ツール比較・ノウハウ
  - `low`: エンジニア寄り・技術深掘り・機能説明
- `suggested_priority`: `high`（即日告知したい）/ `normal`（通常）

**target_fit が `low` の場合、x-run 側で告知スキップされる** ことがある（`CLAUDE.md` のターゲット遵守ルール準拠）。

### Step 1: noteURL ヒアリング & Notion 自動更新

Phase 8 の保存完了直後に、ユーザーへ以下のフォーマットで質問する:

```

```

**URL が返ってきた場合の処理:**

1. `mcp__notion__notion-search` で 記事管理DB から記事名マッチでレコード取得
   - `data_source_url`: `collection://812aa728-8d3e-42e4-a9cd-6a91c303b2c2`
   - 記事名は Phase 8 で確定した最終タイトル
2. `mcp__notion__notion-update-page` で該当レコードを `update_properties` 更新:


- 「OK、投稿後に Notion UI から `noteURL` `note公開日` を手動入力するか、次回 /note-run 実行時に改めて聞くよ」と案内
- `ステータス` は `進行中` のまま維持

### voice-samples.md 自動更新

### 再要求時のルール（記憶ベース再生成禁止）

セッション内・次セッション問わず、以下の再要求が来たら**必ず `today/delivery_{日付}_{トピック要約}.md` を Read してから応答する**。記憶だけで再生成しない。コンテキスト圧縮後は特に厳守。

- サムネプロンプトを再表示してほしい
- 投稿セット全体を再表示してほしい
- タグを確認したい
- 貼り付けるURLを確認したい
- 投稿推奨タイミングを確認したい
- Phase 6スコアを確認したい

対象 delivery ファイルが見つからない場合は、「該当ファイルが見当たりません。ファイルパスを確認するか、再生成が必要なら /note-run を再実行してください」と返す。記憶から適当に埋めない。
