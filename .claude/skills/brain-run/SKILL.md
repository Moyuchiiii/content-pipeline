# /brain-run

note で書いた記事を Brain 向けに拡張リライトし、販売ドラフトを自動生成するスキル。
note は「実体験の記録・入門・ダイジェスト（無料 or ¥500）」、Brain は「実践ノウハウ・コード全文・テンプレ付きの完成版（¥1,980〜）」という役割分担で運用する。

**投稿は手動**。Brain は公開 API なし・reCAPTCHA ありで自動投稿リスク高。ドラフト作成までが自動の現実的ライン。

## Phase 0: コンテキスト読み込み + 対象記事判定

### ① コンテキスト読み込み

**Brain 特化コンテキスト（最優先）:**
- `context/brain-profile.md` — Brain プロフィール（3パターン・文体ガイド・カテゴリ・タグ戦略）
- `context/brain-strategy.md` — Brain 運用戦略・商品ラインナップ計画・価格戦略・審査対策

**共通コンテキスト:**
- `context/note-profile.md` — プロフィール・ポジショニング
- `context/note-strategy.md` — 収益化戦略
- `context/voice-samples.md` — 文体サンプル
- `context/writing-rules.md` — 証拠優先ルール・AI禁止フレーズ
- `context/article-frameworks.md` — AIDA・Star Story Solution
- `context/existing-articles.md` — 過去記事一覧
- `context/content-memory.md` — 成功パターン（あれば）

**参照レポート（相場・競合分析の裏取り用）:**
- `D:/Claude/bussines/search/brain-competitors-2026-04-20.md` — 競合 10 人プロフィール・商品ラインナップ・サムネテンプレ
- `D:/Claude/bussines/search/brain-article-structure-2026-04-21.md` — Brain 売れ筋 8 商品の本文構成パターン・目次の作り方・無料/有料ゾーン設計・15 項目チェックリスト（**Phase 2 で必ず参照**）

### ② 対象 note 記事の特定

以下の優先順で対象記事を決定する：

**ルート A（自動）: 引数なし**
1. `note/today/` 配下に HTML ファイルがあればそれを対象
2. なければ `note/drafts/{sokuho,jituroku,knowhow}/` から最新更新日のファイルを選択
3. 見つからない場合はエラー終了（「Brain化できる note 記事が見つかりません。先に /note-run を実行してください」）

**ルート B（手動）: 引数で指定**
```
/brain-run {記事パス}
/brain-run latest         # 最新ドラフト（ルート A と同じ）
/brain-run url {note_url} # 既に投稿済み note 記事を URL で指定
```

### ③ 対象記事タイプと Brain 化可否判定

| 記事タイプ | Brain 化 | 拡張方針 |
|---|---|---|
| 実録（jituroku） | ◎ 最適 | コード全文・プロンプト全文・受注テンプレ・単価交渉スクリプトを追加 |
| ノウハウ（knowhow） | ◎ 最適 | テンプレ・チェックリスト・設定ファイル全文を追加 |
| 速報（sokuho） | △ 要検討 | 速報は鮮度が命で Brain の長期販売と相性悪い。対象外推奨 |

**速報記事を対象にしようとした場合:**
「速報記事は Brain 販売に向きません（鮮度依存のため）。それでも続けますか？」とユーザーに確認。

### ④ 同一記事の重複出品チェック

Notion「note記事管理」DB で対象記事の `Brainステータス` を確認:
- `投稿済み` → 「すでに Brain 出品済みです。リライト再生成しますか？」と確認
- `草稿完成` → 「すでに Brain 草稿があります。上書きしますか？」と確認
- `未作成` または該当フィールドなし → そのまま次 Phase へ

## Phase 2: Brain 用拡張リライト

note 版との差別化のため、「買う価値」を明確にするコンテンツを追加する。

### 絶対ルール

- **売れる構成 絶対ルール遵守（後述 A〜F）**: 見出し具体化（Brain 自動目次前提）・冒頭 3 点セット（ペイン→実績→ベネフィット）・無料ゾーン末尾フック・`<ul>/<ol>/<table>` 禁止（「・ 項目」段落と blockquote 連続で代替）の 4 大要素は Phase 5 の構成適合性スウィープで必ず検証される。違反したら自動 rewrite
- **note 版の完全コピー禁止**: 最低 30% 以上の追加・再構成を行う
- **個人体験談として明記**: タイトル・冒頭に「文系大学生が」「自分の場合」等を入れる
- **誇大表現禁止**: 後述のブラックリスト参照
- **実績証拠を盛り込む**: スクショ配置指示・具体的な金額・件数を残す
- **匿名化**: クライアント名・個人特定情報は伏せる

### 追加すべきコンテンツ（記事タイプ別）

**実録（jituroku）拡張:**

- note 版の導入・結果は流用
- 以下を追加:
  - クライアントとのやり取り全文（匿名化・Before/After 付き）
  - 単価交渉時のメッセージテンプレ
  - Claude Code に投げた実際のプロンプト全文
  - 生成されたコード全文（動作確認済み版）
  - 納品チェックリスト
  - トラブルシュート集（自分が実際に踏んだエラーと解決手順）
  - 受注〜納品のスケジュール表（時間配分の数字）
  - この手法を他ジャンルに応用する方法

**ノウハウ（knowhow）拡張:**

- note 版の概要は流用
- 以下を追加:
  - 設定ファイル全文（`.claude/settings.json` 等）
  - プロンプトテンプレ集（コピペで使える状態）
  - カスタマイズ方法（初心者向け・中級者向け）
  - よくある失敗パターンと回避策
  - 自分の運用ログ（数週間分のデータ）
  - 応用展開のアイデア 3〜5 個

### Brain エディタ対応フォーマット（2026-04-21 実機確認済み）

出力は **HTML 形式と Markdown 形式の両方** を保存する。**Brain エディタへの投稿時は HTML 版をそのままペーストする**（note と同じフローで装飾が自動反映される。Markdown記法のペーストでは装飾変換されない）。Markdown 版はバックアップ・編集・後続ツール連携用。

#### 使用可能な HTML タグ

- `<h2>` `<h3>` `<h4>` — 見出し（`<h1>` はタイトルフィールドに使うため本文では使わない）
- `<p>` — 段落
- `<b>` — 強調
- `<blockquote>` — 引用（ラベル付きは `<b>ラベル</b><br>本文` 形式・コロン禁止）
- `<pre><code>` — コードブロック
- `<hr>` — 区切り線
- `<a href="...">` — リンク
- `<code>` — インラインコード

#### 使用禁止 HTML タグ（Brain で崩れる・実機確認済み）

- `<ul><li>` / `<ol><li>` — **箇条書き・番号付きリスト禁止**（表示が崩れる）
- `<table><tr><td>` — **表禁止**（特に「同梱物一覧」「カリキュラム表」「比較表」等の構造化情報で致命的に崩れる）

#### リスト・テーブルの代替形式（YOU MUST FOLLOW）

Brain で崩れない以下の形式で書く。リスト構造や表構造の情報は全てこの 3 パターンに変換する。

**1. 短い箇条書き（ペイン・ベネフィット・おすすめ等）:**
```html
<p>・ 項目 1 の内容</p>
<p>・ 項目 2 の内容</p>
<p>・ 項目 3 の内容</p>
```

**2. 番号付き手順（チェックリスト・ステップバイステップ）:**
```html
<p><b>①</b> 手順 1 の内容</p>
<p><b>②</b> 手順 2 の内容</p>
<p><b>③</b> 手順 3 の内容</p>
```

**3. 構造化情報（同梱物リスト・カリキュラム表・比較表）:**
```html
<blockquote>
<b>ラベル 1（数量や種類）</b><br>
説明文 / 補足情報 / 属性（スラッシュや読点で区切る・コロン禁止）
</blockquote>

<blockquote>
<b>ラベル 2（数量や種類）</b><br>
説明文 / 補足情報 / 属性
</blockquote>
```

blockquote は Brain で「引用枠」として表示され、ラベル+本文の 2 行構造を保てる。テーブルの代替として最も実用的。

#### 使用可能な Markdown 記法（バックアップ用・Brain に直接は貼らない）

- `# ` `## ` `### ` — 見出し
- `**太字**` — 強調
- `> ` — 引用
- `` `コード` `` — インラインコード
- ` ``` ` — コードブロック（言語指定付き）
- `---` — 区切り線
- `[リンク](URL)` — ハイパーリンク
- **Markdown の `- ` / `1. ` リストと `| ... |` 表は使わない**（HTML 出力時に `<ul>/<ol>/<table>` に変換されて崩れるため。代わりに上記の代替形式を使う）

### 売れる構成 絶対ルール（YOU MUST FOLLOW）

Brain 売れ筋 8 商品（しゅうへい / リツト / こうめい / まな / トシノ×2 / メラメラくん / ユニコ）の本文構成リサーチ（`search/brain-article-structure-2026-04-21.md`）から抽出した必須ルール。違反したら Phase 5 の品質ゲートで即リジェクト扱い。

#### A. Brain サイドバー自動目次を活かす見出し設計

- Brain の商品ページは **H2 見出しから自動的にサイドバー目次を生成する**。本文内に別途「目次」節を置く必要はない（置くと二重になる）
- 重要なのは **H2 見出しを具体的なタイトルにすること**。自動目次の質を決めるのは見出しの言葉選びだけ
- H2 見出しに章番号（「1.」「2.」）を付けない。Brain が自動で番号を振るため二重化する
- 売れてる 8 商品は全て H2 で構造を示しつつ、具体的な見出しを使っている。「第 1 章: 基礎」ではなく「なぜ○○は 9 割失敗するのか」等の具体タイトル

#### B. 冒頭は 3 点セット（順序固定）

全ての記事は冒頭 500〜1,500 文字以内で以下を順番で出し切る：

1. **ペインポイント提示（3〜6 項目の箇条書き）** — 「こんな悩みありませんか？」形式
2. **著者の実績数字（部数・売上・期間・フォロワー）** — 「文系大学生が 2 ヶ月で月 14 万」等、具体数字
3. **この記事で得られるベネフィット（3〜8 項目）** — 購入後の未来を明示

目次は Brain 自動生成に任せる（本文内に置かない）。

**NG 順序（過去の失敗事例）**: 「この記事で手に入るもの（ベネフィットリスト）」から始めて、ペインも実績数字も出さない順序。これは売れない。

#### C. 見出しタイトルは具体性勝負

- ✅ 良い型: 「なぜ日本語 LP は 9 割崩れるのか」「9 割が挫折する本当の理由」「Day1: 環境構築で最初の感動体験」「【証拠】これが "創造神" の力」
- ❌ 避ける型: 「第 1 章: 基礎」「2. ツールの使い方」「3. 実践」「まとめ」

章番号 + 意味ある章タイトル、または Day 別（Day1〜Day7）、章-サブ章（1-1, 1-2）の階層を使う。「第○部」という部構成は調査 8 商品中ゼロ採用なので不要。

#### D. 章数は 6〜18 章

少なすぎる（5 章以下）と「薄い」と判断される。多すぎる（20 章以上）と読めない。教科書型は Day 別で 7 日カリキュラム等、実録型は章 + Bonus 4 本、ツールキット型は 15 項目 + スタイル 30 種のような同梱物リストで厚みを出す。

#### E. 無料ゾーン末尾のフック（6 型から必ず 1 つ採用）

有料ゾーン直前の「ここから先は ¥○,○○○」の直前に、必ず以下のどれかを置く：

1. **やること一覧見せ切り型**（教科書型向き）: 有料ゾーンの章タイトルを箇条書きで列挙し、「2 名に紹介すれば元が取れる計算」等の価格正当化を追加。しゅうへいの型
2. **ストーリークロージング型**（ブランド訴求型）: 「今日から、あなたは一人ではありません」等の情緒的クロージング。リツトの型
3. **第 1 章の冒頭 1 文型**（実録型向き）: 有料ゾーンの最初の 1 文だけを公開して「続きが気になる」状態を作る。こうめいの型
4. **共感クロージング型**（ストーリー型）: 「そんな時期がありました。でも今は〜」と著者の変化を再度提示。まなの型
5. **想い × 直接 CTA 型**（高単価向き）: 開発コスト・時間の提示で価値正当化 → 「今すぐ購入ボタンを押してください」の直接指示。ユニコの型
6. **今がタイミング宣言型**: 「今がその一歩を踏み出すタイミングです」等の決断促進。メラメラくんの型

フックなしで「ここから先は ¥○,○○○」に直行は NG。

#### F. 無料/有料ゾーンの分量比率

| タイプ | 無料ゾーン | 有料ゾーン |
|---|---|---|
| 教科書型（しゅうへい・リツト） | 全体の 20〜70%（実ノウハウの第 1〜2 章まで） | 残り |
| 実録型（こうめい・まな） | 全体の 50〜90%（LP 部分 + 第 1 章の冒頭 1 文） | 残り |
| ツールキット型（ユニコ） | 全体の 85%（LP 部分のみ） | 残り |

### 記事タイプ判定ロジック

対象記事のタイトル・元ネタ・同梱物から以下のルールで 3 タイプから 1 つ選ぶ：

| 判定条件 | 採用テンプレ |
|---|---|
| タイトルに「教科書」「完全ガイド」「バイブル」「Day別」を含む、または Day 別カリキュラム構成 | **タイプ A: 教科書型** |
| 記事タイプが `jituroku`、または「全記録」「実録」「○ヶ月やってみた」「ケーススタディ」型 | **タイプ B: 実録型** |
| 大量同梱物あり（テンプレ 20 種以上・プロンプト集・GPTs 同梱）、または価格 ¥9,980 以上 | **タイプ C: ツールキット型** |

複数条件該当の場合は、優先度 C > A > B で選ぶ（高単価の訴求を優先）。

## Phase 3: タイトル・商品説明文の生成

Brain の商品ページでは「タイトル」「サブタイトル」「商品説明文（購入前に見える冒頭部分）」が売上を決める。

### Step 1: タイトル A/B テスト

5 本生成してスコアリング。既存 Brain 売れ筋タイトルの型に寄せる。

**Brain で効くタイトル型:**
- **数字実績型**: 「文系大学生が2ヶ月で月14万達成した{具体手段}」
- **完全公開型**: 「{具体タスク}の全手順・プロンプト・コード完全公開」
- **Before/After 型**: 「{困りごと}の状態から{結果}になるまで」
- **具体ツール型**: 「Claude Code × {他ツール} で {結果} を出す方法」

**タイトルスコアリング（各 20 点満点）:**

| 項目 | 判定基準 |
|---|---|
| 具体性 | 数字・固有名詞が入っているか（0-5点） |
| 結果明示 | 「何が手に入るか」が一目でわかるか（0-5点） |
| 誇大表現チェック | 後述ブラックリスト違反がないか（0-5点） |
| 検索キーワード | Brain 内検索で出てきそうなワードか（0-5点） |

**誇大表現ブラックリスト（即リジェクト）:**
- 「誰でも」「絶対」「必ず」「確実に」
- 「再現性 100%」「世界初」
- 「1日○分で月○万」（時間×金額の断定）
- 「知らないと損」「絶対知るべき」
- 「月○万円保証」「収入保証」

### Step 2: サブタイトル生成

- 30〜60 字
- タイトルで言い切れなかった「誰向け」「何が含まれるか」を補足
- 例: 「非エンジニアでも今日から動かせるテンプレ・コード全部入り」

### Step 3: 商品説明文生成

Brain の購入前表示部分（冒頭 200〜400 字）で以下の要素を順番に書く:

1. **フック** — 読者の痛みに触れる 1〜2 文
2. **実績の数字** — 「文系大学生が 2 ヶ月で月 14 万達成」等（個人体験として明記）
3. **この記事を買うと手に入るもの** — 箇条書き 4〜6 項目
4. **対象読者** — 1〜2 文（「こんな人向け」）
5. **注意書き** — 「本記事は個人の体験談であり、同様の成果を保証するものではありません」を必須挿入

### タイトル・説明文の出力フォーマット

## Phase 4: 価格・紹介還元率の提案

### 価格の決め方

Phase 1 の相場調査と以下の条件を組み合わせる:

| 条件 | 推奨価格 |
|---|---|
| Brain 販売実績ゼロ（初投稿） | ¥1,980 |
| 販売実績 1〜5 件 | ¥1,980〜¥2,980 |
| 販売実績 6 件以上 + スキ 10 以上 | ¥2,980〜¥4,980 |
| コード全文＋テンプレ＋設定ファイル付き | +¥1,000 上乗せ可 |
| 速報性がある（リリース 48 時間以内） | +¥500 上乗せ可（鮮度プレミア） |

### 紹介還元率の決め方

Brain の強みはアフィリエイト機能。還元率が高いほどアフィリエイターが拡散しやすい。

| フェーズ | 推奨還元率 |
|---|---|
| 販売初動（実績ゼロ〜5 件） | **50%**（拡散重視） |
| 安定期（6〜20 件） | 40% |
| 安定期（21 件〜） | 30%（利益重視） |

**出力フォーマット:**

```
💰 価格・還元率の提案

価格: ¥{価格}
理由: {根拠3行}

紹介還元率: {%}
理由: {根拠1行}

取り分シミュレーション（月5部販売の場合）
- 自己経由売上: ¥{価格} × 5 × 88% = ¥{金額}
- 紹介経由売上: ¥{価格} × 5 × (100% - {紹介率}% - 12%) = ¥{金額}
- 平均想定: ¥{中央値}/月
```

## Phase 5: 品質ゲート

### Step 1: 誇大表現スウィープ（必須）

記事全文を以下のブラックリストで grep し、該当があれば必ず修正する:

**即リジェクトワード:**
- 「誰でも」「必ず」「絶対」「確実に」
- 「再現性 100%」「世界初」「最強」「最速」
- 「保証」「収入保証」
- 「1日{数字}分で月{数字}万」系
- 「知らないと損」「知らないと危険」

**要注意ワード（文脈で判断）:**
- 「簡単に」「すぐに」「誰でもできる」 → 「個人的には〜」「自分の場合は〜」に言い換え
- 「稼げる」 → 「稼いだ（個人の体験）」に言い換え

**修正後は必ず「個人の体験談」タグが入っているか再確認。**

### Step 2: 構成適合性スウィープ（必須・Phase 2 の絶対ルール違反の検出）

Phase 2 の売れる構成 絶対ルール（A〜F）を記事に対して自動検証する。16 項目のチェックリストを適用し、欠落を rewrite で埋める。

**【見出し・構造】**
1. [ ] H2 見出しが具体的（Brain 自動目次の質を決める。「基礎」「実践」でなく「なぜ○○は 9 割失敗するのか」「Day1: 環境構築で最初の感動体験」）
2. [ ] H2 見出しに章番号（「1.」「2.」）を付けていない（Brain 自動目次と重複するため二重化する）
3. [ ] `<ul>/<ol>/<table>` を使っていない（Brain で崩れる。「・ 項目」段落、「① 項目」番号付き段落、blockquote ラベル付き連続で代替）
4. [ ] 章数は 6〜18 章
5. [ ] 教科書型なら Day 別 or Chapter 別のカリキュラム（blockquote 連続で提示）がある

**【冒頭フック・ペイン提示】**
5. [ ] 冒頭 300〜500 文字以内でペインポイント 3〜6 項目を列挙
6. [ ] 著者の実績数字（部数・売上・期間）が冒頭 1,000 文字以内に出ている
7. [ ] ベネフィット 3〜8 項目を提示している

**【無料ゾーン設計】**
8. [ ] 無料ゾーン末尾に 6 型のどれかのフックがある（やること一覧 / ストーリークロージング / 第 1 章冒頭 1 文 / 共感クロージング / 想い × 直接 CTA / 今がタイミング宣言）
9. [ ] 有料ゾーン直前に価格の正当化フレーズがある（「動画 1 本あたり○円」「開発に○時間」等）
10. [ ] 購入ボタンのすぐ上に直接的な行動 CTA がある

**【社会的証明】**
11. [ ] 購入者の声が本文内（目次や冒頭節）に組み込まれている（発売前は「モニター感想」または将来追記枠を置く）
12. [ ] 販売実績（部数・売上）が本文内に明示されている（発売前は「発売後追記」と明記）
13. [ ] [UPDATE] ログを本文冒頭に用意（発売後、販売数・追加コンテンツの更新を随時追記）

**【コンテンツ価値の見せ方】**
14. [ ] 同梱物（テンプレ・動画・特典）の一覧が具体的な数量・内容付きで示されている
15. [ ] 「やらないこと」「おすすめしない人」リストがある

**判定:**
- 14 項目以上 OK → 公開可
- 11〜13 項目 OK → 軽微修正で公開可（欠落項目のみピンポイント rewrite）
- 10 項目以下 OK → 自動 rewrite 実行（Phase 2 の該当タイプテンプレ A/B/C を参照しながら全体を再構成。最大 2 回、2 回後も基準未達ならユーザー報告）

**スウィープ実行方法:**
記事全文を読み、各項目を自動判定してチェックマークを付ける。欠落項目がある場合はその項目に対応するテンプレ（タイプ A/B/C）の該当セクションを参照して記事に追記・再配置する。特に「ベネフィットから始まっている（ペインなし）」「H2 に章番号が付いている」「`<ul>/<ol>/<table>` を使っている」「フックなし」の 4 大 NG は最優先で修正する。

### Step 3: 規約チェックリスト

Brain 規約（/terms は 404 のためガイドラインベースで確認）に基づく以下を必ずチェック:

- [ ] 誇大表現がない
- [ ] 個人体験である旨が冒頭・末尾に明記されている
- [ ] 他サービス（note・X等）への誘導リンクが「販売誘導」になっていない（著者プロフィール欄の参照のみ OK）
- [ ] 根拠のない統計・データがない
- [ ] 医療・投資助言系の記述がない
- [ ] 「効果には個人差があります」の免責文がある

### Step 4: 120 点ルーブリック採点

**コンテンツ価値（40 点満点）:**
- 「この記事を買う理由」が明確か（0-10）
- コード・プロンプト・テンプレの実用性（0-10）

- note 版との差分が明確か（30% 以上の追加情報）（0-10）

**Brain 最適化（30 点満点）:**
- タイトルの相場適合性（0-10）
- 商品説明文のフック力（0-10）
- 価格の妥当性（0-10）

**構成適合性（20 点満点）:**

- H2 見出しの具体性・章タイトルの型（Why 型 / Day 型 / 問いかけ型等・章番号なし）（0-5）

**リスク（20 点満点）:**

- 匿名化・免責文（0-10）

**voice 一貫性（10 点満点）:**

**判定:**

- 80 点未満 → 自動 rewrite（最大 2 回）、2 回でも 80 未満ならユーザー報告

**採点結果の表示:**

## Phase 6: 保存・出力

### Step 1: 前回の `brain/today/` を正規フォルダに移動

`brain/today/` フォルダの中身を確認し、ファイルがあれば正規フォルダへ移動:
- `brain/today/*.md` → `brain/drafts/{記事タイプ}/`
- `brain/today/*.html` → `brain/drafts/{記事タイプ}/`

（`brain/today/` が空ならスキップ）

### Step 2: 今回のファイルを今日用と正規フォルダの両方に保存（HTML + Markdown）

**本文ファイル（HTML版 — Brain エディタへの投稿はこれを貼る）:**
1. `brain/drafts/{記事タイプ}/brain_{日付}_{トピック要約}.html` — バックアップ
2. `brain/today/brain_{日付}_{トピック要約}.html` — 投稿用（Brain エディタにペースト・装飾が自動反映）

**本文ファイル（Markdown版 — バックアップ・編集・後続ツール連携用）:**
3. `brain/drafts/{記事タイプ}/brain_{日付}_{トピック要約}.md` — バックアップ
4. `brain/today/brain_{日付}_{トピック要約}.md` — すぐ開ける場所

**メタファイル（タイトル・説明文・価格・タグ・還元率を 1 つにまとめる）:**
5. `brain/drafts/{記事タイプ}/brain_{日付}_{トピック要約}_meta.md` — バックアップ
6. `brain/today/brain_{日付}_{トピック要約}_meta.md` — 投稿用

**HTML版と Markdown版は同じ内容で書式だけ違う**（HTML 版は note と同じタグで組む）。片方だけ編集して乖離するのを避けるため、1回で両方生成する。

### サムネイル生成プロンプト（Brain 特化版・2026-04-20 実地競合分析ベース）

#### Brain サムネの 3 大パターン（把握必須）

- **パターン A 情報商材ゴールド型（40%）**: 漆黒 + 金ピカグラデ + 「完全攻略」「教科書」+ 特典バッジ（リツト・ベク等）

- **パターン C テキスト密度型（25%）**: 濃紺/黒 + 白テキスト多段 + ツール名大 + チェックマーク（n8n 攻略バイブル等）
- **パターン D（例外）ライト系クリーン型（5%）**: オフホワイト + ツールロゴ + 顔写真丸型（しゅうへいの Claude 教科書のみ）

#### 優太が狙うポジション

**「パターン C テキスト密度型ベース + 情報商材ギラギラを抑えたクリーンダーク路線」**

- ダーク背景は採用（Brain 相場の 90% がダーク。白系にするとカテゴリ内で浮く）
- 数字訴求は採用（90% が何らかの数字を配置。「月 14 万」は優太の核心差別化）
- 金ピカグラデ・月桂樹バッジ・宇宙星雲・虹色ネオン・アニメキャラは**全て避ける**（コモディティ化 or 優太ブランドと矛盾）
- 代わりに「クリーンなダーク + オレンジ/ゴールド最小限アクセント + 実直なレイアウト」で「文系大学生・実直・若さ」を表現

#### note との違い（展開時の混同防止）

| 比較軸 | note サムネ | Brain サムネ（優太版） |

| 背景色 | オフホワイト/ライトグレー | 漆黒/濃紺 |

## Phase 8: PDCA フィードバック（投稿後）

投稿後にユーザーに促す記録項目:

**投稿直後:**
- Brain URL
- 投稿日
- 最終的に設定した価格・還元率

**1 週間後:**
- 販売数
- Brain 内 PV
- レビュー・コメント

**1 ヶ月後:**
- 累計販売数
- 売上
- アフィリエイター経由の割合
- 学び（何が効いた・効かなかった）

これらは Notion の対象レコードに追記する。次回 /brain-run 実行時に Phase 1 の相場調査と併せて参照される（「前回 X 件売れたから今回は価格を上げよう」等の判断に使う）。

## エラーハンドリング

**対象 note 記事が見つからない:**
```
❌ Brain 化できる note 記事が見つかりません。

対処:
1. /note-run を実行して note 記事を作成する
2. または /brain-run {記事パス} で明示的に指定する
```

**記事タイプが速報の場合:**
```
⚠️ この記事は速報タイプです。
速報は鮮度依存のため Brain の長期販売と相性が悪い傾向があります。
それでも Brain 化しますか？ (y/n)
```

**Phase 5 の品質スコアが 60 点未満（2 回 rewrite 後も）:**
```
⚠️ 品質スコアが基準に達していません。
問題箇所: {スウィープで検出された項目}
ユーザーの判断を仰ぎます。
- そのまま保存して人手で修正
- 再度 rewrite を試行
- 対象記事を変えて再実行
```

Brain 投稿完了後、ユーザーが Notion レコード更新 + Phase 8 の PDCA 記録を行う。
