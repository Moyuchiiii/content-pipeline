# /brain-run

note で書いた記事を Brain 向けに拡張リライトし、販売ドラフトを自動生成するスキル。
note は「実体験の記録・入門・ダイジェスト（無料 or ¥500）」、Brain は「実践ノウハウ・コード全文・テンプレ付きの完成版（¥1,980〜）」という役割分担で運用する。

**投稿は手動**。Brain は公開 API なし・reCAPTCHA ありで自動投稿リスク高。ドラフト作成までが自動の現実的ライン。

### ② 対象 note 記事の特定

以下の優先順で対象記事を決定する：

**ルート A（自動）: 引数なし**
1. `note/today/` 配下に HTML ファイルがあればそれを対象

**ルート C（新・2026-04-24 追加）: Brainネタ帳から取得**

/brain-run from-idea {ネタ名}    # Brainネタ帳の指定ネタを使用

`mcp__notion__notion-fetch` で `collection://1198a5a1-6955-41da-928b-ad79f07d2a1b` を参照:
- ステータス=`未使用`
- target_fit=`high` または `medium`

## Phase 2: Brain 用拡張リライト

note 版との差別化のため、「買う価値」を明確にするコンテンツを追加する。

### 絶対ルール

- **売れる構成 絶対ルール遵守（後述 A〜F）**: 見出し具体化（Brain 自動目次前提）・冒頭 3 点セット（ペイン→実績→ベネフィット）・無料ゾーン末尾フック・`<ul>/<ol>/<table>` 禁止（「・ 項目」段落と blockquote 連続で代替）の 4 大要素は Phase 5 の構成適合性スウィープで必ず検証される。違反したら自動 rewrite
- **note 版の完全コピー禁止**: 最低 30% 以上の追加・再構成を行う
- **個人体験談として明記**: タイトル・冒頭に「文系大学生が」「自分の場合」等を入れる
- **誇大表現禁止**: 後述のブラックリスト参照
- **実績証拠を盛り込む**: スクショ配置指示・具体的な金額・件数を残す
- **匿名化**: クライアント名・個人特定情報は伏せる

### 追加すべきコンテンツ（記事タイプ別）

**実録（jituroku）拡張:**

- note 版の導入・結果は流用
- 以下を追加:

  - 受注〜納品のスケジュール表（時間配分の数字）
  - この手法を他ジャンルに応用する方法

**ノウハウ（knowhow）拡張:**

- note 版の概要は流用
- 以下を追加:
  - 設定ファイル全文（`.claude/settings.json` 等）

  - 自分の運用ログ（数週間分のデータ）
  - 応用展開のアイデア 3〜5 個

### Brain エディタ対応フォーマット（2026-04-21 実機確認済み）

出力は **HTML 形式と Markdown 形式の両方** を保存する。**Brain エディタへの投稿時は HTML 版をそのままペーストする**（note と同じフローで装飾が自動反映される。Markdown記法のペーストでは装飾変換されない）。Markdown 版はバックアップ・編集・後続ツール連携用。

#### 使用可能な HTML タグ

- `<h2>` `<h3>` `<h4>` — 見出し（`<h1>` はタイトルフィールドに使うため本文では使わない）
- `<p>` — 段落
- `<b>` — 強調
- `<blockquote>` — 引用（ラベル付きは `<b>ラベル</b><br>本文` 形式・コロン禁止）
- `<pre><code>` — コードブロック
- `<hr>` — 区切り線
- `<a href="...">` — リンク
- `<code>` — インラインコード

#### 使用禁止 HTML タグ（Brain で崩れる・実機確認済み）

- `<ul><li>` / `<ol><li>` — **箇条書き・番号付きリスト禁止**（表示が崩れる）
- `<table><tr><td>` — **表禁止**（特に「同梱物一覧」「カリキュラム表」「比較表」等の構造化情報で致命的に崩れる）

#### リスト・テーブルの代替形式（YOU MUST FOLLOW）

Brain で崩れない以下の形式で書く。リスト構造や表構造の情報は全てこの 3 パターンに変換する。

**1. 短い箇条書き（ペイン・ベネフィット・おすすめ等）:**
```html
<p>・ 項目 1 の内容</p>
<p>・ 項目 2 の内容</p>
<p>・ 項目 3 の内容</p>
```

**2. 番号付き手順（チェックリスト・ステップバイステップ）:**
```html
<p><b>①</b> 手順 1 の内容</p>
<p><b>②</b> 手順 2 の内容</p>
<p><b>③</b> 手順 3 の内容</p>
```

**3. 構造化情報（同梱物リスト・カリキュラム表・比較表）:**
```html
<blockquote>
<b>ラベル 1（数量や種類）</b><br>
説明文 / 補足情報 / 属性（スラッシュや読点で区切る・コロン禁止）
</blockquote>

<blockquote>
<b>ラベル 2（数量や種類）</b><br>
説明文 / 補足情報 / 属性
</blockquote>
```

blockquote は Brain で「引用枠」として表示され、ラベル+本文の 2 行構造を保てる。テーブルの代替として最も実用的。

#### 使用可能な Markdown 記法（バックアップ用・Brain に直接は貼らない）

- `# ` `## ` `### ` — 見出し
- `**太字**` — 強調
- `> ` — 引用
- `` `コード` `` — インラインコード
- ` ``` ` — コードブロック（言語指定付き）
- `---` — 区切り線
- `[リンク](URL)` — ハイパーリンク
- **Markdown の `- ` / `1. ` リストと `| ... |` 表は使わない**（HTML 出力時に `<ul>/<ol>/<table>` に変換されて崩れるため。代わりに上記の代替形式を使う）

### 売れる構成 絶対ルール（YOU MUST FOLLOW）

Brain 売れ筋 8 商品（しゅうへい / リツト / こうめい / まな / トシノ×2 / メラメラくん / ユニコ）の本文構成リサーチ（`search/brain-article-structure-2026-04-21.md`）から抽出した必須ルール。違反したら Phase 5 の品質ゲートで即リジェクト扱い。

#### A. Brain サイドバー自動目次を活かす見出し設計

- Brain の商品ページは **H2 見出しから自動的にサイドバー目次を生成する**。本文内に別途「目次」節を置く必要はない（置くと二重になる）
- 重要なのは **H2 見出しを具体的なタイトルにすること**。自動目次の質を決めるのは見出しの言葉選びだけ
- **H2 / H3 共に章番号・識別子を一切付けない**。Brain が H2 / H3 それぞれに自動で番号を振るため、本文側で番号を書くと「11-1. Q1. 本当に文系・...」のような二重化が発生する。以下すべて禁止:
  - H2 の「第 1 章」「第 N 章」「Chapter 1」「1.」「2.」等（例: `## 第 1 章 プロフィール改修` → `## プロフィール改修`）
  - H3 の階層番号「1-1.」「1-2.」「2-1.」「B1-1.」「B2-3.」等（例: `### 1-1. ジャンル選定の失敗談` → `### ジャンル選定の失敗談`）
  - FAQ の「Q1.」「Q2.」等の質問番号と、回答行頭の「A.」（例: `### Q1. 文系でも使える？` → `### 文系でも使える？`、`A. 使えます。理由は...` → `使えます。理由は...`）
  - Bonus 章の「Bonus 1」「Bonus 2」や小見出しの「B1-1.」「B2-1.」等（例: `## Bonus 1 他ジャンル応用法` → `## 他ジャンル応用法`、`### B1-1. ライター案件で使う場合` → `### ライター案件で使う場合`）
  - ただし機能的識別子は例外として許可: 「Day1」「Day2」〜「Day N」（教科書型カリキュラムの順序ラベル）、「Week 1-2」「Week 3-4」等の期間表記（章タイトルの一部として残してよい）
- 売れてる 8 商品は全て H2 で構造を示しつつ、具体的な見出しを使っている。「第 1 章: 基礎」ではなく「なぜ○○は 9 割失敗するのか」等の具体タイトル

#### B. 冒頭は 4 点セット（順序固定）

全ての記事は冒頭 500〜1,800 文字以内で以下を順番で出し切る：

1. **冒頭 blockquote 宣言（2 段落必須）** — 「誰に・何を・どうなる」の要約（詳細はルール M 参照）
2. **ペインポイント提示（3〜6 項目の箇条書き）** — 「こんな悩みありませんか？」形式
3. **著者の実績（抽象化表現）** — 「Claude Code × クラウドワークスで提案資料・業務自動化系の案件を継続受注」のような抽象化実績（ルール K の審査回避規定に従い、**具体金額と期間の組み合わせを避ける**）
4. **この記事で得られるベネフィット（3〜8 項目）** — 購入後の未来を明示

目次は Brain 自動生成に任せる（本文内に置かない）。

**NG 順序（過去の失敗事例）**: 「この記事で手に入るもの（ベネフィットリスト）」から始めて、ペインも実績も出さない順序。これは売れない。

#### C. 見出しタイトルは具体性勝負

- ✅ 良い型: 「なぜ日本語 LP は 9 割崩れるのか」「9 割が挫折する本当の理由」「Day1: 環境構築で最初の感動体験」「【証拠】これが "創造神" の力」
- ❌ 避ける型: 「第 1 章: 基礎」「2. ツールの使い方」「3. 実践」「まとめ」

章番号 + 意味ある章タイトル、または Day 別（Day1〜Day7）、章-サブ章（1-1, 1-2）の階層を使う。「第○部」という部構成は調査 8 商品中ゼロ採用なので不要。

#### D. 章数は 6〜18 章

少なすぎる（5 章以下）と「薄い」と判断される。多すぎる（20 章以上）と読めない。教科書型は Day 別で 7 日カリキュラム等、実録型は章 + Bonus 4 本、ツールキット型は 15 項目 + スタイル 30 種のような同梱物リストで厚みを出す。

#### E. 無料ゾーン末尾のフック（6 型から必ず 1 つ採用）

有料ゾーン直前の「ここから先は ¥○,○○○」の直前に、必ず以下のどれかを置く：

1. **やること一覧見せ切り型**（教科書型向き）: 有料ゾーンの章タイトルを箇条書きで列挙し、「2 名に紹介すれば元が取れる計算」等の価格正当化を追加。しゅうへいの型
2. **ストーリークロージング型**（ブランド訴求型）: 「今日から、あなたは一人ではありません」等の情緒的クロージング。リツトの型
3. **第 1 章の冒頭 1 文型**（実録型向き）: 有料ゾーンの最初の 1 文だけを公開して「続きが気になる」状態を作る。こうめいの型
4. **共感クロージング型**（ストーリー型）: 「そんな時期がありました。でも今は〜」と著者の変化を再度提示。まなの型
5. **想い × 直接 CTA 型**（高単価向き）: 開発コスト・時間の提示で価値正当化 → 「今すぐ購入ボタンを押してください」の直接指示。ユニコの型
6. **今がタイミング宣言型**: 「今がその一歩を踏み出すタイミングです」等の決断促進。メラメラくんの型

フックなしで「ここから先は ¥○,○○○」に直行は NG。

#### F. 無料/有料ゾーンの分量比率

| タイプ | 無料ゾーン | 有料ゾーン |
|---|---|---|
| 教科書型（しゅうへい・リツト） | 全体の 20〜70%（実ノウハウの第 1〜2 章まで） | 残り |
| 実録型（こうめい・まな） | 全体の 50〜90%（LP 部分 + 第 1 章の冒頭 1 文） | 残り |
| ツールキット型（ユニコ） | 全体の 85%（LP 部分のみ） | 残り |

#### G. 段落細分化ルール（1段落1メッセージ）

- **1段落は60〜150字**。1段落に2〜3文まで。長い論理展開を1段落に詰め込まない
- **独立1行段落でリズムを作る**。印象的な一文・感情的な転換点・決定的な結論は、1段落1文で独立させる
  - 例: `<p>でも、眠れなかった。</p>` / `<p>それだけ。</p>` / `<p>5分で答えが返ってきた。</p>`
- 売れ筋の段落平均: 明鏡 150-250字、メアリ 23-100字。中間の60-150字がバランス良い
- **1段落400字以上は禁止**（読み飛ばされる・スマホで圧迫感）

#### H. 太字使用ルール（tiptap マーク継続バグ回避・絶対厳守）

Brain エディタは tiptap ベースで、**HTMLの段落中間に `<b>` / `<strong>` があると、段落全体を `<strong>` で包むバグ**がある（マーク継続バグ）。これを避けるため以下を厳守:

- **`<b>` / `<strong>` は段落全体を囲む形でのみ使用**: `<p><strong>全文</strong></p>` の形式のみ許可
- **段落中間・末尾の部分強調は禁止**: `<p>...続いて、<strong>重要な一文</strong>と続く</p>` はバグ発動で全段落太字化する
- 強調したい一文は**独立段落に切り出し**てから `<strong>` で包む（段落分割必須）
- 名詞句の太字（「情報じゃなくて視点の切り替え装置」等）は、独立段落化できないなら太字を諦める
- 太字の目安: **本文 1000 字あたり strong 7.0 以上**（明鏡 7.9/1000字 水準）。1 記事あたりの絶対数ではなく密度で判定する
  - 参考密度: メアリ 9.1 / 明鏡 7.9 / hulu35 2.0（薄い判定）
  - 教科書型 16k 字なら `strong` 120 個以上が目標
  - Phase 5 で段落生成後、自動付与ロジックで数字含む段落・結論フレーズ・決めフレーズに段落全体 strong を付与（詳細は Phase 5 参照）
- **確認コマンド**: `grep -nE '<p>[^<]{5,}<b>' {file}` で段落中間の `<b>` が検出されたら即修正（0件必須）

#### I. 話し言葉・読者への問いかけ（感情接点）

- 各H2セクションに**読者への問いかけを1回以上**: 「あなたも〜ですか？」「〜じゃないですか？」「〜したことありますか？」等
- 話し言葉を混ぜる: 「〜だった」「〜じゃないですか」「〜んです」「まあ」「正直」等
- AI感の出る定型フレーズ禁止: 「〜と言えるでしょう」「〜ではないでしょうか」「重要なのは」
- 1記事で「あなた」が10回以上出るのが目安

#### J. blockquote 使用制限

以下の5種類のみ blockquote で囲む。それ以外は地の文 or 独立段落で書く:

1. **プロンプト全文（単発・1個だけ使う場合のみ）** — ただし**プロンプト集のように連続させる場合（Day2 のプロンプト 20 本等）は blockquote ではなく `<pre><code>...</code></pre>` のコードブロック必須**。pre/code なら連続時の改行問題を回避でき、Brain でコピペボタンが付く
2. **テンプレ全文（単発）**（提案文・メッセージテンプレ） — 連続配置する場合は pre/code 化する
3. **Before/After 事例**（プロフィール改修の Before/After 等・連続 2 個まで）
4. **実際のクライアントメッセージ**（匿名化済み・連続 2 個まで）
5. **印象的な一言の単独強調**（「バイトの方が時給よくない？」のような独立セリフ・1記事に2〜3個まで）

**NG 使用**:
- **blockquote 連続 3 個以上は全面禁止**（2026-04-22 実機確認で、連続時に大きな改行マージンが挟まって読みにくいと確認。目次的な章リスト・同梱物一覧・カリキュラム表・用途リスト・案件タイプ別一覧等の構造化情報は全て `<p><b>タイトル</b></p>` + `<p>説明</p>` の段落ペアで代替）
- **プロンプト全文を連続 blockquote で列挙**（Day2 のプロンプト 20 本のような集合は `<p><b>プロンプト N タイトル</b></p>` + `<pre><code>プロンプト本文</code></pre>` の組み合わせで書く）
- 数値データの列挙（地の文化する）
- 箇条書きの代替（リスト禁止への対応として blockquote を乱用しない）

## Phase 3: タイトル・冒頭 blockquote の生成

Brain の商品ページには**「タイトル」欄のみ存在**（2026-04-22 実機確認）。購入前表示の訴求は**本文冒頭の blockquote（ルール M）**が担う。「サブタイトル」「商品説明文」欄は Brain に存在しないため、このフェーズで生成するのは以下の 2 つだけ。

### Step 1: タイトル A/B テスト

5 本生成してスコアリング。既存 Brain 売れ筋タイトルの型に寄せる。**ルール L のタイトル規約を必ず遵守**。

**Brain で効く（かつ審査通過する）タイトル型:**
- **実戦メモ型**: 「{ツール名} 実戦メモ — {対象} が {期間} で触った記録」
- **実録型**: 「{ツール名} 実録 — {実作業の具体内容}」
- **作業ログ型**: 「{ツール名} を {用途} で使った作業ログ」
- **Before/After 型**: 「{困りごと} の状態から {結果} になるまで」
- **具体ツール型**: 「{ツール名} × {他ツール} で {結果} を出した記録」

**タイトルスコアリング（各 20 点満点・合計 100 点）:**

| 項目 | 判定基準 |
|---|---|
| 具体性 | 固有名詞（ツール名）が入っているか（0-5点） |
| 結果明示 | 「何が書かれているか」が一目でわかるか（0-5点） |
| 審査規約適合 | ルール L のブラックリスト違反がないか（0-5点） |
| 検索キーワード | Brain 内検索で出てきそうなワードか（0-5点） |

**ルール L 違反タイトル（即リジェクト）:**
- `完全教科書` `完全攻略` `完全ガイド` `完全公開` `完全版` — 「完全」系
- `○○付き` `○大特典付き` `○○プレゼント` — 物量訴求
- `バイブル` `聖典` — 権威訴求
- `月○万円` `○ヶ月で○万` — 金額訴求（期間+金額の組み合わせ）
- `稼げる` `儲かる` `必ず` `確実に` `絶対` `保証` — 確定成果訴求
- `誰でも` `再現性 100%` — 絶対化
- `知らないと損` `絶対知るべき` — 脅し

## Phase 5: 品質ゲート

### Step 0: 事実確認スウィープ（全件Web検索・必須）

**IMPORTANT: 記憶・学習データで判断することを禁止する。以下の全項目を必ず WebSearch / WebFetch で実際に調べてから判定すること。**

Brain は商品としての信頼性が命。誇張・誤情報・動かないプロンプトは即レビュー最悪化＆返金要求＆アカウント信頼失墜に直結する。

#### Anthropic 公式ハルシネーション対策テクニック（2026-04-23 追加・7手法）

[Anthropic 公式ガイド](https://platform.claude.com/docs/en/test-and-evaluate/strengthen-guardrails/reduce-hallucinations) の推奨テクニックを本 Step 0 に全て適用:

1. **「わからない」の許可** — 公式ソースで確認できなかったクレームは無理に書かず、「未確認のため本文から削除」と明示して削除する。虚偽の断言よりも削除が優先
2. **直接引用の強制** — 長文ドキュメント（20k トークン超）を読む場合、先に該当箇所を抜粋引用してから判定する。抜粋なしの「私が読んだ記憶では」禁止
3. **引用検証** — 記事生成後、各クレームの裏付け引用を探して本文内に明示。見つからないクレームは撤回（削除 or `[要検証]` タグ付き）
4. **Chain-of-Thought 検証** — 最終判定前に「このクレームが正しいと言える根拠」を段階的に書き出してから判定（推論過程を明示）
5. **外部知識制限** — 提供されたソース（WebFetch で取得した公式ページ）のみを使用。一般知識・記憶ベースの追加は禁止
6. **Best-of-N 検証（重要クレーム限定）** — 「月 14 万」等の根幹数字は別セッション or fact-checker エージェントで複数回検証して矛盾がないことを確認
7. **Iterative Refinement** — 検証結果を受けて本文を修正し、修正後にもう一度 Step 0 を走らせる（複数ラウンドで矛盾露出）

#### 対象クレームの抽出

記事全文を読んで、以下の種別に該当する「事実の主張」をすべて列挙する：

| 種別 | 例 |
|---|---|
| A. リリース日・バージョン | 「Claude Design は 2026-04-17 リリース」「v2.1.89 で追加」 |
| B. 機能名・コマンド名 | 「Claude Code の `/voice` コマンド」「MCP Elicitation」 |
| C. 機能の仕様・動作 | 「Claude Design は Pro/Max/Team プラン限定」「月 20 ドル」 |
| D. コマンド構文・設定値・URL | 「`claude.ai/design`」「`claude remote-control`」 |
| E. 外部サービスの仕様 | 「CW の報酬形態・手数料率」「Brain の還元率上限 50%」 |
| F. 統計・データ（自身の体験以外） | 「AI 副業市場の平均単価」等の一般的な数値 |
| G. プロンプト・コードの動作 | 「このプロンプトで 30 秒でスライド5枚が出る」「この CSS で日本語 LP が直る」 |

**対象外（事実確認不要）**: 自分の体験・Before/After 実測値（「3時間→15分になった」「95件応募した」等）

#### 判定と修正

各クレームについて以下を判定し、記録する：

```
✅ 検証済み: [クレーム内容] → ソース: [URL] または [実行結果ログ]
⚠️ 要修正: [クレーム内容] → 実際は [正しい情報] → 修正済み
❌ 削除: [クレーム内容] → ソース見つからず、または再現失敗 → 記事から削除
```

**修正ルール:**
- 間違いが発覚したら記事を即修正してから Step 1 に進む
- ソースが見つからない情報は削除するか「〜と言われている」等の留保表現に変える
- 動かないプロンプトは削除する（修正できない限り載せない）
- 「確認できなかった」は「記憶で判断して OK」ではない。削除か留保が必須

### Step 2.8: 強調密度自動付与ロジック（必須）

Brain 販売ページの購入転換率は**太字密度**に強く相関する。競合分析実測:

**目標密度: 7.0 / 1000字 以上**（明鏡水準）。

#### 自動付与対象パラグラフの判定基準

以下のパターンに該当する段落を優先して**段落全体 `<strong>`** で囲む（ルール H の tiptap バグ回避遵守）:

1. **数字 + 単位**: `\d+\s*(秒|分|時間|日|週|ヶ月|万|円|本|行|%|倍|回|人|章|枚|個|ポイント)` にマッチ
2. **結論フレーズ**: 「〜できる」「〜になる」「〜が変わる」「〜で済む」等の到達状態文
3. **所要時間・到達目標**: `所要時間 \d+分` `到達目標 .*状態` を含む段落（Day 扉系）
4. **Before/After 圧縮表現**: `\d+\s*時間\s*→\s*\d+\s*分` `\d+/\d+` のような劇的変化
5. **インパクト短文**: 20 字以下の独立段落で、命題性のある一文
6. **価格正当化文**: 「1 件受けたら回収できる」「紹介 2 名で元が取れる」等
7. **決めフレーズ**: 記事末・章末の「景色が変わる」「材料は渡した」系

#### 自動付与スクリプト例（ProseMirror / tiptap 環境向け）

```js
const doc = editor.state.doc;
const numberPattern = /\d+\s*(秒|分|時間|日|週|ヶ月|万|円|本|行|%|倍|回|人|章|枚|個|ポイント)/;
const impactKeywords = ['到達目標', '所要時間', '→', 'Before', 'After', '圧縮', '差が', '変わる', '回収', '紹介', '元が取れる'];

const candidates = [];
doc.descendants((node, pos) => {
  if (node.type.name !== 'paragraph') return;
  const text = node.textContent.trim();
  if (!text) return;
  let allBold = true, hasText = false;
  node.descendants(c => {
    if (c.isText) { hasText = true; if (!c.marks.some(m => m.type.name === 'bold')) allBold = false; }
  });
  if (!hasText || allBold) return;
  if (numberPattern.test(text) || impactKeywords.some(k => text.includes(k)) || text.length <= 20) {
    candidates.push({ pos, nodeSize: node.nodeSize });
  }
});

// 目標密度になるまで candidate に bold を付与
const boldMark = editor.schema.marks.bold.create();
let tr = editor.state.tr;
candidates.forEach(({ pos, nodeSize }) => {
  tr = tr.addMark(pos + 1, pos + nodeSize - 1, boldMark);
});
editor.view.dispatch(tr);
```

#### 目標判定

```js

// density >= 7.0 を合格。未達なら候補追加して再付与
```

##### プロンプト 1: 実録記事型（`jituroku` + 受注・納品・副業ワークフロー系）

巨大数字（受注ジャンル数・実績数字）を主役に・キアロスクーロドラマチック。

```
プレミアムビジネス雑誌 FORBES や WIRED の表紙のような、ドラマチックで高級感のあるサムネイル画像を 16:9 横長で作ってほしい。

被写体・主役は画面右側中央に配置する巨大な立体メタリックゴールドの数字「{最大実績数字（例: 3）}」だ。高さは画面の約75%を占めるサイズで、ドラマチックな存在感を持つ。数字の質感は磨き上げた18金の光沢で、表面に微細なヘアライン仕上げが入る。正面からやや上からのローアングルで、彫像のような威厳を持たせる。数字の周りには、淡いゴールドのボリューメトリック光（体積光）が放射状にうっすらと広がり、神秘的なオーラを生む。

背景はミッドナイトブラックを基調に、左上から右下へとわずかに暗紫にグラデーションする。背景には繊細なゴールドの幾何学ライン（細い直線が斜めに交差するネットワークパターン）が薄く浮かび、奥行き感を演出する。テクノロジーカンファレンスの会場のような洗練された空間。

照明はキアロスクーロ技法でコントラストを極端に強くする。左側から強い主光（キーライト）が当たり、数字のハイライトを鋭く輝かせる。右側は深い影に落とし、照度比は1:8。バックライトはゴールデンアワーの逆光で、数字の輪郭からドラマチックな光漏れを生む。3点照明（ソフトボックス使用）のプロフェッショナル品質。被写界深度は浅く（f/1.8相当）、背景が柔らかくぼけて主役が前面に浮き立つ。

1段目（メインタイトル・最大・極太）: 「{ツール名（例: Claude Code）} × {対象（例: クラウドワークス）}」
　オフホワイトのサンセリフ太字、わずかにメタリック質感、文字の下に微かなドロップシャドウで立体感を出す
2段目（サブタイトル・大・極太）: 「{N}ジャンル{記事性質（例: 受注実録）}」
　メタリックゴールドの太字、1段目の約75%サイズ、数字「{N}」と色を呼応させる
3段目（具体内容・中・太字）: 「{具体例: データ入力 / Excel / Webアプリ}」
　オフホワイト、1段目の約35%サイズ
4段目（付加情報・小・太字）: 「{訴求文（例: 応募文・プロンプト・コード全公開）}」
　オフホワイト、3段目と同サイズ

タイポグラフィのフォントは Impact または Gotham Bold のような極太の現代的サンセリフ。視認性を最大化する。

数字「{N}」の左下、タイトルとの境界付近に、ミニマルで上質なフラット線画のアイコン3つ（{記事性質に合わせたアイコン例: コード括弧</>、Excelの表、Webブラウザ}）をゴールドで横一列に小さく配置する。

#### プロンプト選択ロジック（スキル内部で自動判定）

記事タイプと商品特性から以下のルールで選択:

1. 記事タイプが `jituroku` → **プロンプト 1（実録記事型）**
2. 記事タイプが `knowhow` かつタイトルに「教科書」「完全ガイド」「バイブル」を含む → **プロンプト 2（教科書型）**
3. 商品にテンプレ・GPTs・プロンプト集・設定ファイル等が同梱 → **プロンプト 3（ツールキット型）**
4. 価格 ¥7,980 以上 または タイトルに「完全ロードマップ」「完全公開」「全手順」を含む → **プロンプト 4（フラッグシップ型）**

複数条件に該当する場合は優先度 4 > 2 > 3 > 1 の順に選ぶ。

#### プレースホルダ置換ルール

上記プロンプト内の `{...}` は以下で置換:

- `{ツール名}` `{対象}`: 「Claude Code」「× クラウドワークス」等
- `{最大実績数字}`: 受注ジャンル数（3）等の整数（記事固有の数字）

### 記事内差し込み画像プロンプト生成（必須・サムネとは別）

販売ページ（Brain 公開部分）での購入転換率はビジュアル訴求に依存する。競合分析（2026-04-22 実施・メアリ 148部¥24,980 / 明鏡 ¥29,800 / hulu35 ¥3,980）で判明した事実:

| 記事 | 価格 | 画像枚数 |
|---|---|---|
| メアリ | ¥24,980 | 78 |
| 明鏡 | ¥29,800 | 23 |
| hulu35 | ¥3,980 | 29 |

**結論**: Brain で「画像ゼロ」はリジェクト級。最低 15 枚・推奨 20 枚以上を狙う。本文差し込み画像は「サムネとは別」に生成する必要がある。

#### プレースホルダー記法（記事 HTML 内）

**絶対に見落とさない形式**で本文 HTML に埋める。ユーザーが Brain エディタ上で画像をアップロードする際の目印として機能させる。

```html
<blockquote>
<p><strong>🔴🔴🔴 画像N「{ラベル}」挿入位置 🔴🔴🔴</strong></p>
<p><strong>⚠️ 生成プロンプトは {記事ID}_images.md の「## 画像N」を参照</strong></p>
</blockquote>
```

- blockquote + 太字 + 🔴絵文字 3 連続で視覚的に最大限目立たせる（Brain エディタ上でも紫系背景で表示される）
- `N` は通し番号（1 から）
- `{ラベル}` は何の画像かわかる短文（例: 「7つの約束」「月14万実績」「Day1扉」）
- `{記事ID}` は `brain_{YYYYMMDD}_{suffix}` 形式
- 本番時はユーザーが画像をアップロード → このブロックを削除 or 画像ノードで置換

#### 画像プロンプトファイル `_images.md` 生成

本文 HTML（`brain_{date}_{suffix}.html`）・メタ（`_meta.md`）と同階層に、画像プロンプト専用ファイルを生成:

- ファイル名: `brain_{YYYYMMDD}_{suffix}_images.md`
- 場所: `brain/today/`（本文と同じディレクトリ）

## Phase 6.5: Brain エディタへの流し込み（Ctrl+V 禁止・JavaScript API 経由）

Brain エディタ（https://brain-market.com/a/{ID}/edit）は **tiptap ベース**で、Ctrl+V ペースト経路に重大な「マーク継続バグ」がある。HTMLに `<strong>` が一箇所でもあると、以降の段落すべてが強制的に `<strong>` で包まれて全段落太字化する。

このため、**Brain エディタへの流し込みは必ず以下の手順で行う**。ユーザーに Ctrl+V させるのは禁止。

### 手順

1. Claude が Brain エディタのタブを mcp__claude-in-chrome__tabs_context_mcp で把握（なければ tabs_create_mcp で新規作成してユーザーに Brain エディタページを開いてもらう）
2. `document.querySelector('.tiptap.ProseMirror').editor` で editor instance を取得
3. `editor.commands.setContent(html)` で全置換、または `editor.commands.focus('end')` → `insertContent(html)` で末尾追加

### 分割挿入前の HTML minify（必須）

**流し込み前に必ず HTML を minify する**。段落間の改行・空白を除去しないと、tiptap が改行を hardBreak 付き空段落（`<p><br></p>` 相当、`node.textContent===''` かつ `nodeSize:4`）に変換し、表示上「空のセクションが大量」状態になる。

```js
const cleanHtml = rawHtml
  .replace(/<!--[\s\S]*?-->/g, '')  // HTML コメント除去（先頭メタデータ対策）
  .replace(/>\s+</g, '><')           // タグ間空白・改行除去
  .trim();
editor.commands.setContent(cleanHtml);
```

### 分割挿入（1記事30KB超の場合）

javascript_tool の text パラメータは 1 回あたり 20〜30KB が安全圏。超える記事は分割:

- Part 1: 無料ゾーン前半（~20KB、minify 済） → `editor.commands.setContent(part1)`
- Part 2: 無料ゾーン後半（minify 済） → `editor.commands.focus('end')` → `insertContent(part2)`
- Part 3: 有料ゾーン（minify 済） → `editor.commands.focus('end')` → `insertContent(part3)`

### 検証（必須）

流し込み後、以下 3 点を全てパスすることを確認:

**① 部分太字（マーク継続バグ症状）が 0 件**:

```js
const el = document.querySelector('.tiptap.ProseMirror');
const paragraphs = [...el.querySelectorAll('p')];
const partial = paragraphs.filter(p => {
  const s = p.querySelector('strong');
  return s && s.textContent.trim() !== p.textContent.trim();
}).length;
// partial === 0 であること
```

**② 空段落（見えない空セクション）が 0 件**:

```js
const editor = document.querySelector('.ProseMirror').editor;
let emptyCount = 0;
editor.state.doc.descendants((node) => {
  if (node.type.name === 'paragraph' && node.textContent.trim() === '') {
    emptyCount++;
  }
});
// emptyCount === 0 であること
// ⚠️ content.size === 0 のみでは検知できない。hardBreak だけの段落は size>0 だが表示上は空
```

**③ 空段落が検出された場合は後ろから一括削除**:

```js
const editor = document.querySelector('.ProseMirror').editor;
const positions = [];
editor.state.doc.descendants((node, pos) => {
  if (node.type.name === 'paragraph' && node.textContent.trim() === '') {
    positions.push({ pos, nodeSize: node.nodeSize });
  }
});
let tr = editor.state.tr;
positions.reverse().forEach(({ pos, nodeSize }) => {
  tr = tr.delete(pos, pos + nodeSize);
});
editor.view.dispatch(tr);
```

### 絶対禁止

- ❌ ユーザーに HTML 全文をコピーさせて Ctrl+V させる（全段落太字化する）
- ❌ HTML の段落中間に `<b>` / `<strong>` を書く（Phase 2 H ルール違反）
- ❌ setContent / insertContent の結果を検証せずに完了扱いにする
- ❌ minify を省略して raw HTML を流し込む（空段落が大量発生する）
- ❌ 空段落チェックを `content.size === 0` だけで行う（`textContent.trim() === ''` で判定する）

### 画像・有料ゾーン境界・サムネ設定

HTML 流し込みで本文は完了するが、以下は Brain エディタの UI 操作が別途必要:

- **有料ゾーン境界**: 本文の「ここから先は ¥○,○○○」の後ろで、エディタ UI の「有料部分の開始」ボタンをクリック
- **画像アップロード**: `editor.commands.setImage({src, alt})` で外部 URL 挿入も可能だが、Brain の画像は JPEG/PNG ファイルアップロード運用が基本
- **サムネ**: メインサムネ画像は記事ヘッダーの「メイン画像をアップロードする」から別途アップ

## Phase 7: Notion 記録

### 対象記事のレコードを更新

Notion「記事管理」DB（data_source: `collection://812aa728-8d3e-42e4-a9cd-6a91c303b2c2`）の対象記事レコードに以下フィールドを更新する:

- `Brainステータス`: `草稿完成`
- `Brain価格`: `¥{価格}`
- `Brain還元率`: `{%}%`

### pending_cta 保存（x-run 連携用）

Brain 商品の告知候補メタ情報を `x/pending_cta/` に JSON で保存する。
**告知文本体の生成は x-run に委譲**。x-run が daily-auto 実行時に自動判定して夜前半枠に組み込む。

ファイル名: `x/pending_cta/brain_{YYYYMMDD}_{slug}.json`

保存内容:

```json
{
  "type": "brain",
  "created_at": "2026-04-23T22:00:00+09:00",
  "source": "brain/today/brain_20260423_xxx.md",
  "title": "{Brain 商品の最終タイトル}",
  "url": null,
  "atomic_ideas": [
    "{核心1: Brain版で追加された最強の価値（コード全文・プロンプト全文等）}",
    "{核心2: note版との差分（note¥XXX → Brain¥XXX で何が増えたか）}",
    "{核心3: 読むべき人・解決する悩み}"
  ],
  "image_path": "{Brain サムネの絶対パス}",
  "article_type": "brain_jituroku",
  "price": 1980,
  "referral_rate": 50,
  "target_fit": "high",
  "suggested_priority": "high"
}
```

**フィールドの判定基準:**

- `url`: Brain 公開前は `null`。投稿後にユーザーまたは x-run が追記
- `atomic_ideas`: **note版との差分（Brain版の「買う価値」）を必ず含める** — コード全文・プロンプト全文・テンプレ数など
- `article_type`: `brain_jituroku` / `brain_knowhow` / `brain_toolkit`
- `price`: 数値（¥1,980 なら `1980`）
- `referral_rate`: 紹介還元率（%・数値のみ）
- `target_fit`: note版と同基準
  - `high`: 副業・実録・初心者向け体験記
  - `medium`: AI活用Tips・ツール比較・ノウハウ
  - `low`: エンジニア寄り・技術深掘り
- `suggested_priority`: Brain は売上直結のため基本 `high` 推奨。note 告知と競合した場合、x-run は brain を優先する

**target_fit が `low` の場合、x-run 側で告知スキップ** されることがある。

### Brainネタ帳ステータス更新（ルート C の場合のみ・2026-04-24 追加）

ルート C（from-idea）でネタを取得した場合、Brainネタ帳の対象レコードを更新:
- `mcp__notion__notion-update-page` でステータス=`採用済み`に変更
- メモ欄に「対象記事: {Brain商品名} / 投稿日: YYYY-MM-DD」追記

ルート A / B の場合はこの更新不要。

## Phase 8: 投稿URL取得 & PDCA フィードバック

### Step 1: BrainURL ヒアリング & Notion 自動更新

Phase 7 の Notion 記録完了後（ユーザーが手動で Brain に投稿した後）、以下の形式で質問する:

```
💎 Brain に投稿した？ BrainURL を教えて（例: https://brain-market.com/c/xxxxx）
  → まだなら「まだ」で OK

投稿済みなら、以下も確認させて:
- 最終価格（Phase 4 の提案価格から変更した？ 変更してないなら「変更なし」）
- 最終還元率（変更した？ なければ「変更なし」）
- 公開日（今日で良ければ「今日」）
```

**URL + メタ情報が返ってきた場合の処理:**

1. `mcp__notion__notion-search` で 記事管理DB から記事名マッチでレコード取得
   - `data_source_url`: `collection://812aa728-8d3e-42e4-a9cd-6a91c303b2c2`
   - 記事名は Phase 0 で確定した対象 note 記事と同一
2. `mcp__notion__notion-update-page` で該当レコードを `update_properties` 更新:
   - `BrainURL`: 受け取った URL
   - `date:Brain公開日:start`: 指定日（または今日 YYYY-MM-DD）
   - `date:Brain公開日:is_datetime`: 0
   - `Brain価格`: 変更指定があれば新価格、なければ Phase 4 の値のまま
   - `Brain還元率`: 変更指定があれば新値、なければ Phase 4 の値のまま
   - `Brainステータス`: `投稿済み`
3. 完了メッセージ:
   ```
   ✅ Notion「記事管理」DBに記録した
   → 月次の Brain売上 は手動で更新してね（推奨: 毎月1日に collect-stats で一括更新）
   ```

**「まだ」が返った場合:**

- Notion 更新はスキップ
- 「OK、Brain 投稿後に改めて聞くから教えてね。Notion UI から手動更新でも可」と案内
- `Brainステータス` は `草稿完成` のまま維持

### Step 2: PDCA フィードバック記録項目（投稿後段階的）

**1 週間後:**
- 販売数
- Brain 内 PV
- レビュー・コメント

**1 ヶ月後:**
- 累計販売数
- 売上
- アフィリエイター経由の割合
- 学び（何が効いた・効かなかった）

これらは Notion の対象レコードに追記する。次回 /brain-run 実行時に Phase 1 の相場調査と併せて参照される（「前回 X 件売れたから今回は価格を上げよう」等の判断に使う）。
